# Промпт для миграции на модель PetOwner

**Роль:** Senior Backend Developer / System Architect

**Задача:** 
Отрефакторить архитектуру владения питомцами. Текущая реализация в приложении `pets` использует два поля в модели `Pet`:
- `main_owner` (ForeignKey на User)
- `owners` (неявный ManyToManyField на User)

Нужно перейти на явную through-модель `PetOwner`, которая будет связывать питомца и пользователя с указанием роли и других метаданных. 

*Расположение модели:* Приложение `pets` (это лучшее архитектурное решение, так как связь неразрывно связана с доменом питомцев).

---

## План действий, который необходимо выполнить:

### 1. Изменения в моделях (`pets/models.py`)
- Создай новую модель `PetOwner`:
  - `pet`: ForeignKey на `Pet` (CASCADE - используется только на уровне БД для консистентности, но физически мы объекты не удаляем).
  - `user`: ForeignKey на `User` (CASCADE - аналогично).
  - `role`: CharField с choices (`'main'` - Основной владелец, `'coowner'` - Совладелец).
  - `created_at`: DateTimeField(auto_now_add=True).
  - `Meta`: `constraints = [models.UniqueConstraint(fields=['pet', 'user'], name='unique_pet_owner_rel')]` (вместо устаревшего `unique_together`).
- Обнови модель `Pet`:
  - **Удали** поле `main_owner`.
  - Обнови поле `owners`: `owners = models.ManyToManyField(User, through='PetOwner', related_name='pets', ...)`.
  - Важное правило: **МЫ НЕ УДАЛЯЕМ ЯВНО НИКОГО (ПОЛЬЗОВАТЕЛЕЙ ИЛИ ПИТОМЦЕВ). МЫ ДОЛЖНЫ ИХ ТОЛЬКО ДЕАКТИВИРОВАТЬ** (soft delete, `is_active=False`). Это фундаментальное правило системы.
  - Добавь property `main_owner` в саму модель `Pet`, которое будет возвращать пользователя с `role='main'`. **Важно для производительности:** чтобы не ложить базу тысячами запросов (N+1), это property должно искать владельца среди уже предзагруженного списка в памяти (например, `[owner for owner in getattr(self, "prefetched_petowners", self.petowner_set.all()) if owner.role == 'main']` или перебором `petowner_set.all()`, если используется `prefetch_related`).
  - Удали из метода `Pet.clean()` проверки, связанные с полем `main_owner`.
  - Удали кастомную логику сохранения `main_owner` в `Pet.save()` (если она там есть).

### 2. Безопасная миграция данных
Так как база данных на данный момент очищена от питомцев, сложная data-миграция не требуется.
Просто удали поле `main_owner`, удали текущее поле `owners`, создай модель `PetOwner` и добавь поле `owners` заново с `through='PetOwner'`.
Сделай всё это в рамках обычных `makemigrations` и `migrate`.

### 3. Права доступа (Permissions в DRF)
- Создай/обнови классы разрешений (Permissions в DRF), например `IsMainOwner` и `IsCoowner`.
- Логика доступов:
  - **Совладелец (`IsCoowner`)**: может ходить с питомцем куда угодно, записывать на приемы, редактировать обычный профиль питомца и т.д.
  - **Основной владелец (`IsMainOwner`)**: имеет все права совладельца, и **ТОЛЬКО ОН МОЖЕТ**:
    1. **Деактивировать** (НЕ удалять физически!) питомца.
    2. Сделать инвайт (приглашение) на совладение другому пользователю.
    3. Передать права основного владельца другому лицу.
- *Примечание:* Остальные сложные случаи (типа смерти владельца) на MVP решаются **только через поддержку**. Не нужно усложнять систему обработкой таких кейсов в коде; текущий `FunctionalDesign.md` в этом плане избыточен.

### 4. Транзакции и безопасность (`pets/serializers.py`)
- **ОБЯЗАТЕЛЬНО:** Транзакция смены роли с `'main'` на `'coowner'` должна быть строго обернута в `transaction.atomic()` во View или Serializer. (Это прописано в правилах проекта).
- Удали хардкод логики вроде `pet.owners.add(main_owner)` в методах `create` и `update` в `PetSerializer`.
- В `PetSerializer.create()`: при создании питомца явно создавай запись `PetOwner.objects.create(pet=pet, user=validated_data['main_owner'], role='main')`.

### 5. Рефакторинг Admin панели (`pets/admin.py` и кастомная админка)
- Создай `PetOwnerInline` (TabularInline), чтобы в админке питомца можно было удобно добавлять владельцев и назначать им роли.
- В `PetAdmin`:
  - Удали `main_owner` из `list_display`, `list_filter` и `search_fields` (замени на `owners__email` или аналогичные lookup-ы).
  - Добавь `PetOwnerInline` в `inlines`.
- В `PetAdminForm`:
  - Полностью выпили кастомную логику переопределения `self.fields['main_owner']` и ручную настройку `self.fields['owners']`. Теперь этим должен управлять Inline.

### 6. Оптимизация и рефакторинг API (`pets/api_views.py` и Serializers)
- При отдаче списков питомцев **явно используй** `prefetch_related('owners')` или `prefetch_related('petowner_set')` в `get_queryset` во всех View и Serializers. Для получения `main_owner` избегай N+1 запросов: используй либо подзапросы (Subquery) в ORM, либо фильтрацию предзагруженного массива в памяти Питона.
- Исправь эндпоинты, которые могли рассчитывать на прямое обновление `pet.main_owner` (например, эндпоинты трансфера питомца / приглашения совладельцев в `invites` или `pets/api_views.py`). Замени их на обновление `PetOwner.role` через `transaction.atomic()`.

### 7. Сигналы и консистентность
- Если необходимо, добавь сигнал (pre_save/post_save на `PetOwner`), который будет гарантировать, что у питомца всегда ровно один `role='main'`.

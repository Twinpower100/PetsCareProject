# Prompt: Реализация поиска услуг и процесса бронирования (Service Booking Flow)

**Контекст проекта:**
Проект "PetsCare" состоит из бэкенда на Django/DRF (`PetsCare-backend`) и клиентского фронтенда на React/TypeScript (`Petscare-web`).
Текущая задача заключается в реализации процесса поиска локаций, предоставляющих услуги для конкретного питомца, и бронирования окон времени.

**Роль ИИ:**
Ты Senior Fullstack Developer (Django + React/TS). Твоя задача — реализовать поиск услуг и бронирование в соответствии с требованиями UI/UX и бизнес-логикой, описанными ниже. Код должен быть чистым, с типизацией (TS), использовать современные хуки React, а на бэкенде — оптимизированные запросы Django ORM с поддержкой PostGIS для фильтрации по геолокации.

## 1. UX/UI Требования и Бизнес-логика

Ниже приведены утвержденные требования к пользовательскому пути (основано на документе UI/UX). Вся реализация должна строго им следовать:

### 1.1. Паттерн поиска "Услуга для конкретного питомца"
Любой поиск начинается по цепочке: **Питомец -> Услуга -> Расплывчатое время -> Локация**.

1. **Селектор питомца:**
   - Горизонтальная карусель аватаров питомцев (или выпадающий список для мобайла).
   - *Бизнес-логика:* Выбор питомца автоматически ограничивает доступный список услуг в каталоге на фронтенде/бэкенде (сверяем с `allowed_pet_types` у `Service`).
2. **Точка входа к услугам (Omnibox + Каталог):**
   - Единое поле ввода с автодополнением (поиск по названию `Service` листового уровня, имени провайдера или адресу локации).
   - Кнопка "Каталог" для вызова компонента Cascader/Tree (для навигации по иерархии `catalog.Service`). Корень абстрактно не выводим.
3. **Фильтр грубого времени:**
   - Четыре "чипса" под поиском: "Сегодня", "Завтра", "На выходных", "Любое время".
   - *Бизнес-логика:* Отсекать провайдеров (`ProviderLocation`), у которых нет свободных слотов в эти даты на этапе выдачи результатов.
4. **Локация пользователя и кнопка "Рядом":**
   - Использовать API браузера `navigator.geolocation`. При отказе — фолбек по IP или текстовый ввод.
   - Кнопка 📍 "Рядом со мной" возле строки поиска: включает запрос координат и вызывает эндпоинт поиска, ожидающий параметры `lat` / `lon` для сортировки и расчета дистанции.

### 1.2. Экран результатов (Связка "Карта + Список")
Отказаться от переключения вкладок. Реализовать **Split View**:
1. **Desktop:** Слева список локаций, справа — карта с пинами (`ProviderLocation`).
2. **Mobile:** Карта + Bottom Sheet. При свайпе вверх списка карточек — карта скрывается. Свайп вниз — карта + карусель карточек поверх карты.
3. **Интерактивность:** Клик/наведение на карточку в списке подсвечивает пин на карте, и наоборот. Карточка должна моментально считываться.
4. **Карточка локации (Provider Location Card):**
   - Заголовок: Название локации.
   - Метрики: Рейтинг и самое главное — **Дистанция от пользователя**.
   - *Мобайл-акцент:* Дистанция выводится приметным бейджем ("🚗 1.2 км", "🚶 800 м").
   - Сразу выводим цену и длительность (`price` и `duration_minutes` из `ProviderLocationService`).
   - Кнопка: "Показать свободные окна" или "Записаться".

### 1.3. Процесс бронирования (Booking Flow)
Окно бронирования открывается в модалке/шторке без перехода на новую страницу, так как Питомец и Услуга уже в контексте.

1. **Шаг 1: Выбор окна времени (Time Slots):**
   - Свайпаемый горизонтальный календарь (на 7-14 дней вперед). Начальная дата = "Грубое время" из поиска.
   - Дни без доступных окон — задизейблены.
   - *Бизнес-логика:* Слоты группируются ("Утро", "День", "Вечер"). Выводятся только слоты, вмещающие услугу (рассчитывается на бэкенде с учетом `duration_minutes` и `tech_break_minutes`).
2. **Шаг 2: Подтверждение записи (Summary):**
   - Отображаются: Название услуги, фото/имя питомца, адрес локации, дата/время, итоговая стоимость.
   - Кнопка: "Забронировать". Комментарии к записи на этом шаге не нужны (берутся из карточки питомца).
3. **Обязательные UX фичи:**
   - **Без мертвых тупиков:** Авто-перемотка на ближайший свободный день ("Сегодня занято, доступно во вторник").
   - **Skeleton Loaders:** Использовать при подгрузке слотов и расписаний.
   - **Breadcrumbs в каталоге:** Для быстрого перехода на уровень выше по дереву услуг.

## 2. Задачи для бэкенда (Django/DRF)

Твоя задача — изучить (смотри `PetsCare/providers/models.py` и `PetsCare/catalog/models.py`) и реализовать или обновить следующие эндпоинты:

1. **Эндпоинт поиска провайдеров (`GET /api/booking/search/` или подобный):**
   - Должен принимать: `pet_id` (для фильтрации `Service.allowed_pet_types`), опционально `service_code`/`id` или текстовый поисковый запрос (по названию услуги, провайдера, адресу локации).
   - Фильтрация геолокации (`lat`, `lon`, радиус сортировки гео-объектов `ProviderLocation`).
   - Фильтр "грубого времени": дата или период. Бэкенд должен провернуть пересечение `duration` + `tech_break_minutes` со слотами `ProviderLocation` и вернуть только те локации, где есть доступные окна.
   - В ответе помимо данных локации, выводить дистанцию в метрах/километрах (`distance`). Оптимизировать запросы с аннотацией дистанции (PostGIS `Distance`).

2. **Эндпоинт получения свободных слотов (`GET /api/booking/locations/<id>/slots/`):**
   - Должен принимать: `date_start`, `date_end`, `service_id` (или `duration_minutes`).
   - Логика: Возвращать список словарей сгруппированных по дням со слотами (`start_time`, `end_time`). Необходимо учитывать уже существующие записи (брони) и исключать их, а также проверять расписание локации/сотрудников.

3. **Эндпоинт создания бронирования (`POST /api/booking/appointments/`):**
   - Принимает: `provider_location_id`, `service_id`, `pet_id`, `date`, `start_time`.
   - Проверки (Clean): доступна ли услуга питомцу, свободно ли окно (защита от race conditions/двойного бронирования).

## 3. Задачи для фронтенда (React/TypeScript)

В каталоге фронтенда (`Petscare-web`) реализуй:

1. **Стейт-менеджмент поиска:**
   - Создать контекст или Redux slice для хранения параметров бронирования: `selectedPet`, `selectedService`, `roughTime`, `userLocation`.

2. **Компоненты главного экрана поиска:**
   - `<PetSelector />`: Карусель выбора питомцев текущего юзера.
   - `<ServiceSearchOmnibox />`: Строка поиска с dropdown/автодополнением списка услуг и кнопкой для открытия `<ServiceCascader />`.
   - Кнопка "Рядом" с логикой `navigator.geolocation.getCurrentPosition()`.

3. **Компоненты Split View:**
   - Адаптивный `<LocationsSplitView />` разделяющий экран на список (`<LocationList />`) и `<MapComponent />` (Leaflet или Google Maps).
   - `<LocationCard />`: компонент вывода информации о локации (дистанция, цена, время услуги из `ProviderLocationService`).
   - Синхронизация hovered/selected стейта между картой и списком.

4. **Booking Flow (Модалка/Шторка):**
   - `<TimeSlotSelector />`: Горизонтальная лента дат, сетка чипсов со временем, логика автоперехода на свободный день, показ Skeleton при загрузке.
   - `<BookingSummary />`: Финальный экран подтверждения и вызов мутации создания брони.

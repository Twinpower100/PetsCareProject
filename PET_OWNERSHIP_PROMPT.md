# Промпт для Senior Fullstack Developer: Инвайт совладельца и передача прав на питомца

**Роль:** Senior Fullstack Developer (Django/DRF + React/TypeScript)
**Задача:** Реализовать функционал управления совладельцами питомцев на вкладке «Мои питомцы» в личном кабинете пользователя (`http://localhost:3000/account`).

Функционал охватывает три основных процесса:
1. Инвайт (приглашение) другого пользователя на роль совладельца (Co-owner) питомца.
2. Передача полномочий основного владельца (Primary Owner) другому пользователю.
3. Деактивация (удаление) питомца основным владельцем.

Необходимо спроектировать и реализовать как API на бэкенде (`PetsCare-backend`), так и UI на фронтенде (`Petscare-web`).
*(Примечание: базовые модели и часть бэкенда для доступов/удаления уже могут быть реализованы — обязательно проверьте текущий код).*

---

## Технические и архитектурные требования (Критично)

1. **Транзакционность (`transaction.atomic`)**: Все операции по передаче прав и принятию инвайтов должны быть строго обернуты в `transaction.atomic` (и при необходимости `select_for_update()`), чтобы исключить гонки (race conditions) и нарушение целостности данных (например, когда питомец остается без основного владельца или появляется два основных владельца одновременно).
2. **Mobile First**: Верстка всех форм, списков и модальных окон должна проектироваться в первую очередь для корректного отображения на мобильных устройствах.
3. **Переиспользование кода**: 
   - На фронтенде используйте уже существующие компоненты (кнопки, модалки, инпуты из shadcn/ui).
   - Разносите логику по небольшим переиспользуемым компонентам (не делайте "простыню" в одном файле).
4. **Валидация и безопасность**: 
   - Передавать права основного владельца или удалять пользователей может *только* текущий основной владелец питомца (`IsPrimaryOwner` permission).
   - Пользователь не может пригласить сам себя или пригласить того, кто уже является владельцем.
   - Строгая типизация запросов (Zod на клиенте, сериализаторы на сервере).

## 1. Backend (Django / DRF)

### Модели данных (Контекст)
Скорее всего, промежуточные таблицы уже реализованы в `pets/models.py`. 
- Проверьте модель `PetOwner` (роли: `main`, `coowner`). У одного питомца может быть **строго один** `main` владелец.
- Убедитесь в наличии модели инвайтов (например, `PetOwnershipInvite` с полями для типа инвайта — invite/transfer).
Если их нет — создайте или обновите.

### Бизнес-логика и Endpoints (проверьте существующие `pets/api_views.py`)

1. **Создание инвайта (Invite Co-owner)**  
   `POST /api/v1/pets/<pet_id>/invites/`
   - **Body:** `email` (кого приглашаем).
   - **Логика:** Проверить, что запрашивающий — `PRIMARY_OWNER`. Проверить, что `email` еще не является совладельцем, но является пользователем системы. Создать запись в `PetInvite`. Сгенерировать уникальный токен. Отправить email-уведомление с ссылкой на принятие инвайта.

2. **Список совладельцев и инвайтов**  
   `GET /api/v1/pets/<pet_id>/owners/`
   - **Ответ:** Список текущих владельцев (с их ролями) и список активных отправленных инвайтов (Pending Invites).

3. **Принятие/Отклонение инвайта**  
   `POST /api/v1/pets/invites/<token>/accept/` (или `reject/`)
   - **Логика (Обязательно `transaction.atomic`!):** 
     - Заблокировать строку питомца/инвайта.
     - Если `accept`: Добавить запись в `PetOwnership` с ролью `CO_OWNER`, пометить токен как использованный (или удалить).
     - (Так как инвайт можно выслать *только* зарегистрированному пользователю, дополнительная проверка регистрации здесь не требуется, достаточно провалидировать токен и сессию текущего пользователя).

4. **Удаление совладельца / Отзыв инвайта**  
   `DELETE /api/v1/pets/<pet_id>/co-owners/<user_id>/`
   `DELETE /api/v1/pets/invites/<invite_id>/`
   - Роуминг прав: Основной владелец может удалить любого совладельца. Совладелец может сам удалить себя (покинуть питомца). Основного владельца удалить нельзя, пока он не передал права.

5. **Передача прав основного владельца (Transfer Ownership)**  
   `POST /api/v1/pets/<pet_id>/transfer-ownership/`
   - **Body:** `new_owner_id` (ID пользователя, который станет новым `PRIMARY_OWNER`). Этот пользователь **обязан** быть среди `CO_OWNER` этого питомца.
   - **Логика (Обязательно `transaction.atomic`!):**
     - Lock записей `PetOwnership` для этого питомца.
     - Проверка прав: Инициатор запроса должен быть `PRIMARY_OWNER`.
     - Запросы в БД: Текущему `PRIMARY_OWNER` меняем роль на `CO_OWNER`. Целевому сотруднику (`new_owner_id`) меняем роль на `PRIMARY_OWNER`. Обе операции в одной транзакции.
     - Выслать email-уведомление обоим юзерам об успешной передаче прав.

6. **Удаление (деактивация) питомца**  
   `DELETE /api/v1/pets/<pet_id>/` (возможно, уже реализовано через `PetViewSet.perform_destroy` или `PetDeleteAPIView`)
   - **Логика:** Устанавливает `is_active = False` на питомце. Доступно только основному владельцу. Фронтенду нужно просто правильно вызывать этот эндпоинт и затем скрывать питомца из UI.

## 2. Frontend (React / TypeScript)

### Расположение в UI
Вкладка `Мои питомцы` (`http://localhost:3000/account`).
Внутри детальной карточки/страницы конкретного питомца нужно добавить секцию **«Владельцы и доступы» (Ownership & Access)**.

### UI / UX Компоненты

1. **Список владельцев (`PetOwnersList.tsx`)**
   - Отображает карточки/список пользователей, имеющих доступ к питомцу.
   - Текущий владелец помечен бейджем `Основной владелец` (Primary Owner).
   - Остальные имеют бейдж `Совладелец` (Co-owner). Для каждого совладельца отображается иконка удаления/корзины (видна только основному владельцу).
   - Раздел «Ожидающие приглашения» (Pending Invites) показывает зависшие инвайты с кнопкой «Отозвать» (Revoke).

2. **Приглашение совладельца (`InviteCoOwnerForm.tsx` или Modal)**
   - Модальное окно (Dialog) формы с полем ввода E-mail.
   - Кнопка «Отправить приглашение».
   - Лоадер во время отправки (isSubmitting) и строгая валидация email через Zod.
   - Уведомление об успехе (Toast).

3. **Передача полномочий (`TransferOwnershipModal.tsx`)**
   - Кнопка «Передать права на питомца» (Transfer Ownership) видна **только** основному владельцу.
   - При клике: модальное окно (AlertDialog).
   - Внутри: Выбор кому передать права из текущего списка совладельцев (Select/Dropdown).
   - **Danger Zone / UX:** Текст с предупреждением: «Внимание: вы передаете полные права на питомца. После передачи вы станете обычным совладельцем. Отменить действие можно будет только если новый владелец передаст права обратно».
   - *(Опционально)* Для подтверждения требовать ввести имя питомца или нажать специальный чекбокс "Я понимаю".
   - Состояние загрузки, Toast об успехе и обновление локального списка (теперь у нас пропадает кнопка передачи прав).

4. **Принятие инвайта (Роутинг `AcceptPetInvite.tsx`)**
   - Маршрут вида: `http://localhost:3000/invites/pet/<token>`.
   - Страница показывает: «Пользователь X приглашает вас стать совладельцем питомца Y».
   - Действия: «Принять приглашение» / «Отклонить».
   - После принятия перенаправлять в `http://localhost:3000/account` (вкладку питомцев).

5. **Модалка удаления питомца (`DeletePetDialog.tsx`)**
   - Кнопка «Удалить питомца» (в самом низу или в шапке карточки), доступна **только основному владельцу**.
   - AlertDialog с красным текстом. Сообщение: «Вы действительно хотите удалить карточку питомца? Данные перейдут в архив и будут скрыты из списка».
   - Лоадер на время запроса к `/api/v1/pets/<pet_id>/` (DELETE). После успеха закрытие окна и обновление списка (питомец должен пропасть из UI).

### Интеграция API
Создать сервис, например `petOwnershipAPI.ts`.
Описать строгие TypeScript `interface` для ответов.
Рекомендуется использовать оптимистичное обновление для удаления инвайтов. Ошибки от DRF перехватывать и парсить (выводить в UI, а не просто консоль).

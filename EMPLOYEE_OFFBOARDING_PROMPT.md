# Роль: Сеньор фуллстек разработчик
# Задача: Реализация механизма увольнения (деактивации) сотрудника филиала и флоу работы с записями

## 📌 Контекст
В системе есть необходимость "мягко удалять" (offboarding) сотрудников из конкретных филиалов (локаций), так как жесткое удаление (hard delete) недопустимо из-за необходимости сохранения истории бронирований, расчетов и аналитики.
Кроме того, у нас уже существуют модели отпусков (`Vacation`) и больничных (`SickLeave`) (в `scheduling/models.py`), где также возникает потребность отменять или переносить будущие записи (Booking). 

Следовательно, механизм отмены/замены записей должен быть универсальным, а увольнение сотрудника из локации — одним из триггеров этого механизма.

## 🎯 Цели
1. Внедрить **Soft Delete / Deactivation** для сотрудника в рамках конкретной локации.
2. Реализовать **Сценарий с решением (Resolution Payload)** для массовой обработки предстоящих визитов (`Booking`), связанных с этим сотрудником:
   - Перенос (Reassign) на другого мастера.
   - Отмена (Cancel) визитов.
3. Обеспечить возможность переиспользования логики обработки конфликтов (записей на будущее) для других жизненных ситуаций (например, при уходе в отпуск или на больничный).

## 🛠 Архитектурные изменения

### 1. Модель данных (БД)
Текущая связь сотрудника с локацией (`Employee.locations`) реализована через автоматическую `ManyToManyField`.
- **Действие:** Необходимо сделать эту связь историчной и управляемой.
- **Решение:** (Выбрать оптимальный путь при реализации)
  - *Вариант А:* Добавить `is_active` (boolean) и `end_date` (date/datetime) в качестве полей в явную промежуточную (through) таблицу для `Employee.locations`.
  - *Вариант Б:* Если таблица `EmployeeLocationRole` уже выступает фактической связкой 1-к-1 для пары "Сотрудник-Локация", можно добавить флаг `is_active` туда. Однако семантически правильнее использовать выделенную through-модель `EmployeeLocation`.
- **Расписание и Услуги:** Убедиться, что к деактивированному сотруднику (или для неактивной связки с локацией) недоступна запись, и он не отображается в списке активных мастеров локации на фронте.

### 2. Универсальный сервис отмены/переноса записей
Создать сервисный класс (например, `BookingReassignmentService`), который умеет принимать `[booking_ids]` или параметры фильтрации (например, "все будущие активные записи сотрудника X в локации Y начиная с даты Z") и выполнять два действия:
- **`reassign`**: Изменяет `employee` у бронирования на `target_employee_id`. Проверяет, что новый сотрудник доступен в это время, выполняет нужные услуги и привязан к локации. Освобождает старые `TimeSlot` и занимает новые.
- **`cancel`**: Переводит бронирования в статус `cancelled_by_provider`. Освобождает `TimeSlot`. (Опционально формирует события для рефандов, если была оплата).

## 🚀 API Эндпоинты

Требуется разработать/обновить эндпоинт для деактивации сотрудника в локации:
**`PATCH /api/locations/{location_id}/staff/{employee_id}/deactivate/`**

### Фаза 1: Обычный запрос (Dry Run)
Если фронтенд отправляет простой запрос на деактивацию без дополнительных параметров:

**Payload:** `{}`

**Логика:**
Бэкенд ищет все будущие бронирования (`Booking`) в статусе `active` со временем старта больше текущего `now()`, где исполнитель — `employee_id` в рамках `location_id`.

- **Если будущих записей НЕТ:**
  - Деактивируем связь сотрудника с локацией (ставим `is_active=False` и `end_date=now()`).
  - Убираем токены/доступ сотрудника к ресурсам этой локации (если требуется).
  - Ответ: `200 OK`.

- **Если будущие записи ЕСТЬ:**
  - Деактивация **прерывается**.
  - Ответ: `409 Conflict` (или `422 Unprocessable Entity`).
  - Формат ответа:
    ```json
    {
      "error": "has_future_bookings",
      "future_bookings_count": 5,
      "message": "У сотрудника есть предстоящие записи. Выберите действие: отменить их или перенести на другого специалиста.",
      "bookings": [ ...опционально краткий список визитов... ]
    }
    ```

### Фаза 2: Запрос с решением (Resolution Payload)
После получения `409`, фронтенд (менеджер локации) показывает модальное окно. Менеджер принимает решение, что делать с будущими визитами, и повторяет запрос.

**Параметры:**
```json
{
  "resolution": "reassign", // либо "cancel"
  "target_employee_id": 45, // Обязателен при "reassign"
  "cancel_reason": "Специалист больше не работает в филиале" // Доп. инфо при "cancel"
}
```

**Логика (Оборачивается в `transaction.atomic()`):**
1. Применяется выбранный `resolution` ко всем будущим `Booking` через наш универсальный `BookingReassignmentService`.
2. В случае `reassign`: проверяются конфликты у `target_employee_id`. Если есть непреодолимые наложения времени (которые система не позволяет), возвращается ошибка `400 Bad Request` с указанием конкретных конфликтов.
3. Рассылаются уведомления клиентам:
   - "Визит перенесен к мастеру [Имя]..."
   - "Ваш визит отменен по причине..."
4. Сотрудник деактивируется в локации.
5. Ответ: `200 OK`.

## 🔄 Связь с другими бизнес-процессами

Механизм **Resolution Payload (Сценарий Б)** должен быть спроектирован так, чтобы его можно было легко применить при:
1. **Создании `Vacation` (Отпуска)**:
   Если менеджер добавляет отпуск на даты, в которых у сотрудника уже есть записи — API создания отпуска также должно возвращать `409 Conflict` со списком записей и требовать `resolution` (Cancel/Reassign).
2. **Создании `SickLeave` (Больничного)**:
   Внезапный больничный — API создания больничного также должно отловить все визиты на период больничного и потребовать `resolution`.

## 🛑 Самостоятельное удаление аккаунта пользователем
Если пользователь (сотрудник) решил удалить свой аккаунт целиком через систему (self-deletion):
- При вызове удаления аккаунта бэкенд должен проверить, является ли этот `User` сотрудником (`Employee`) с будущими визитами (`Booking`) в любых локациях.
- **Действие:** Нельзя применять `hard delete` к пользователю.
- Пользователь переводится в статус `is_active=False` (на уровне `User` и `Employee`), доступ к логину отзывается (очищается e-mail/телефон для соблюдения GDPR через обфускацию, например: `deleted_12345@example.com`).
- Все его будущие записи (`Booking`) должны быть **автоматически переведены в статус `cancelled_by_provider`**.
- **ОБЯЗАТЕЛЬНО** отправляются два типа уведомлений:
  1. **Клиентам** — об отмене их визита (включая причину и извинения).
  2. **Менеджерам локаций** (`location_manager`) — системное оповещение и/или email о том, что их сотрудник удалил аккаунт и все его будущие визиты были отменены (чтобы менеджер мог срочно перезвонить клиентам и вручную предложить им перенос).
*Примечание:* Здесь мы не требуем автоматического `Reassign` через бэкенд, так как удаление пользователя инициируется им самим и происходит асинхронно. Менеджеры разбираются с последствиями вручную на основе полученных уведомлений.
## ✅ Критерии приемки (DoD)

### Бэкенд (Логика и Данные)
1. **БД**: `Employee.locations` и другие относящиеся связи корректно обрабатывают мягкое удаление (`is_active` / `end_date`). Исторические визиты и расчёты с "уволенным" мастером открываются без ошибок.
2. **Остановка букинга**: Сотрудник немедленно пропадает из списка доступных специалистов для онлайн- и офлайн-записей (`where is_active=true`).
3. **Отзыв доступов**: При деактивации система должна инвалидировать/отзывать права доступа сотрудника к расписанию и клиентам *конкретно этого филиала*, сохраняя доступы к другим филиалам сети, если он там продолжает работать (например, проверка прав в JWT или middleware).
4. **Универсальный сервис**: Внедрен сервис массовой отмены/замены (`BookingReassignmentService`), учитывающий тайм-слоты и графики новых исполнителей. Заглушен флоу для Отпусков (`Vacation`) и Больничных (`SickLeave`).
5. **API Флоу**: Реализован эндпоинт `deactivate` с отдачей `409 Conflict` и последующей обработкой `resolution_payload` (включая перенос слотов и `target_employee_id`).

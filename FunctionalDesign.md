# Функциональный дизайн системы PetCare

## История изменений (Changelog)

**Актуализация 2026-03 (бронирования, escort и реальные слоты):**

- **Единая доменная логика бронирований:** Расчёт доступности слотов, предвалидация и создание бронирования переведены на единый backend service layer. Поиск слотов и фактическое создание записи используют один и тот же source of truth, чтобы убрать расхождения между "показали слот" и "не смогли забронировать".
- **Обязательный escort owner:** У каждого бронирования теперь обязательно хранится `escort_owner` - конкретный владелец питомца, который сопровождает питомца на визит. Если у питомца несколько владельцев и из контекста нельзя однозначно определить сопровождающего, backend возвращает structured-response с требованием явного выбора escort owner.
- **Immutable occupied duration:** В бронировании сохраняется `occupied_duration_minutes` - фактическая занятая длительность окна на момент создания. Это snapshot, который не пересчитывается задним числом при последующих изменениях карточки услуги.
- **Реальные bookable slots:** В поиске и выдаче слотов логика сведена к минимуму — показываются окна с учётом расписания филиала, расписания и отпусков/отсутствий сотрудника, а также самой длительности услуги. Проверки конфликтов самого сотрудника с другими бронированиями, конфликтов расписания питомца (pet owner conflicts), доступности сопровождающего (escort conflicts) и логистики (travel time) полностью вынесены на финальный этап предвалидации/создания бронирования для ускорения поиска.
- **Routing-only проверка логистики (при валидации):** Проверка возможности доехать между визитами переведена на routing integration в момент бронирования. Если routing API недоступен, валидация возвращает явную ошибку вместо "приближённого" решения.
- **Кастомная админка бронирований расширена:** В Django admin для бронирований теперь явно выводятся код записи, escort owner, список owners/co-owners питомца, provider/location, occupied duration; добавлены фильтры по escort/provider/location и сохранение через ту же доменную booking-логику.

**Исправления 2025-02 (вчера/сегодня):**

- **Единая таблица связи пользователь–провайдер:** Роли в провайдере хранятся только в **EmployeeProvider** (поле role). Модель **ProviderAdmin (Managed providers)** удалена из приложения users; в админке Django раздел Managed providers у пользователей убран. Дублирования с user_types нет.
- **Разделение прав админа провайдера и сотрудника:** В приложение «Админка провайдеров» входят все работники провайдера (активная связь через EmployeeProvider + привязка к филиалам через EmployeeLocationRole). Уровень доступа внутри приложения определяется ролью: владелец/менеджер/админ провайдера — настройки провайдера, локаций, расписаний, управление персоналом; сервисный работник — своё расписание, свои визиты, карточки питомцев по приёмам; техработник — только своё расписание.
- **Два типа работников в системе:** (1) Внутренние работники проекта — системные роли (system_admin, billing_manager и т.д.), доступ к общим разделам и биллингу платформы. (2) Работники провайдеров — роли в рамках учреждения/филиала (владелец, админ провайдера, руководитель филиала, сервисный работник, техработник), доступ только к своим организациям/филиалам.
- **Ролевая модель персонала провайдера:** Введена явная иерархия из шести ролей (см. раздел 1.2.4) с правилами «кто кого может уволить» и назначить.
- **Технические услуги (не оказываемые клиентам):** В каталог услуг добавлен тип «технические услуги» (например, клинер, санитар) — услуги не бронируются клиентами, используются только для расписания и учёта персонала филиала (раздел 1.2.5).

## Обзор системы

PetCare - это комплексная система управления ветеринарными услугами, которая объединяет владельцев домашних животных, поставщиков услуг (ветклиники, груминг-салоны) и специалистов в единую экосистему.

**Платформы:**

- Веб-приложение на React для всех пользователей
- Мобильное приложение (React Native) для владельцев питомцев и ситтеров
- Push-уведомления для мгновенного информирования о важных событиях

## 1. УПРАВЛЕНИЕ ПОЛЬЗОВАТЕЛЯМИ И РОЛЯМИ

### 1.1 Регистрация и аутентификация

**Обычная регистрация:**

**Шаг 1: Сбор данных**
Пользователь заполняет форму регистрации, вводя:

- Email (обязательно, уникальный)
- Пароль (обязательно)
- Имя (обязательно)
- Фамилию (обязательно)
- Номер телефона (обязательно, уникальный)

**Шаг 2: Проверка в реальном времени**
Система автоматически проверяет:

- Корректность формата email
- Корректность формата телефона
- Уникальность email (нет ли уже такого пользователя)
- Уникальность телефона (нет ли уже такого номера)

**Шаг 3: Временное хранение**
Если все проверки прошли успешно, данные сохраняются в сессии браузера (временное хранилище на сервере).

**Шаг 4: Финальная проверка и создание аккаунта**
Перед созданием аккаунта система еще раз проверяет уникальность всех полей (защита от гонок транзакций) и только после этого создает пользователя в базе данных.

**Шаг 5: Автоматический вход**
После успешной регистрации пользователь автоматически входит в систему и получает базовый набор прав.

**Регистрация через Google OAuth (с телефоном):**

**Шаг 1: Авторизация в Google**
Пользователь нажимает "Войти через Google", открывается окно Google, пользователь входит в свой аккаунт и разрешает доступ к данным.

**Шаг 2: Получение данных от Google**
Google возвращает данные пользователя:

- Email
- Имя и фамилию
- Номер телефона (если указан в Google аккаунте)
- Фото профиля

**Шаг 3: Проверка уникальности**
Система проверяет:

- Есть ли уже пользователь с таким email
- Есть ли уже пользователь с таким номером телефона

**Шаг 4: Создание аккаунта**
Если все проверки прошли успешно, создается новый пользователь в системе с данными от Google.

**Шаг 5: Автоматический вход**
Пользователь автоматически входит в систему и получает базовый набор прав.

**Регистрация через Google OAuth (без телефона):**

**Шаг 1-2: Аналогично предыдущему сценарию**
Пользователь авторизуется через Google, но Google не предоставляет номер телефона.

**Шаг 3: Проверка email**
Система проверяет только уникальность email (телефона нет).

**Шаг 4: Временное хранение**
Данные от Google сохраняются в сессии браузера с пометкой "нужен телефон".

**Шаг 5: Запрос номера телефона**
Пользователь перенаправляется на страницу, где его просят ввести номер телефона.

**Шаг 6: Проверка телефона**
Пользователь вводит номер телефона, система проверяет его уникальность.

**Шаг 7: Создание аккаунта**
Если телефон уникален, создается пользователь с данными от Google + введенным телефоном.

**Шаг 8: Автоматический вход**
Пользователь автоматически входит в систему и получает базовый набор прав.

**Процесс входа:**

**Обычный вход:**

1. Пользователь вводит email и пароль
2. Система проверяет учетные данные
3. Генерируется JWT-токен для сессии
4. Пользователь получает доступ к функциям согласно роли

**Вход через Google OAuth:**

1. Пользователь нажимает "Войти через Google"
2. Если аккаунт уже существует - происходит вход
3. Если аккаунта нет - запускается процесс регистрации через Google
4. Генерируется JWT-токен для сессии
5. Пользователь получает доступ к функциям согласно роли

**Две отдельные панели управления:**

- **Админка Django** (`/admin/`) — на MVP входят только персонал проекта (поддержка и админы): системные администраторы и суперпользователи. Биллинг-менеджеры, администраторы провайдеров и персонал провайдеров в Django admin не входят.
- **Приложение «Админка провайдеров» (Provider Admin App)** — отдельное SPA (отдельный домен/поддомен в проде, например `https://admin.petscare.com`). Вход разрешён любому пользователю, имеющему активную связь с хотя бы одним провайдером через EmployeeProvider (и/или привязку к филиалу). Внутри приложения доступ к разделам и данным ограничивается по ролям (владелец, менеджер, админ провайдера, руководитель филиала, сервисный работник, техработник). Вход — тот же JWT (email/пароль или Google OAuth через общий бэкенд).

**Обработка ошибок:**

**Если email уже существует:**

- Показывается ошибка "Пользователь с таким email уже существует"
- Предлагается войти в систему или восстановить пароль

**Если телефон уже существует:**

- Показывается ошибка "Пользователь с таким номером телефона уже существует"
- Предлагается ввести другой номер

**Если пользователь закрыл браузер:**

- Данные в сессии сохраняются на некоторое время
- При возврате пользователь может продолжить с того места, где остановился
- Если время истекло, процесс начинается заново

**Защита от проблем:**

- Все проверки уникальности происходят в атомарных транзакциях
- База данных блокируется на время создания пользователя
- Повторные проверки перед финальным созданием
- Уникальные ограничения в базе данных
- Очистка временных данных после успешного создания

**Создание пользователей через админку:**
Суперпользователь может создавать пользователей через системную админку Django, назначая им роли и права доступа вручную. Это позволяет администраторам системы управлять пользователями без необходимости прохождения процесса регистрации.

**Базовые права зарегистрированных пользователей:**
Все зарегистрированные пользователи автоматически получают базовый набор прав, включающий:

- Просмотр и редактирование собственного профиля
- Создание и управление питомцами
- Просмотр доступных услуг
- Бронирование услуг
- Просмотр истории бронирований
- Получение уведомлений о бронированиях

### 1.2 Система ролей и разрешений

**Общее описание:**
Система поддерживает мультиролевую архитектуру, где пользователи могут иметь несколько ролей одновременно. Права доступа определяются через предопределенные наборы разрешений, что обеспечивает консистентность и упрощает управление.

**Типы ролей и их назначение:**

**Автоматически назначаемые роли:**

- **basic_user** - Базовый пользователь (базовая роль)

  - Назначается автоматически при регистрации пользователя
  - Предоставляет минимальные права для работы с системой
  - НЕ дает доступ к админке
  - Требуется для создания питомцев, профилей ситтеров, заявок учреждений, принятия инвайтов
- **pet_owner** - Владелец питомца (функциональная роль)

  - Назначается автоматически при создании первого питомца
  - Предоставляет права для управления питомцами и бронирования услуг
- **pet_sitter** - Передержка питомцев (функциональная роль)

  - Назначается автоматически при создании профиля ситтера
  - Предоставляет права для оказания услуг передержки

**Роли, назначаемые через систему инвайтов:**

- **provider_manager / provider_admin / branch_manager / specialist**
  - Назначаются через единый invite API в зависимости от контекста (организация/филиал)
  - Используются для управления организацией, филиалом и персоналом филиала

**Роли, создаваемые через системную админку:**

- **institution_admin** - Администратор учреждения

  - Создается системным администратором
  - Предоставляет права управления учреждением и его сотрудниками
- **billing_manager** - Менеджер по биллингу

  - Создается системным администратором
  - Предоставляет права управления финансовыми операциями
- **booking_manager** - Менеджер по бронированиям

  - Создается системным администратором
  - Предоставляет права управления бронированиями и разрешения конфликтов
- **system_admin** - Системный администратор

  - Создается системным администратором
  - Предоставляет полный доступ ко всем функциям системы

**Алгоритм назначения ролей:**

1. **Базовая роль (basic_user)** назначается автоматически при регистрации пользователя
2. **Функциональные роли** назначаются автоматически при выполнении соответствующих действий:
   - `pet_owner` - при создании первого питомца
   - `pet_sitter` - при создании профиля ситтера
3. **Роли учреждений** назначаются через систему инвайтов администратором учреждения
4. **Системные роли** создаются и назначаются системным администратором
5. **Права доступа** определяются предопределенными наборами разрешений (см. раздел 1.2.3)
6. **Пользователь может иметь несколько ролей** одновременно с накоплением прав

**Архитектура системы прав:**

**1. Словарь разрешений с описаниями:**

- Централизованный словарь всех доступных разрешений в системе
- Каждое разрешение имеет техническое название и понятное описание
- Примеры: `users.add_user` → "Adding new users", `pets.view_pet` → "Viewing pets"
- Словарь используется для валидации и отображения прав в интерфейсе
- **Конфигурируется только через хардкод** - не настраивается пользователями через интерфейс

**2. Предопределенные наборы прав для ролей:**

- **system_admin** - полный доступ ко всем функциям системы
- **institution_admin** - управление учреждением и его сотрудниками
- **billing_manager** - управление финансовыми операциями
- **booking_manager** - управление бронированиями и разрешение конфликтов
- **employee** - базовые права для работы с клиентами
- **basic_user** - минимальные права для работы с системой (без доступа к админке)
- **pet_owner** - права владельца питомца
- **pet_sitter** - права для предоставления услуг передержки

**3. Выбор набора прав при создании роли:**

- В админке Django поле "Permission Set" с выбором из предопределенных наборов
- Поле "Custom Permissions" (используется только если выбран "custom")
- Автоматическое заполнение разрешений на основе выбранного набора
- Возможность создания кастомных ролей с индивидуальными правами

#### 1.2.1 Система инвайтов

**Общее описание:**
Система должна поддерживать приглашения пользователей на различные роли и права доступа. Инвайты генерируются авторизованными пользователями и содержат уникальные токены или QR-коды для быстрого принятия приглашений.

**Типы инвайтов:**

**Инвайты на роли в учреждениях:**

- `provider_manager` - приглашение менеджера организации
- `provider_admin` - приглашение администратора организации
- `branch_manager` - приглашение руководителя филиала
- `specialist` - приглашение сотрудника филиала

**Инвайты на роли в сервисе:**

- Для системных ролей (`billing_manager`, `system_admin` и др.) invite-поток не используется; назначение выполняется в админке.

**Инвайты на совладение питомцами:**

- **Добавление совладельца (invite)** - приглашение стать совладельцем питомца
- **Передача прав основного владельца (transfer)** - передача прав основного владельца другому пользователю

**Процесс создания инвайта:**

1. **Инициация** - авторизованный пользователь создает инвайт
2. **Генерация токена/QR-кода** - система создает уникальный идентификатор
3. **Установка срока действия** - инвайт имеет ограниченный срок действия
4. **Отправка уведомления** - система отправляет уведомление о создании инвайта

**Процесс принятия инвайта:**

1. **Получение токена/QR-кода** - пользователь получает токен или сканирует QR-код
2. **Верификация** - система проверяет валидность и срок действия инвайта
3. **Подтверждение** - пользователь подтверждает принятие роли/прав
4. **Активация** - система назначает пользователю соответствующую роль или права
5. **Уведомления** - отправляются уведомления инициатору и новому участнику

**Функциональные требования:**

- **Генерация токенов/QR-кодов** - уникальные идентификаторы для каждого инвайта
- **Валидация инвайтов** - проверка срока действия и статуса
- **Отслеживание статуса** - pending, accepted, expired, cancelled
- **История инвайтов** - сохранение всех созданных и принятых инвайтов
- **Отзыв инвайтов** - возможность отменить инвайт до его принятия

**Бизнес-правила:**

- Только авторизованные пользователи могут создавать соответствующие инвайты
- Пользователь может совмещать несколько ролей в рамках одного провайдера/филиала
- Инвайт на организационные и филиальные роли создаётся только для уже существующего пользователя (по email)
- Инвайт автоматически истекает по истечении срока действия
- Принятый инвайт нельзя отменить
- Токен/QR-код уникален для каждого инвайта и не может быть переиспользован
- Система предотвращает создание дублирующих инвайтов

**Интеграция:**

- Интеграция с системой ролей и разрешений
- Интеграция с системой уведомлений
- Интеграция с мобильным приложением для сканирования QR-кодов
- Интеграция с системой управления учреждениями и питомцами

#### 1.2.2 Передача полномочий менеджера учреждения

**Общее описание:**
Система должна поддерживать процесс передачи полномочий администратора учреждения другому пользователю. Это критически важная функция для обеспечения непрерывности управления учреждением при смене руководства или временном отсутствии основного администратора.

**Процесс инициации передачи:**

1. **Инициация** - текущий администратор учреждения инициирует процесс передачи полномочий
2. **Выбор кандидата** - администратор выбирает сотрудника для передачи полномочий
3. **Создание инвайта** - система создает специальный инвайт на передачу полномочий

**Процесс принятия передачи:**

1. **Получение уведомления** - кандидат получает уведомление о предложении полномочий
2. **Подтверждение** - кандидат подтверждает принятие полномочий
3. **Активация** - система активирует передачу полномочий
4. **Уведомления** - отправляются уведомления всем заинтересованным сторонам

**Функциональные требования:**

- **Создание инвайтов передачи** - генерация специальных инвайтов для передачи полномочий
- **Валидация кандидатов** - проверка прав кандидата на получение полномочий
- **Отслеживание статуса** - pending, accepted, rejected, expired, active, completed
- **История передач** - сохранение всех операций передачи полномочий

**Бизнес-правила:**

- Только администратор учреждения может инициировать передачу полномочий
- Кандидат должен быть сотрудником учреждения
- В учреждении должен быть как минимум один активный администратор
- Система предотвращает передачу полномочий неактивным пользователям
- Передача полномочий не влияет на другие роли пользователя в учреждении

**Интеграция:**

- Интеграция с системой ролей и разрешений учреждения
- Интеграция с системой уведомлений
- Интеграция с системой управления учреждениями

#### 1.2.3 Система управления правами и разрешениями

**Общее описание:**
Система должна поддерживать гибкое и управляемое распределение прав доступа для различных ролей пользователей. Права определяются через предопределенные наборы разрешений, что обеспечивает консистентность и упрощает управление.

**Архитектура системы прав:**

**1. Словарь разрешений с описаниями:**

- Централизованный словарь всех доступных разрешений в системе
- Каждое разрешение имеет техническое название и понятное описание
- Примеры: `users.add_user` → "Adding new users", `pets.view_pet` → "Viewing pets"
- Словарь используется для валидации и отображения прав в интерфейсе

**2. Предопределенные наборы прав для ролей:**

- **system_admin** - полный доступ ко всем функциям системы
- **institution_admin** - управление учреждением и его сотрудниками
- **billing_manager** - управление финансовыми операциями
- **booking_manager** - управление бронированиями и разрешение конфликтов
- **employee** - базовые права для работы с клиентами
- **basic_user** - минимальные права для работы с системой (без доступа к админке)
- **pet_owner** - права владельца питомца
- **pet_sitter** - права для предоставления услуг передержки

**3. Выбор набора прав при создании роли:**

- В админке Django поле "Permission Set" с выбором из предопределенных наборов
- Поле "Custom Permissions" (используется только если выбран "custom")
- Автоматическое заполнение разрешений на основе выбранного набора
- Возможность создания кастомных ролей с индивидуальными правами

**Функциональные требования:**

**Управление ролями:**

- Создание новых ролей через выбор предопределенного набора прав
- Редактирование существующих ролей с сохранением истории изменений
- Валидация прав при создании/изменении ролей
- Проверка конфликтов прав между ролями

**Проверка прав доступа:**

- Автоматическая проверка прав пользователя при доступе к функциям
- Кэширование прав пользователя для оптимизации производительности
- Логирование попыток доступа к запрещенным функциям
- Поддержка множественных ролей у одного пользователя

**Административный интерфейс:**

- Просмотр всех доступных разрешений с описаниями
- Создание и редактирование ролей через выбор наборов прав
- Предварительный просмотр прав роли перед сохранением
- Статистика использования ролей в системе

**Бизнес-правила:**

- Роль `basic_user` назначается автоматически при регистрации пользователя
- Роль `pet_owner` назначается автоматически при создании первого питомца
- Роль `pet_sitter` назначается автоматически при создании профиля ситтера
- Роли `employee` назначаются администратором учреждения через систему инвайтов
- Роли `institution_admin`, `billing_manager`, `booking_manager`, `system_admin` создаются через системную админку
- Только пользователи с ролью `basic_user`+ могут создавать питомцев, профили ситтеров, заявки учреждений, принимать инвайты
- Пользователь может иметь несколько ролей одновременно
- При изменении набора прав роли автоматически обновляются права всех пользователей с этой ролью

**Интеграция:**

- Интеграция с системой аутентификации Django
- Связь с системой аудита для отслеживания изменений прав
- Интеграция с API для проверки прав в REST эндпоинтах
- Поддержка декораторов для проверки прав в представлениях

**Безопасность:**

- Валидация всех разрешений при создании ролей
- Защита от создания ролей с несуществующими правами
- Аудит всех изменений в системе прав
- Резервное копирование конфигурации прав

**Единая связь пользователь–провайдер (без дублирования в user_types).**

- **Роли в провайдере** определяются связями в **EmployeeProvider** (owner / provider_manager / provider_admin / service_worker / technical_worker) и ролями на уровне филиала в **EmployeeLocationRole** (location_manager / service_worker / technical_worker). `User.user_types` используются как системные/глобальные роли и не являются единственным источником правды по привязке к конкретному провайдеру/филиалу.
- **Где действуют права** определяется отношением пользователя к провайдеру и/или филиалу:
  - Уровень провайдера: **Employee** + **EmployeeProvider** (employee→user, provider, роль, период работы) — привязка к провайдеру и права уровня организации.
  - Уровень филиала: **Employee** + **EmployeeLocationRole** (employee, provider_location, role, is_active, end_date) и **ProviderLocation.manager** — привязка к филиалу и роль в филиале.

**Доступ в приложение «Админка провайдеров»:**

- **Критерий входа:** пользователь имеет хотя бы одну активную связь с провайдером через **Employee** + **EmployeeProvider**. Доступ не ограничивается только ролью `provider_admin`: любой работник провайдера (в т.ч. сервисный работник, техработник) может войти в приложение, но с ограниченными правами.
- **Уровень доступа внутри приложения** определяется ролями (user_types и ролью в провайдере/филиале): владелец — вся информация и отчёты организации; менеджер/админ провайдера — управление персоналом и настройками; руководитель филиала — информация филиала; сервисный работник — своё расписание, свои визиты, карточки питомцев по приёмам; техработник — только своё расписание.

**Реализация в модели (текущая схема + целевая логика доступа):**

- **Единственный источник правды:** **EmployeeProvider** с полем **role** (owner | provider_manager | provider_admin | service_worker | technical_worker). employee, provider, role, start_date, end_date. Деактивация = end_date. Модель ProviderAdmin (Managed providers) удалена. Роли в user_types для персонала провайдера не дублируются.
- **Роли в филиале:** Employee + EmployeeLocationRole (employee, provider_location, role: location_manager | service_worker | technical_worker). ProviderLocation.manager — руководитель филиала (инвайт по email).

**Сервисный vs техработник:** различаются по типу услуг в локации (см. 1.2.5).

**Кто кого назначает:**

- **Владелец, менеджер провайдера и администратор** (лицо с доступом в админку провайдера) задаются при регистрации: в заявке провайдера (ProviderForm) указываются три обязательных email — owner_email, provider_manager_email, admin_email. Одно и то же лицо может совмещать роли (один email в нескольких полях). При одобрении заявки система создаёт записи EmployeeProvider с ролями owner, provider_manager, provider_admin и назначает пользователям соответствующие user_types; уведомления отправляются **по одному письму на каждый уникальный email** (не дублировать письма одному адресату).
- Дальнейшее назначение/замена менеджера и админа провайдера — владелец (деактивация + инвайт на освободившуюся роль).
- Руководителя филиала — владелец, менеджер или админ провайдера.
- Сервисных и техработников — руководитель филиала или владелец/менеджер/админ провайдера.

**Назначение прав работникам (страница персонала):**

- Права работникам учреждения (роли в филиале, привязка к услугам и т.д.) назначает **владелец**, **менеджер** или **админ провайдера** на странице персонала в приложении «Админка провайдеров».
- На странице персонала необходимо отображать **владельца провайдера** (лицо с ролью owner), а также **менеджера** и **админа** — отдельными ролями; один пользователь может быть указан в нескольких ролях.

**Роли персонала провайдера (только в EmployeeProvider, не в user_types):**

- Роли уровня провайдера (owner, provider_manager, provider_admin, service_worker, technical_worker) хранятся и проверяются только по таблице **EmployeeProvider**. В User.user_types они не дублируются. В админке Django раздел «Managed providers» у пользователей удалён; роли по организациям смотрят и правят в разделе Service Providers → Employee Providers.

**Кнопки «Пригласить менеджера» и «Я сам себе менеджер» (админка провайдера, блок «Владелец и менеджер»):**

При регистрации провайдера в заявке (ProviderForm) обязательны три email — owner_email, provider_manager_email, admin_email; при одобрении заявки всегда создаются владелец, менеджер и админ. Сценария «первого входа в пустую организацию» (нет ни владельца, ни менеджера, ни админа) в нормальном потоке нет.

- **а) Видимость (когда и кому кнопки показываются):**
  - Обе кнопки показываются **только владельцу провайдера** (is_owner в EmployeeProvider для этого провайдера).
  - **«Пригласить менеджера»:** видна только владельцу.
  - **«Я сам себе менеджер»:** видна только владельцу и только если **менеджер ≠ владелец** (т.е. текущий владелец ещё не является менеджером). Типичный случай: менеджера уволили, новый ещё не приглашён — владелец может временно назначить себя менеджером.
  - Менеджеру и админу провайдера (без роли владельца) кнопки **не показываются**.

- **б) Доступность (когда кнопки активны и действие разрешено):**
  - **«Пригласить менеджера»:** активна и разрешена только для **владельца**. Бэкенд должен проверять is_owner для этого провайдера; иначе — 403.
  - **«Я сам себе менеджер»:** активна и разрешена только для **владельца** и только когда владелец **ещё не является менеджером** (manager ≠ owner). Если владелец уже менеджер — кнопка неактивна или скрыта.

- **в) Недоступность (когда кнопки скрыты или неактивны):**
  - **Скрыты:** пользователь не владелец — обе кнопки не показывать. Менеджер и админ видят блок «Владелец и менеджер» (информация), но без кнопок назначения менеджера.
  - **«Пригласить менеджера»:** не показывать/не активировать, если пользователь не владелец.
  - **«Я сам себе менеджер»:** не показывать/не активировать, если пользователь не владелец или уже является менеджером (менеджер = владелец).

Для реализации на фронте: флаги в API ответа страницы организации — `can_invite_manager` (is_owner), `can_assign_self_manager` (is_owner и не is_manager), `is_owner`, `is_manager`.

**Один менеджер на провайдера (передача роли):**

- У организации может быть **только один** пользователь в роли менеджера (is_provider_manager). При принятии инвайта на роль менеджера или при самоназначении «Я сам себе менеджер» система в **одной транзакции**: (1) снимает роль менеджера у всех текущих (is_provider_manager=False, is_manager=False у остальных активных EmployeeProvider по этому провайдеру); (2) назначает менеджером принявшего инвайт или текущего пользователя. Записи EmployeeProvider у «бывших» менеджеров не завершаются (end_date не ставится) — человек остаётся в организации (владелец, админ и т.д.), у него лишь снимается флаг менеджера.

**Унифицированная система инвайтов (приложение `invites`, модель `Invite`):**

- В продукте используется **invite-only** сценарий для назначения ролей в организации/филиале: `provider_manager`, `provider_admin`, `branch_manager`, `specialist` (владелец провайдера через invites не назначается).
- **Типы инвайтов (invite_type):** `provider_manager` — менеджер организации; `provider_admin` — админ организации; `branch_manager` — руководитель филиала (один на локацию); `specialist` — сотрудник в персонал филиала (много на локацию); `pet_co_owner` — совладелец питомца; `pet_transfer` — передача прав основного владельца питомца.
- Единый API: **`/api/v1/invites/`** — создание (POST с invite_type, email, provider_id | provider_location_id | pet_id), список (GET), приём (POST `/api/v1/invites/accept/` с token), отклонение (POST `/api/v1/invites/decline/`), отмена (POST `/api/v1/invites/<id>/cancel/`), по токену (GET `token/<token>/`), QR (GET `<id>/qr-code/`), ожидающие (GET `pending/`).
- **EmployeeJoinRequest** не входит в целевой процесс. Сценарий «пользователь просится в организацию» исключён из требований; соответствующая модель/API рассматриваются как legacy и выводятся из системы.

**Инвайты по локациям (ответ БА):**

- **branch_manager** (в Invite) — приглашение на роль **руководителя конкретного филиала**. Один руководитель на одну локацию; сценарий «пригласить руководителя филиала по email → пользователь вводит код → становится руководителем этой локации».
- **specialist** (в Invite) — приглашение **сотрудника в персонал конкретного филиала**. В один филиал можно пригласить много сотрудников; сценарий «пригласить в филиал по email → пользователь вводит код → привязывается как сотрудник к провайдеру и этой локации».

**Почему два типа инвайта по локации, а не один?** Роль и ограничения разные: у филиала один руководитель (уникальная роль, ограничение «один на локацию») и много сотрудников (персонал без такого ограничения). В единой модели Invite это отражено разными значениями invite_type и разной логикой при приёме.

**Где смотреть в Django admin:** раздел **Invites** в кастомной админке (custom_admin_site); приложение `invites`, модель `Invite`.

#### 1.2.5 Технические услуги (не оказываемые клиентам)

**Общее описание:**
Технические услуги — это работы, которые не бронируются клиентами (уборка, санитарные процедуры и т.п.). Они используются для расписания и учёта персонала филиала (техработники).

**Модель:**

- В каталоге услуг (Service) у каждой услуги есть признак **«оказывается клиенту»** (client-facing). У технических услуг этот признак выключен (`is_client_facing = False`).
- Справочник технических услуг — глобальный: провайдеры/филиалы выбирают из общего списка (единообразие, отчётность). При необходимости в список добавляются новые позиции (системой или поддержкой).
- Сотрудник привязывается к локации и услугам (в т.ч. техническим) через связь «сотрудник — локация — услуга». Если в локации у сотрудника есть только технические услуги — он считается техработником в этой локации; иначе — сервисный работник.

**Поведение в системе:**

- Технические услуги не отображаются клиентам при выборе услуги для бронирования.
- Они участвуют в расписании сотрудников и в отчётности по загрузке персонала филиала.

#### 1.2.6 Актуализация реализации: персонал провайдера, инвайты, отсутствия и «шахматка» (2026-02)

Этот подпункт фиксирует фактическую реализованную логику и имеет приоритет над более общими/устаревшими формулировками выше.

**Персонал провайдера и филиала (фактическая модель):**

- Привязка пользователя к организации хранится в `EmployeeProvider` (роли `owner`, `provider_manager`, `provider_admin`, `service_worker`, `technical_worker` + период работы `start_date/end_date`).
- Привязка к филиалу и локальная активность хранится в `EmployeeLocationRole` (роли `location_manager`, `service_worker`, `technical_worker`, флаг `is_active`, дата `end_date`).
- Для одного филиала действует ограничение «один руководитель филиала» (уникальность `location_manager` на `provider_location`).
- Деактивация сотрудника в филиале выполняется как soft-delete на уровне `EmployeeLocationRole` (`is_active=False`, `end_date=now`) и не обязательно завершает связь `EmployeeProvider`.

**Унифицированные инвайты (реализовано в приложении `invites`):**

- Все приглашения сведены в единую модель `Invite` с типами: `provider_manager`, `provider_admin`, `branch_manager`, `specialist`, `pet_co_owner`, `pet_transfer`.
- Для приёма используется 6-значный код (`token`), срок действия по умолчанию 7 дней, статусы: `pending`, `accepted`, `declined`, `expired`, `cancelled`.
- Единый API: `POST/GET /api/v1/invites/`, `POST /api/v1/invites/accept/`, `POST /api/v1/invites/decline/`, `POST /api/v1/invites/<id>/cancel/`, `GET /api/v1/invites/pending/`, `GET /api/v1/invites/token/<token>/`, `GET /api/v1/invites/<id>/qr-code/`.
- Инвайт сотрудника филиала (`specialist`) и руководителя филиала (`branch_manager`) создаётся только для существующего пользователя по email (если пользователь не зарегистрирован, API возвращает валидационную ошибку).
- При приёме инвайта система автоматически создаёт/обновляет `Employee`, `EmployeeProvider`, привязку к филиалу и соответствующие пользовательские роли (`provider_manager`, `provider_admin`, `branch_manager`, `specialist`).

**Оффбординг/реактивация сотрудника филиала:**

- `PATCH /api/v1/provider-locations/<location_pk>/staff/<employee_id>/deactivate/`:
  - при наличии будущих бронирований и отсутствии стратегии разрешения возвращает `409` (`conflict_type=has_future_bookings`) со списком конфликтующих записей;
  - поддерживает стратегии `resolution_action=cancel|reassign` (с `target_employee_id` для `reassign`);
  - после обработки бронирований деактивирует сотрудника в филиале (`EmployeeLocationRole.is_active=False`).
- `PATCH /api/v1/provider-locations/<location_pk>/staff/<employee_id>/reactivate/` реактивирует сотрудника в филиале (`is_active=True`, `end_date=null`).

**Отпуска и больничные (в т.ч. отмена и досрочное завершение):**

- Создание: `POST /api/v1/employees/<employee_id>/vacations/` и `POST /api/v1/employees/<employee_id>/sick-leaves/`.
- Для отпусков/больничных реализованы:
  - проверка пересечения с уже существующими отсутствиями (учитываются локальные и глобальные записи);
  - обработка конфликтов с будущими бронированиями в том же формате `409` + `resolution_action` (cancel/reassign/granular).
- Отмена отсутствия целиком: `DELETE /api/v1/vacations/<id>/` и `DELETE /api/v1/sick-leaves/<id>/`.
- Досрочный выход (прерывание отпуска/больничного): `PATCH /api/v1/vacations/<id>/` или `PATCH /api/v1/sick-leaves/<id>/` с изменением `end_date` (дата выбирается пользователем в UI и валидируется в пределах периода отсутствия).
- Валидация сериализаторов поддерживает частичные обновления (PATCH) без `KeyError`: отсутствующие поля берутся из `instance`.

**«Шахматка» (график работы персонала) в админке провайдера:**

- Данные для табличного вида загружаются через `GET /api/v1/provider-locations/<location_id>/aggregated-schedule/?start_date=YYYY-MM-DD&end_date=YYYY-MM-DD`.
- Ответ агрегирует для каждого сотрудника:
  - недельный паттерн смен (`schedule`, day_of_week, start/end/break),
  - отсутствия (`vacation`, `sick_leave`) с датами начала/окончания и статусами.
- На UI реализованы действия из ячейки/иконки отсутствия: оформить отсутствие, удалить отпуск/больничный, оформить досрочный выход с выбором даты, а также периодная навигация по доске.
- После создания/обновления/удаления отсутствия доска обновляется автоматически через рефетч aggregated schedule.

### 1.3 Управление профилем пользователя

**Управление безопасностью:**

- Изменение пароля (стандартная функция Django)
- Выход из системы (завершение сессии)

**Деактивация аккаунта:**
Пользователь может самостоятельно деактивировать свой аккаунт, что временно отключает доступ к системе с возможностью восстановления в будущем.

**Ограничения на деактивацию:**

- **Служебные роли** - пользователи с ролями `system_admin`, `billing_manager`, `booking_manager`, `employee` не могут деактивировать аккаунт
- **Администратор учреждения** - не может деактивировать аккаунт (должен передать права или обратиться в службу поддержки)
- **Активные услуги** - пользователи с активными услугами или передержками не могут деактивировать аккаунт (кроме пользователей только с ролью `basic_user`)
- **Пользователи только с ролью `basic_user`** - могут деактивировать аккаунт сразу (не имеют активных услуг)

**Процесс деактивации:**

1. **Проверка прав и ролей** - система проверяет, разрешена ли деактивация для данного пользователя
2. **Проверка активных услуг** - система проверяет наличие активных услуг или передержек (кроме `basic_user`)
3. **Предупреждение о последствиях** - если пользователь является основным владельцем питомцев:
   - Система предупреждает о деактивации всех питомцев
   - Система предупреждает, что все совладельцы потеряют доступ к питомцу
4. **Отмена будущих услуг** - если у пользователя есть запланированные услуги в будущем:
   - Система автоматически отменяет все будущие бронирования
   - Отправляются уведомления поставщикам услуг об отмене
   - Пользователь получает уведомление об отмене услуг
5. **Подтверждение деактивации** - пользователь должен подтвердить деактивацию аккаунта
6. **Обработка питомцев** - если пользователь является основным владельцем питомцев:
   - Все питомцы деактивируются
   - Все связи с совладельцами удаляются
   - Совладельцы без других питомцев получают роль `basic_user`
7. **Обработка ролей владельца и ситтера** - если пользователь имеет роли `pet_owner` или `pet_sitter`:
   - Роль `pet_owner` удаляется (так как питомцы деактивированы)
   - Роль `pet_sitter` удаляется (профиль ситтера деактивируется)
   - Пользователь получает только базовую роль `basic_user`
8. **Деактивация пользователя** - аккаунт помечается как неактивный
9. **Анонимизация данных** - при включенной настройке данные пользователя анонимизируются:
   - Email заменяется на `deactivated_{id}@anonymized.local`
   - Номер телефона заменяется на уникальный анонимный номер
   - Имя и фамилия заменяются на "Deactivated User"
10. **Уведомления** - отправляются уведомления пользователю и бывшим совладельцам

**Восстановление аккаунта:**

- Деактивированный аккаунт может быть восстановлен администратором системы
- При восстановлении все данные и связи восстанавливаются
- Пользователь получает уведомление о восстановлении аккаунта

**Сохранение данных:**

- Деактивированные аккаунты сохраняются в системе для статистики и аудита
- Данные могут быть анонимизированы, но не удаляются
- Система поддерживает возможность восстановления аккаунта в любое время

**Профиль ситтера (дополнительные поля):**

1. Ситтер указывает адрес для оказания услуг (через связь с моделью Address)
2. Указывает опыт работы с животными (годы)
3. Описывает услуги передержки
4. Указывает типы питомцев и максимальное количество
5. Настраивает период доступности и расстояние

## 2. УПРАВЛЕНИЕ ПИТОМЦАМИ И ВЛАДЕЛЬЦАМИ

### 2.1 Множественные владельцы питомцев

- **Основной владелец (main_owner)**: Один пользователь с полными правами управления питомцем
- **Совладельцы (owners)**: Множественные пользователи с правами доступа к информации о питомце
- **Система инвайтов (Invite)**: Приглашение совладельцев или передача прав основного владельца через единый API `/api/v1/invites/` (типы `pet_co_owner`, `pet_transfer`), email с 6-значным кодом
- **Типы инвайтов по питомцу**: `pet_co_owner` (добавление совладельца) и `pet_transfer` (передача прав основного владельца)
- **QR-коды**: Генерация QR-кодов для быстрого подтверждения инвайтов

### 2.2 Создание профиля питомца

**Алгоритм добавления питомца:**

1. Владелец нажимает "Добавить питомца"
2. Заполняет обязательные поля: имя, тип (PetType), порода (Breed), дата рождения
3. Опционально: вес, описание, особые потребности, медицинские условия, фото
4. Система автоматически устанавливает текущего пользователя как основного владельца (main_owner)
5. Система добавляет основного владельца в список владельцев (owners)
6. Система создает уникальный ID питомца
7. Питомец добавляется в список владельца

**Валидация данных:**

- Проверка корректности даты рождения (не может быть в будущем)
- Валидация веса (должен быть положительным числом)
- Проверка размера загружаемого фото (стандартная валидация Django)
- Обязательное указание основного владельца (main_owner не может быть пустым)
- Проверка наличия хотя бы одного владельца в списке owners
- Проверка, что основной владелец входит в список владельцев

### 2.3 Фильтрация и поиск питомцев

**Текущая реализация:**

- Базовая фильтрация по владельцам (только питомцы текущего пользователя)
- Поиск по имени питомца
- Сортировка по дате создания

**Расширенная фильтрация (PetFilter):**

- **Фильтрация по типу и породе:**

  - `pet_type` - фильтр по типу питомца
  - `pet_type_code` - фильтр по коду типа
  - `breed` - фильтр по породе
  - `breed_code` - фильтр по коду породы
- **Фильтрация по возрасту:**

  - `age_min` - минимальный возраст в годах
  - `age_max` - максимальный возраст в годах
  - `age_range` - диапазон возраста (например, 2-5 лет)
- **Фильтрация по весу:**

  - `weight_min` - минимальный вес в кг
  - `weight_max` - максимальный вес в кг
  - `weight_range` - диапазон веса (например, 5-15 кг)
- **Фильтрация по геолокации:**

  - `location_lat` - широта для поиска
  - `location_lng` - долгота для поиска
  - `radius_km` - радиус поиска в километрах
  - Использует PostGIS для точного расчета расстояний
- **Фильтрация по медицинским условиям:**

  - `has_medical_conditions` - наличие медицинских условий
  - `medical_condition` - поиск по конкретному условию
- **Фильтрация по особым потребностям:**

  - `has_special_needs` - наличие особых потребностей
  - `special_need` - поиск по конкретной потребности
- **Фильтрация по датам:**

  - `created_after/before` - дата создания питомца
  - `updated_after/before` - дата обновления
  - `last_visit_after/before` - дата последнего посещения
- **Фильтрация по владельцам:**

  - `owner` - фильтр по ID владельца
  - `main_owner` - фильтр по основному владельцу
- **Фильтрация по статусу:**

  - `is_active` - активные/неактивные питомцы

**API эндпоинты:**

- `GET /api/pets/?age_min=2&age_max=5&weight_range=5,15` - фильтрация по возрасту и весу
- `GET /api/pets/?location_lat=55.7558&location_lng=37.6176&radius_km=10` - поиск по геолокации
- `GET /api/pets/?has_medical_conditions=true&medical_condition=диабет` - поиск по медицинским условиям

### 2.4 Медицинская карта питомца

**Структура карты:**

- Основная информация о питомце (Pet): имя, тип, порода, дата рождения, вес, описание, особые потребности, медицинские условия, фото
- Медицинские записи (MedicalRecord): заголовок, описание, дата, вложения, дата следующего посещения
- Записи о процедурах (PetRecord): выполненные услуги с привязкой к учреждению, сотруднику, описанием, результатами, рекомендациями, серийным номером
- Файлы с метаданными (PetRecordFile): документы с расширенной информацией:
  - Привязка к питомцу, медицинским записям или записям о процедурах
  - Метаданные: дата выдачи, срок действия, номер документа, орган выдачи
  - Типизация через систему типов документов
  - Валидация обязательных полей согласно типу документа
  - Отслеживание сроков с автоматическим определением истечения
- Типы документов (DocumentType): настройка обязательных полей для разных типов документов

**Доступ к медицинской карте:**

- **Владельцы питомца** (`pet__owners`) имеют полный доступ к медицинской карте (просмотр, создание, редактирование)
- **Временный доступ** предоставляется через модель `PetAccess`:
  - Владелец может дать временный доступ другому пользователю (например, ситтеру во время передержки)
  - Доступ имеет срок действия (`expires_at`)
  - Настраиваемые права доступа (`permissions`): чтение, бронирование, редактирование
- **Сотрудники учреждений** могут создавать записи для питомцев, которым они оказывают услуги
- **Системные администраторы** имеют полный доступ

**Алгоритм ведения карты:**

1. Владелец питомца или специалист учреждения открывает карту питомца
2. Добавляет запись о процедуре с описанием и результатами
3. Прикрепляет документы (файлы) с метаданными
4. Указывает рекомендации и дату следующего посещения
5. Система сохраняет запись с временной меткой

### 2.5 Управление владельцами питомца

**Основной владелец:**

- Создает профиль питомца
- Имеет полные права на управление
- Может приглашать совладельцев через систему инвайтов
- Может передавать права основного владельца
- Получает все уведомления о питомце

**Совладельцы:**

- Приглашаются основным владельцем через систему инвайтов
- Имеют права доступа к информации о питомце
- Могут просматривать медицинскую карту и записи
- Получают уведомления о важных событиях
- НЕ могут изменять основную информацию или удалять питомца

**Алгоритм назначения совладельца:**

1. Основной владелец создает инвайт типа pet_co_owner (добавление совладельца) через POST /api/v1/invites/
2. Вводит email потенциального совладельца
3. Система отправляет email с токеном для подтверждения
4. Пользователь принимает инвайт через ссылку или QR-код
5. Получает доступ к профилю питомца

**Удаление совладельца:**

1. Основной владелец открывает список совладельцев
2. Выбирает совладельца для удаления
3. Подтверждает действие
4. Совладелец теряет доступ к питомцу
5. Отправляется уведомление об удалении

**Примечание:** Совладелец также может самостоятельно удалить себя из списка владельцев (см. раздел 2.6).

### 2.5.1 Передача питомца другому владельцу

**Алгоритм передачи:**

1. Основной владелец создает инвайт типа 'transfer'
2. Вводит email нового основного владельца
3. Система отправляет email с токеном для подтверждения
4. Новый владелец принимает инвайт
5. Права основного владельца передаются
6. Старый владелец становится совладельцем

### 2.5.2 Система обработки недееспособности владельцев питомцев

**Общее описание:**
Система автоматически отслеживает неактивных **основных владельцев** питомцев и обрабатывает ситуации их недееспособности. При обнаружении проблем система выполняет автоматические действия для обеспечения благополучия питомцев.

**Функциональные требования:**

**Автоматическое обнаружение неактивности основного владельца:**

- Мониторинг последнего входа основного владельца
- Создание записей о недееспособности **основного владельца** при превышении порога неактивности
- Отправка уведомления **основному владельцу** с просьбой подтвердить статус питомца
- Установка дедлайна для подтверждения (настраиваемый параметр, по умолчанию 30 дней)

**Обработка отчетов от владельцев:**

- **Только основной владелец** может сообщить о потере/смерти питомца → **прямое удаление питомца**
- **Только совладельцы** могут сообщить о недееспособности **основного владельца** → отправка уведомления основному владельцу для подтверждения
- Создание записей о недееспособности **основного владельца** с указанием причины

**Система подтверждений и дедлайнов:**

- **При автоматическом обнаружении неактивности:** отправка уведомления основному владельцу о необходимости подтверждения статуса питомца
- **При отчете совладельца о недееспособности:** отправка уведомления основному владельцу для опровержения/подтверждения
- **Только основной владелец** может подтвердить статус питомца:
  - "Питомец в порядке" → закрытие дела о недееспособности **основного владельца**
  - "Питомец потерян/умер" → удаление питомца из системы
- Если основной владелец не подтверждает в течение дедлайна → автоматическое действие

**Автоматические действия по истечении дедлайна:**

- **Если питомец в порядке:** закрытие дела о недееспособности **основного владельца**
- **Если питомец потерян/умер:** автоматическое удаление питомца из системы
- **Если основной владелец недееспособен:** автоматическое назначение совладельца основным владельцем
- **Если нет совладельцев:** автоматическое удаление питомца

**Уведомления:**

- Email-уведомления **основному владельцу** о необходимости подтверждения статуса питомца (только при автоматическом обнаружении или отчете совладельца)
- Уведомления о предупреждении приближающегося дедлайна
- Уведомления о выполненных автоматических действиях
- Уведомления о разрешении ситуации

**Бизнес-правила:**

- **Только основной владелец** может сообщить о потере/смерти питомца → **прямое удаление без подтверждения**
- **Только совладельцы** могут сообщать о недееспособности **основного владельца**
- **Только основной владелец** может подтвердить статус питомца (при автоматическом обнаружении или отчете совладельца)
- Автоматические действия выполняются только после истечения дедлайна
- При назначении нового основного владельца выбирается наиболее подходящий совладелец
- Все действия логируются и отслеживаются

**Интеграция:**

- Периодические задачи Celery для автоматического обнаружения и обработки
- Интеграция с системой уведомлений для отправки email
- Связь с системой блокировок для получения настроек дедлайнов

**Статусы записей о недееспособности основного владельца:**

- `pending_confirmation` - ожидание подтверждения статуса питомца **основным владельцем**
- `confirmed_incapacity` - недееспособность **основного владельца** подтверждена
- `pet_lost` - питомец потерян/умер
- `resolved` - ситуация разрешена
- `auto_deleted` - питомец автоматически удален
- `coowner_assigned` - назначен новый основной владелец

**Типы потоков обработки:**

- `automatic_detection` - автоматическое обнаружение неактивности **основного владельца**
- `coowner_report_pet_lost` - отчет совладельца о потере питомца (**ОШИБКА В КОДЕ**)
- `coowner_report_owner_incapacity` - отчет совладельца о недееспособности **основного владельца**

**Управление совладельцами:**

- Основной владелец может исключить любого совладельца из списка владельцев
- Совладелец может самостоятельно удалить себя из списка владельцев
- При удалении совладельца отправляются уведомления основному владельцу

**Примечание:** В текущей реализации кода есть **критические ошибки**:

1. Любой владелец может сообщить о потере питомца (должен быть только основной владелец)
2. Любой владелец может подтвердить статус питомца (должен быть только основной владелец)

Код требует исправления согласно описанной бизнес-логике.

### 2.6 Самостоятельное снятие обязанностей совладельца

**Алгоритм самостоятельного снятия:**

1. Совладелец открывает профиль питомца
2. Нажимает "Покинуть питомца" или "Удалить себя из совладельцев"
3. Система проверяет, что он не является основным владельцем
4. Если он не основной владелец - совладелец удаляется из списка владельцев
5. Если он основной владелец - система показывает сообщение о необходимости передачи прав
6. Отправляются уведомления основному владельцу и другим совладельцам

**Ограничения самостоятельного снятия:**

- Совладелец может удалить себя только если он не является основным владелецем
- Если совладелец является единственным владельцем, он должен сначала передать права
- При снятии обязанностей отправляются уведомления всем заинтересованным сторонам

**Уведомления:**

- Основной владелец получает уведомление о том, что совладелец покинул питомца
- Другие совладельцы получают уведомление об изменении состава владельцев
- Система ведет историю изменений владельцев для аудита

### 2.7 Удаление питомца

**Алгоритм удаления питомца:**

1. **Только основной владелец (main_owner)** может удалить питомца
2. Основной владелец открывает профиль питомца
3. Нажимает "Удалить питомца"
4. Система запрашивает подтверждение и причину удаления:
   - Смерть питомца
   - Потеря питомца
   - Передача в приют/другому владельцу
   - Другая причина (с описанием)
5. Если питомец имеет нескольких владельцев - система уведомляет всех совладельцев
6. Питомец удаляется из системы с сохранением истории в архиве

**Ограничения:**

- Только основной владелец (main_owner) или системный администратор могут удалить питомца
- Совладельцы НЕ могут удалить питомца
- При попытке удаления совладельцем система возвращает ошибку доступа
- Система требует указания причины удаления
- При удалении отправляются уведомления всем совладельцам

3. Система создает запись `PetOwnerIncapacity` с типом `coowner_report_pet_lost`
4. Отправляются уведомления всем владельцам питомца
5. Устанавливается дедлайн подтверждения (30 дней)
6. Если нет подтверждения в течение дедлайна, питомец автоматически удаляется

#### **ФЛОУ 3: Совладелец сообщает о недееспособности основного владельца**

**Триггер:** Совладелец создает отчет "Основной владелец недееспособен"
**Цель:** Назначить нового основного владельца
**Действие:** Уведомление всех, назначение совладельца основным через месяц

**Алгоритм:**

1. Совладелец использует API эндпоинт `/api/incapacity/report_owner_incapacity/`
2. Указывает ID питомца и причину недееспособности (обязательно)
3. Система создает запись `PetOwnerIncapacity` с типом `coowner_report_owner_incapacity`
4. Отправляются уведомления всем владельцам питомца
5. Устанавливается дедлайн подтверждения (30 дней)
6. Если нет подтверждения в течение дедлайна, система автоматически назначает совладельца основным владельцем

#### **Автоматические действия**

**Настройки автоматических действий (глобальные параметры):**

- `inactive_owner_threshold_days` - порог неактивности владельца (по умолчанию 180 дней)
- `pet_confirmation_deadline_days` - дедлайн подтверждения статуса питомца (по умолчанию 30 дней)
- `auto_delete_unconfirmed_pets` - автоматическое удаление неподтвержденных питомцев (по умолчанию True)
- `auto_assign_coowner_as_main` - автоматическое назначение совладельца основным (по умолчанию True)
- `coowner_assignment_priority` - приоритет назначения совладельца ('oldest', 'newest', 'random')

**Логика автоматических действий:**

1. **Для ФЛОУ 1 и 2** (утрата питомца):

   - Если `auto_delete_unconfirmed_pets = True` → удалить питомца
   - Если `auto_delete_unconfirmed_pets = False` → ничего не делать
2. **Для ФЛОУ 3** (недееспособность владельца):

   - Если `auto_assign_coowner_as_main = True` → назначить совладельца основным владельцем
   - Приоритет назначения определяется параметром `coowner_assignment_priority`

#### **API эндпоинты**

**Основные эндпоинты:**

- `GET /api/incapacity/` - список случаев недееспособности
- `POST /api/incapacity/report_pet_lost/` - сообщить о потере питомца
- `POST /api/incapacity/report_owner_incapacity/` - сообщить о недееспособности владельца
- `POST /api/incapacity/{id}/confirm_pet_status/` - подтвердить статус питомца
- `GET /api/incapacity/my_cases/` - мои случаи недееспособности
- `GET /api/incapacity/{id}/notifications/` - уведомления по случаю

**Эндпоинты уведомлений:**

- `GET /api/incapacity-notifications/` - список уведомлений пользователя
- `POST /api/incapacity-notifications/{id}/mark_as_read/` - отметить как прочитанное

#### **Периодические задачи**

**Автоматические задачи (Celery):**

- `check_inactive_owners_task` - еженедельная проверка неактивных владельцев
- `process_deadline_actions_task` - ежедневная обработка действий по истечении дедлайнов
- `send_incapacity_notifications_task` - отправка уведомлений
- `cleanup_old_incapacity_records_task` - очистка старых записей (ежемесячно)

#### **Модели данных**

**PetOwnerIncapacity:**

- Связь с питомцем и основным владельцем
- Тип флоу (автоматическое обнаружение, отчет о потере, отчет о недееспособности)
- Статус (ожидание подтверждения, подтверждено, разрешено, автоматически удалено, назначен совладелец)

- Автоматическое действие и новый основной владелец
- История уведомлений

**PetIncapacityNotification:**

- Связь с записью о недееспособности
- Тип уведомления (запрос подтверждения, предупреждение о дедлайне, уведомление о действии)
- Получатель и статус отправки
- Содержание уведомления

#### **Уведомления**

**Типы уведомлений:**

- `confirmation_request` - запрос подтверждения статуса питомца
- `deadline_warning` - предупреждение о приближении дедлайна
- `auto_action_notification` - уведомление о выполненных автоматических действиях
- `resolution_notification` - уведомление о разрешении случая

**Каналы доставки:**

- Email уведомления через Gmail API
- Push-уведомления (планируется)
- Внутренние уведомления в системе

#### **Безопасность и права доступа**

**Проверки прав доступа:**

- Только владельцы питомца могут подтверждать его статус
- Только совладельцы могут сообщать о недееспособности основного владельца
- Любой владелец может сообщить о потере питомца
- Администраторы видят все случаи недееспособности

**Защита от злоупотреблений:**

- Проверка на существование активных случаев недееспособности
- Валидация входных данных
- Логирование всех действий
- Транзакционная обработка критических операций

### 2.8 Система документов питомцев

**Базовые правила**

- В каталоге только 5 типов документов.
- `Формат файла` не является типом документа.
- `Другое` не показываем.
- Из `карточки питомца` загружает только владелец / совладелец.
- Из `визита` или `дополнения к визиту` загружает только провайдер.
- Право `записи/удаления` зависит от контекста загрузки, не от самого типа документа.

**Типы документов**

| Тип документа | Где доступен для загрузки | Что сюда входит | Обязательные поля | Необязательные, но полезные поля |
|---|---|---|---|---|
| `Паспорт / идентификация` | только карточка питомца | ветпаспорт, данные чипа, регистрационный документ | `файл` | `название`, `номер документа`, `дата документа`, `комментарий` |
| `Ветеринарная справка / сертификат` | только карточка питомца | ветеринарные справки, сертификаты, официальные бумаги | `файл`, `дата документа` | `срок действия`, `номер документа`, `кем выдан`, `комментарий` |
| `Анализы` | карточка питомца, визит, дополнение к визиту | кровь, моча, кал, ПЦР, посев, цитология, гистология и т.п. | `файл`, `дата исследования` | `название`, `лаборатория / источник`, `комментарий` |
| `Диагностика` | карточка питомца, визит, дополнение к визиту | рентген, УЗИ, КТ, МРТ, ЭХО, заключения по исследованиям | `файл`, `дата исследования` | `название`, `клиника / источник`, `комментарий` |
| `Выписка / назначения врача` | карточка питомца, визит, дополнение к визиту | выписка, эпикриз, назначения, рецепт, рекомендации, заключение другой клиники | `файл`, `дата документа` | `название`, `клиника / источник`, `комментарий` |

**Права по контексту**

| Контекст | Кто создаёт | Кто удаляет |
|---|---|---|
| Документ в карточке питомца | только владелец / совладелец | только владелец / совладелец |
| Документ в визите | сотрудник провайдера / админ провайдера по своим правам | сотрудник провайдера / админ провайдера по своим правам |
| Документ в дополнении к визиту | сотрудник провайдера / админ провайдера по своим правам | сотрудник провайдера / админ провайдера по своим правам |

**Что важно отдельно**

- Провайдер не должен писать и удалять документы напрямую в карточке питомца.
- Владелец не должен удалять клинические документы, созданные провайдером в визите.
- Для UX стоит оставить одно общее текстовое поле `Название`, чтобы пользователь мог уточнить смысл документа без раздувания типов.
  Примеры:
  - для `Анализы`: `Биохимия крови`
  - для `Диагностика`: `МРТ головы`
  - для `Выписка / назначения врача`: `Выписка из клиники X`

**Форматы файлов**

- `PDF`
- `JPG/JPEG`
- `PNG`

Видео в `v1` не поддерживается.

## 3. Система передержки питомцев

### 3.1 Поиск ситтеров

**Алгоритм поиска:**

1. Владелец указывает параметры поиска:
   - Координаты или местоположение (геолокация устройства)
   - Радиус поиска в км
   - Тип питомца
   - Даты передержки
   - Тип компенсации (платная/бесплатная)
   - Минимальный рейтинг ситтера
   - Максимальная цена (для платных услуг)
2. Система фильтрует ситтеров по критериям:
   - Расстояние от указанной точки
   - Типы питомцев, с которыми работает ситтер
   - Доступность в указанные даты
   - Тип компенсации
   - Рейтинг ситтера
   - Цена услуг
3. Возвращает отсортированный список подходящих ситтеров

**Фильтры поиска:**

- По расстоянию от указанной точки
- По типам питомцев
- По датам доступности ситтера
- По типу компенсации (платная/бесплатная)
- По рейтингу ситтера
- По цене услуг
- По расписанию работы ситтера

**Результат поиска:**

- Список ситтеров с информацией о расстоянии, рейтинге, услугах
- Возможность просмотра профиля ситтера
- Возможность создания объявления о передержке для выбранного ситтера

### 3.2 Создание объявления о передержке

**Алгоритм создания объявления:**

1. Владелец питомца создает объявление (PetSittingAd) после поиска ситтеров
2. Указывает питомца, даты передержки, описание
3. Выбирает тип компенсации (платная/бесплатная)
4. Указывает максимальное расстояние для поиска ситтеров
5. Система публикует объявление и уведомляет подходящих ситтеров

**Объявление о передержке (PetSittingAd):**

- Связь с питомцем и владельцем
- Даты начала и окончания передержки
- Описание требований (опционально)
- Статус объявления ('active' / 'closed')
- Структурированный адрес (связь с моделью Address)
- Максимальное расстояние для поиска ситтеров
- Тип компенсации ('paid' / 'unpaid')

### 3.3 Отклики ситтеров

**Алгоритм отклика:**

1. Ситтер просматривает доступные объявления
2. Откликается на подходящее объявление (PetSittingResponse)
3. Добавляет сообщение владельцу (опционально)
4. Система уведомляет владельца о новом отклике
5. Владелец принимает или отклоняет отклик

**Отклик ситтера (PetSittingResponse):**

- Связь с объявлением и ситтером
- Сообщение владельцу (опционально) - текстовое поле для краткого сообщения
- Статус отклика ('pending' / 'accepted' / 'rejected')

**Примечание:** Система диалогов между владельцами и ситтерами реализована и доступна для использования (см. раздел 3.7).

### 3.6 Управление процессом передержки

**Алгоритм управления:**

1. После принятия отклика создается запись PetSitting
2. Система отслеживает статусы подтверждения передачи и возврата
3. По завершении передержки владелец может оставить отзыв
4. Система обновляет рейтинг ситтера

**Статусы передержки:**

- 'waiting_start' - ожидание начала
- 'active' - передержка активна
- 'waiting_end' - ожидание завершения
- 'completed' - завершена
- 'cancelled' - отменена

**Подтверждения:**

- owner_confirmed_start - владелец подтвердил передачу
- sitter_confirmed_start - ситтер подтвердил получение
- owner_confirmed_end - владелец подтвердил возврат
- sitter_confirmed_end - ситтер подтвердил возврат

**Отмена передержки:**

- Передержку может отменить как владелец питомца, так и ситтер, если процесс ещё не завершён или не отменён ранее.
- После отмены обе стороны получают уведомления.

### 3.7 Система диалогов между участниками

**Назначение:** Обеспечивает полноценную коммуникацию между владельцами питомцев и ситтерами для координации процесса передержки.

**Алгоритм работы:**

1. Диалог создается автоматически при первом отклике ситтера на объявление
2. Участники могут обмениваться сообщениями через REST API
3. Система уведомляет о новых сообщениях
4. Сообщения автоматически шифруются при сохранении в базе данных
5. Пользователи могут отмечать сообщения как прочитанные

**Модели данных:**

**Диалог (Conversation):**

- Связь с объявлением о передержке (PetSittingRequest)
- Связь с активной передержкой (PetSitting, опционально)
- Автоматическое создание при первом отклике
- Методы для получения другого участника диалога

**Сообщение (Message):**

- Связь с диалогом (Conversation)
- Отправитель (sender) - пользователь, отправивший сообщение
- Получатель (recipient) - пользователь, которому адресовано сообщение
- Зашифрованный текст сообщения (text)
- Временная метка создания (created_at)
- Статус прочтения (is_read)
- Автоматическое шифрование при сохранении
- Метод для получения расшифрованного текста (decrypted_text)

**API эндпоинты:**

- `GET /api/conversations/` - список диалогов пользователя
- `GET /api/conversations/{id}/` - детали диалога с сообщениями
- `POST /api/conversations/{id}/send_message/` - отправка сообщения
- `POST /api/conversations/{id}/mark_as_read/` - отметка сообщений как прочитанные
- `POST /api/conversations/create/` - создание нового диалога

**Безопасность:**

- Шифрование сообщений с использованием AES-256-GCM
- Ключ шифрования генерируется из SECRET_KEY Django
- Автоматическое шифрование при сохранении и расшифровка при чтении
- Защита от несанкционированного доступа к содержимому сообщений

**Уведомления:**

- Автоматические уведомления о новых сообщениях
- Интеграция с существующей системой уведомлений
- Отображение количества непрочитанных сообщений в списке диалогов

### 3.5 Система отзывов

**Алгоритм отзыва:**

1. После завершения передержки система предлагает владельцу оставить отзыв (опционально)
2. **Только владелец питомца** может оставить отзыв о ситтере
3. **Ситтер НЕ может оценить себя сам** - система предотвращает самооценку
4. Владелец оценивает ситтера (1-5 звезд, опционально)
5. Добавляет текстовый отзыв (опционально)
6. Система обновляет рейтинг ситтера только при наличии оценки

**Отзыв (Review):**

- Связь с записью о передержке и автором отзыва (только владелец)
- Рейтинг (1-5 звезд, опционально)
- Текстовый отзыв (опционально)
- Защита от самооценки ситтера

## 4. ГЛОБАЛЬНАЯ НАСТРОЙКА И СПРАВОЧНИКИ

### 4.1 Глобальный справочник услуг

**Структура:**

- Единая модель Service для категорий и услуг
- Иерархия: категории → подкатегории → услуги (через parent)
- Базовые поля: название, описание, код, иконка, периодичность, обязательность, напоминания
- **Связь с типами животных**: поле `allowed_pet_types` для ограничения доступности услуг

**Связь услуг с типами животных:**

- **Назначение**: Ограничение доступности услуг для определенных типов животных
- **Примеры использования**:
  - Прививка от бешенства: доступна только для собак и кошек, недоступна для змей
  - Стрижка когтей: доступна для собак, кошек, птиц, но не для рыб
  - Ультразвуковая чистка зубов: только для собак и кошек
- **Логика работы**:
  - Если `allowed_pet_types` пустое → услуга доступна для всех типов животных
  - Если указаны типы → услуга доступна только для указанных типов
  - При создании записи о процедуре система проверяет совместимость

**Валидация совместимости:**

- При создании записи о процедуре (PetRecord) система автоматически проверяет совместимость услуги с типом животного
- Если услуга не совместима с типом животного → ошибка валидации
- API предоставляет методы для проверки совместимости перед созданием записи

**Использование:**

- Категории и услуги создаются и редактируются администратором
- Учреждения используют эти категории для создания собственных услуг
- Система автоматически фильтрует доступные услуги по типу животного

**API для работы с совместимостью:**

- `GET /api/services/for_pet_type/?pet_type_id=<id>` - получить услуги для конкретного типа животного
- `POST /api/services/check_compatibility/` - проверить совместимость услуги с типом животного
- `GET /api/services/<id>/` - получить услугу с информацией о доступных типах животных

**Примечание:**

- Связь с типами животных обеспечивает корректность медицинских процедур и ухода
- Периодичность услуг применяется одинаково для всех совместимых типов животных
- Система предотвращает назначение неподходящих процедур (например, прививки змеям)

### 4.2 Справочник пород животных

**Структура справочника:**

- **Типы животных** - собаки, кошки, птицы, грызуны и т.д.
- **Породы** - конкретные породы в рамках типа
- **Базовые поля** - название, код, описание породы

**Алгоритм управления:**

1. Системный администратор добавляет новый тип животного
2. Создает породы для данного типа
3. Указывает название, код и описание породы
4. Справочник используется при создании профилей питомцев

### 4.3 Справочник документов

**Структура справочника:**

- **Типы документов** - фиксированный каталог из 5 согласованных типов:
  - `Паспорт / идентификация`
  - `Ветеринарная справка / сертификат`
  - `Анализы`
  - `Диагностика`
  - `Выписка / назначения врача`
- **Настройка обязательных полей** - обязательные поля задаются логикой типа документа, а не произвольной настройкой администратора
- **Контекст загрузки** - один и тот же тип документа может загружаться либо в карточку питомца, либо в визит / дополнение к визиту, в зависимости от сценария

**Алгоритм управления:**

1. Система поставляет фиксированный каталог типов документов и поддерживает его миграциями данных
2. Пользователь при загрузке документа выбирает тип из согласованного списка
3. Backend валидирует обязательные поля по выбранному типу документа
4. Права на создание и удаление документа определяются контекстом загрузки: карточка питомца или визит / дополнение к визиту

### 4.4 Глобальные настройки системы

**Настройки геолокации:**

- Длительность кэша геокодирования (30 дней)
- Таймаут API запросов (10 секунд)
- Максимальное количество попыток API (3)
- Задержка между попытками (1 секунда)
- Минимальный уровень уверенности геокодирования (0.8)
- Язык по умолчанию для API (en)
- Включение/выключение кэширования
- Период очистки истекшего кэша (7 дней)

**Настройки уведомлений:**

- **Шаблоны email-сообщений** - создание и редактирование текста уведомлений (HTML и текстовые версии)
- **Настройки push-уведомлений** - типы уведомлений, обязательные/необязательные, включение/выключение
- **Время отправки напоминаний** - мгновенно, за 30 мин, за 1 час, за 2 часа, за 6 часов, за 12 часов, за 24 часа
- **Частота уведомлений** - ежедневно, еженедельно, ежемесячно, ежегодно, произвольный интервал
- **Каналы доставки** - email, push, in-app уведомления
- **Настройки пользователей** - предпочтения по типам уведомлений и каналам

**Примечание:** Связь конкретных событий (создание бронирования, отмена, изменение цены и т.д.) с уведомлениями жестко прописана в коде через Django сигналы. Администратор настраивает только содержание, время и каналы доставки уведомлений.

**Примечание 2:** Планируется реализация гибкой системы связей событий с шаблонами уведомлений через админский интерфейс, что позволит администраторам создавать новые правила уведомлений без изменения кода.

**Гибкая система правил уведомлений:**

- **Модель правил** - создание и управление правилами связей событий с уведомлениями
- **Условия срабатывания** - настройка условий для отправки уведомлений (VIP-клиенты, суммы, время)
- **Приоритеты правил** - система приоритизации (critical, high, medium, low)
- **Наследование правил** - глобальные → пользовательские настройки
- **Валидация условий** - проверка корректности правил при создании
- **Тестирование правил** - возможность протестировать правило на тестовых данных
- **Статистика срабатываний** - мониторинг использования правил
- **API управления** - программное управление правилами через API

**Настройки безопасности:**

- Политика паролей
- Настройки сессий
- Ограничения на вход
- Настройки аудита
  **ГЭП**

**Настройки бизнес-логики:**

**Время отмены бронирования**: Настраиваемые параметры для определения допустимого времени отмены бронирования без штрафных санкций. Система учитывает различные типы услуг и учреждений, позволяя администратору устанавливать гибкие правила отмены в зависимости от специфики услуг.

**Автоматическое завершение:**

1. Система раз в час (или с другой заданной периодичностью) проверяет все бронирования, которые начались более N дней назад (обычно 1 день = 24 часа), но не были завершены вручную.
2. Такие бронирования автоматически переводятся в статус "завершено".
3. Клиенту отправляется уведомление о завершении.
4. Все отчёты, аналитика, рейтинги и дашборды автоматически учитывают автозавершённые записи (например, количество завершённых бронирований, процент успешных записей, выручка, активность пользователей и т.д.).

**Настройки рейтингов**: Параметры для расчета рейтингов учреждений и сотрудников, включая весовые коэффициенты для отзывов, жалоб и отмен. Система поддерживает модерацию отзывов, обнаружение подозрительной активности и автоматическую фильтрацию недобросовестных оценок.

**Параметры блокировки учреждений**: Шаблоны и настройки для автоматической блокировки учреждений при нарушении правил. Включает различные типы нарушений, сроки блокировки, условия восстановления и систему уведомлений о нарушениях с возможностью обжалования.

## 5. УПРАВЛЕНИЕ УЧРЕЖДЕНИЯМИ

### 5.1 Регистрация учреждения (Мастер регистрации)

**Общее описание:**
Регистрация провайдера осуществляется через пошаговый мастер (wizard), который объединяет все необходимые этапы в единый процесс. Мастер обеспечивает валидацию данных, получение юридических документов с учетом страны регистрации, и автоматическую активацию провайдера после завершения всех шагов.

**Важно:** Мастер регистрации доступен только для авторизованных пользователей. Владелец бизнеса должен явно указать email зарегистрированного пользователя, который будет назначен администратором провайдера. Email администратора может отличаться от email пользователя, который заполняет форму, и от email организации.

**Примечание:** Privacy Policy и Terms of Service читаются всеми пользователями при регистрации в системе (раздел 1.1) и не требуются в мастере регистрации провайдера.

**Алгоритм регистрации через мастер:**

#### Шаг 1: Выбор страны регистрации

1. Провайдер (Owner) выбирает страну регистрации организации (ISO 3166-1 alpha-2 код)
2. Система проверяет наличие `VATRate` для выбранной страны:
   - **Если VATRate отсутствует:**
     - Отображается сообщение: "Мы пока не работаем в этой стране. Пожалуйста, обратитесь в поддержку для запроса добавления вашей страны."
     - Регистрация блокируется до добавления VATRate администратором
   - **Если VATRate существует:**
     - Система автоматически определяет регион через `get_region_code(country_code)`:
       - EU страны (DE, FR, IT и т.д.) → регион 'EU'
       - RU → регион 'RU'
       - UA → регион 'UA'
       - EAEU страны → регион 'EAEU'
     - Переход к следующему шагу

#### Шаг 2: Заполнение базовых данных организации

1. **Название организации** (обязательно, уникальное)
2. **Юридический адрес организации** (обязательно)
   - Автоматическое геокодирование через Google Maps API
   - Сохранение нормализованного адреса
3. **Email организации** (обязательно, уникальный на уровне организации)
   - **Валидация формата на фронтенде:**
     - Проверка формата email (regex валидация) с задержкой (debounce) 500-800ms после остановки ввода
     - Валидация выполняется только после того, как пользователь перестал вводить данные
     - Отображение ошибки формата в реальном времени (если формат неверный)
     - Проверка уникальности на бэкенде после валидации формата
4. **Телефон организации** (обязательно, уникальный на уровне организации)
   - **Валидация формата на фронтенде:**
     - Проверка формата телефона (международный формат E.164 или национальный формат) с задержкой (debounce) 500-800ms после остановки ввода
     - Валидация выполняется только после того, как пользователь перестал вводить данные
     - Отображение ошибки формата в реальном времени (если формат неверный)
     - Проверка уникальности на бэкенде после валидации формата
     - Автоматическое форматирование телефона при вводе (маска ввода)
5. **Email администратора провайдера** (обязательно)
   - Email зарегистрированного пользователя, который будет назначен администратором провайдера
   - **Валидация:**
     - Проверка формата email на фронтенде (regex валидация) с задержкой (debounce) 500-800ms
     - **Проверка существования пользователя на бэкенде:**
       - После валидации формата отправляется запрос на бэкенд для проверки существования пользователя
       - Пользователь с указанным email должен существовать в системе и быть активным (`is_active=True`)
       - Если пользователь не найден, отображается ошибка: "Пользователь с таким email не зарегистрирован в системе. Пожалуйста, укажите email существующего пользователя или попросите пользователя зарегистрироваться."
     - Email администратора может отличаться от email пользователя, который заполняет форму
     - Email администратора может отличаться от email организации
6. **Категории услуг уровня 0** (обязательно, минимум 1 категория)
   - Выбор из каталога услуг
   - Только категории уровня 0 (корневые категории)
7. **Документы** (опционально)
   - Загрузка лицензий или сертификатов по желанию. Загрузка **не обязательна** и **не блокирует** регистрацию.
   - Платформа как агрегатор не требует проверки лицензий; ответственность за наличие лицензий лежит на провайдере.
   - Если провайдер загрузил документы: в карточке организации для потребителей показывается отметка «Есть документы», по запросу отображаются сами документы.
   - В блоке «Документы» на шаге 2 отображается поясняющий текст (в т.ч. DE, EN, RU, ME) о том, что загрузка не обязательна и для чего она нужна.

**Валидация на этом шаге:**

- **Фронтенд валидация (в реальном времени с debounce):**
  - Email организации: проверка формата через regex (например, стандартный паттерн для email)
  - Телефон: проверка формата (E.164 или национальный формат в зависимости от страны)
  - Email администратора: проверка формата через regex
  - Задержка валидации: 500-800ms после остановки ввода (debounce)
  - Отображение ошибок валидации под полями ввода
- **Бэкенд валидация (при отправке формы):**
  - Проверка уникальности названия организации
  - Проверка уникальности email и телефона организации
  - **Проверка существования пользователя-администратора:**
    - Пользователь с указанным email администратора должен существовать в системе
    - Пользователь должен быть активным (`is_active=True`)
    - Если пользователь не найден, регистрация блокируется с сообщением об ошибке
  - Повторная валидация формата контактных данных (дополнительная проверка безопасности)
  - Проверка выбора минимум одной категории услуг
  - Геокодирование адреса организации

#### Шаг 3: Заполнение юридических реквизитов

1. **Tax ID / ИНН** (обязательно, уникален по составному ключу «страна + Tax ID»)
   - Формат: буквы, цифры, пробелы, дефисы
   - Минимум 3 символа
   - **Валидация через API** (в зависимости от региона):

       - Для юрлиц/ИП: ИНН валидируется через ЕГРЮЛ API или DaData API

   - Валидация формата: буквы, цифры, пробелы, дефисы, точки (минимум 2 символа)

   - KPP (для РФ)

**Бизнес-процесс валидации реквизитов провайдера:**

**Общее описание:**
Система проверяет валидность всех реквизитов провайдера на двух уровнях: проверка формата (для всех стран и всех реквизитов) и проверка через внешние API (для стран, где доступны соответствующие сервисы). Процесс включает автоматическую проверку при регистрации, механизм fallback при недоступности API, возможность ручной проверки в админке и ручное подтверждение администраторами.

**Этап 1: Проверка формата всех реквизитов (обязательна, всегда выполняется)**

**Общие принципы валидации формата:**

- Все реквизиты проверяются на соответствие формату в реальном времени на фронтенде (с задержкой 500-800ms после остановки ввода)
- При отправке формы на сервер выполняется повторная проверка формата на бэкенде
- Если формат неверный → регистрация блокируется с указанием конкретной ошибки
- Если формат валидный → переход к следующему этапу (проверка через API, если доступна)

**Валидация формата по типам реквизитов:**

**Когда проверяется уникальность:** проверка уникальности (Tax ID, Registration Number, VAT Number — по связке со страной; IBAN — глобально) выполняется в трёх точках: **при уходе с поля** (on blur, через API check-iban, check-tax-id, check-registration-number, check-vat-number), **при переходе к следующему шагу** (API step3-validate) и **при окончании работы мастера** (валидация в сериализаторе при финальной отправке). Так реализовано на бэкенде; на фронте при уходе с поля вызываются соответствующие check-* эндпоинты, при нажатии «Далее» — step3-validate, при отправке формы — полная валидация в мастере.

**1.1. Tax ID / ИНН (обязательно для всех стран):**

- **Германия и другие страны ЕС:**
  - Если провайдер - плательщик НДС: формат VAT ID = код страны (2 буквы) + 9 цифр (например, `DE123456789`)
  - Если провайдер - неплательщик НДС: Tax ID не требуется
- **Россия (RU):**
  - Для юрлиц: 10 цифр (например, `7707083893`)
  - Для ИП: 12 цифр (например, `500100732259`)
  - Валидация контрольной суммы (алгоритм проверки ИНН)
- **Украина (UA):**
  - Для юрлиц (ЄДРПОУ): 8 цифр (например, `12345678`)
  - Для ФОП (ИНН): 10 цифр (например, `1234567890`)
- **Черногория (ME):**
  - PIB (Tax ID): 8 цифр (например, `12345678`)
- **Другие страны:**
  - Формат: буквы, цифры, пробелы, дефисы
  - Минимум 3 символа
  - Максимум 20 символов
- **Уникальность:** проверка в БД по составному ключу (страна + Tax ID); один и тот же ИНН допустим в разных странах, в одной стране — единственный

**1.2. Registration Number (обязательно для всех стран):**

- Формат: буквы, цифры, пробелы, дефисы, точки
- Минимум 3 символа
- Максимум 50 символов
- **Уникальность:** проверка в БД по составному ключу (страна + Registration Number)

**1.3. VAT Number (для стран ЕС, если провайдер - плательщик НДС):**

- Формат: код страны (2 буквы, ISO 3166-1 alpha-2) + 9 цифр
- Примеры:
  - Германия: `DE123456789`
  - Франция: `FR12345678901` (11 цифр)
  - Италия: `IT12345678901` (11 цифр)
  - Испания: `ES12345678901` (11 цифр)
  - Нидерланды: `NL123456789B01` (12 символов, может содержать буквы)
- Нормализация: автоматическое удаление пробелов, приведение к верхнему регистру
- Проверка формата выполняется с учетом специфики каждой страны ЕС
- **Уникальность:** проверка в БД по составному ключу (страна + VAT Number)

**1.4. IBAN (для стран ЕС, если провайдер - плательщик НДС):**

- Формат: 2 буквы (код страны) + 2 цифры (контрольная сумма) + до 30 символов (буквы и цифры)
- Примеры:
  - Германия: `DE89370400440532013000` (22 символа)
  - Франция: `FR1420041010050500013M02606` (27 символов)
- Валидация контрольной суммы IBAN (алгоритм MOD-97-10)
- Проверка формата выполняется с учетом специфики каждой страны
- **Уникальность:** проверка в БД глобально (один IBAN = один счёт = одно юрлицо); дубли IBAN в системе не допускаются

**1.5. KPP (для России, если организация - ООО):**

- Формат: 9 цифр (например, `770701001`)
- Обязателен только для организаций типа "ООО" в России
- Проверка выполняется только если `country = 'RU'` и `organization_type` содержит "ООО" (case-insensitive)

**1.6. SWIFT/BIC:**

- Формат: 8 или 11 символов (4 буквы + 2 буквы + 2 символа + опционально 3 символа)
- Пример: `DEUTDEFF` или `DEUTDEFF500`
- Проверка формата по стандарту ISO 9362

**Этап 2: Проверка через внешние API (для стран с доступными сервисами)**

**2.1. Проверка VAT ID через VIES API для стран ЕС (детально):**

**Процесс проверки:**

1. После успешной проверки формата система отправляет запрос к VIES API для проверки валидности VAT ID
2. VIES API возвращает результат:
   - **Валидный VAT ID:** возвращает данные компании (название, адрес)
   - **Невалидный VAT ID:** возвращает ошибку "INVALID_INPUT"
   - **Ошибка API:** таймаут, недоступность сервиса, сетевые ошибки

**Обработка результатов:**

**Сценарий 2.1: VAT ID валидный (успешная проверка)**

- Система сохраняет результат проверки:
  - Статус проверки: `valid`
  - Дата проверки: текущая дата и время
  - Данные компании: название и адрес из ответа VIES API
  - Результат сохраняется в JSON-поле для аудита
- Провайдер может продолжить регистрацию
- После завершения всех шагов провайдер активируется автоматически (если выполнены все остальные условия)

**Сценарий 2.2: VAT ID невалидный (не найден в реестре)**

- Система сохраняет результат проверки:
  - Статус проверки: `invalid`
  - Дата проверки: текущая дата и время
  - Причина: "VAT ID не найден в реестре ЕС"
- Регистрация блокируется
- Пользователю показывается ошибка: "VAT ID не найден в реестре ЕС. Пожалуйста, проверьте правильность введенного номера."
- Пользователь может исправить VAT ID и повторить попытку

**Сценарий 2.3: VIES API недоступен (fallback)**

- Система сохраняет результат проверки:
  - Статус проверки: `failed`
  - Дата проверки: текущая дата и время
  - Причина ошибки: "VIES API недоступен" (таймаут, 503, сеть и т.д.)
- **Fallback активируется:** регистрация не блокируется, пользователь может продолжить
- Провайдер создается со статусом `pending_verification` (требует проверки)
- Провайдер **не активируется** до подтверждения валидности VAT ID
- Автоматически отправляется уведомление назначенному биллинг-менеджеру:
  - Тема: "Требуется проверка VAT ID для нового провайдера"
  - Содержание: информация о провайдере, VAT ID, причина (API недоступен)
  - Ссылка на страницу провайдера в админке

**Кэширование результатов:**

- Валидные VAT ID кэшируются на 24 часа (для ускорения повторных проверок)
- Невалидные VAT ID кэшируются на 1 час (на случай исправления ошибки пользователем)
- При проверке сначала проверяется кэш, затем VIES API (если кэш пуст или истек)

**Этап 3: Ручная проверка VAT ID в админке (для биллинг-менеджеров и системных администраторов)**

**Доступ к функции:**

- Системный администратор: доступ ко всем провайдерам
- Биллинг-менеджер: доступ только к назначенным провайдерам (через `BillingManagerProvider`)

**Кнопка "Проверить VAT ID сейчас":**

- На странице редактирования провайдера в админке Django отображается кнопка "Проверить VAT ID сейчас"
- При нажатии кнопки:
  1. Система отправляет запрос к VIES API для проверки текущего VAT ID провайдера
  2. Результат проверки отображается на странице:
     - ✅ "VAT ID валидный. Компания: [название], Адрес: [адрес]"
     - ❌ "VAT ID не найден в реестре ЕС"
     - ⚠️ "Сервис проверки недоступен. Попробуйте позже"
  3. Статус проверки обновляется автоматически:
     - Если валидный → статус меняется на `valid`, провайдер может быть активирован
     - Если невалидный → статус меняется на `invalid`, провайдер блокируется
     - Если API недоступен → статус остается `failed`
  4. Результат проверки сохраняется в базе данных с указанием пользователя, который выполнил проверку, и времени проверки

**Массовая проверка:**

- В списке провайдеров доступно действие "Проверить VAT ID" для выбранных провайдеров
- Система проверяет все выбранные провайдеры последовательно (с учетом rate limiting VIES API)
- Результаты отображаются в виде отчета после завершения проверки

  - URL: `https://ec.europa.eu/taxation_customs/vies/?locale=en#/vat-validation`
- Биллинг-менеджер или администратор может:
  1. Открыть ссылку на официальный сайт VIES
  2. Ввести VAT ID вручную на сайте VIES
  3. Получить результат проверки на официальном сайте
  4. Вернуться в админку и подтвердить результат

**Чекбокс "VAT ID подтвержден вручную":**

- На странице провайдера в админке доступен чекбокс "VAT ID подтвержден вручную" (`vat_verification_manual_override`)
- Доступ к редактированию:
  - **Системный администратор:** полный доступ ко всем провайдерам
  - **Биллинг-менеджер:** доступ только к назначенным провайдерам
- При установке чекбокса:
  1. Система требует обязательного заполнения поля "Комментарий" (минимум 10 символов)
  2. Комментарий должен содержать обоснование (например: "Проверено на сайте VIES, компания: Test Company GmbH, адрес: Test Street 1, Berlin")
  3. Система сохраняет:
     - Кто установил чекбокс (пользователь)
     - Когда установлен (дата и время)
     - Комментарий
     - Статус проверки меняется на `valid` (если был `failed`)
  4. Создается запись в системе аудита для отслеживания ручных подтверждений
  5. Отправляется уведомление системному администратору о ручном подтверждении (для контроля)
- При снятии чекбокса:
  - Статус проверки возвращается к предыдущему значению (`failed` или `invalid`)
  - Комментарий сохраняется в истории для аудита

**Этап 5: Логика активации провайдера на основе проверки VAT ID**

**Условия активации провайдера:**
Провайдер может быть активирован (статус `active`), если выполнены **все** условия:

1. Все обязательные поля заполнены (название, адрес, контакты, реквизиты)
2. Оферта принята (`offer_accepted = True`)
3. **VAT ID подтвержден** (одно из условий):
   - Статус проверки: `valid` (проверено через VIES API успешно)
   - **ИЛИ** чекбокс "VAT ID подтвержден вручную" установлен (`vat_verification_manual_override = True`)

**Блокировка активации:**
Провайдер **не может быть активирован**, если:

- Статус проверки: `invalid` (VAT ID не найден в реестре) → **блокировка до исправления**
- Статус проверки: `failed` (API недоступен) **И** чекбокс не установлен → **блокировка до проверки**

**Автоматическая активация:**

- Если при регистрации VAT ID проверен успешно (`valid`) → провайдер активируется автоматически после завершения всех шагов
- Если при регистрации API недоступен (`failed`) → провайдер создается со статусом `pending_verification`, активация блокируется до проверки

**Ручная активация:**

- Системный администратор или биллинг-менеджер может вручную изменить статус провайдера на `active` в админке
- Система проверяет условия активации перед сохранением
- Если условия не выполнены → показывается предупреждение с указанием причин блокировки

**Уведомления:**

- При успешной проверке VAT ID → уведомление биллинг-менеджеру: "VAT ID провайдера [название] успешно проверен"
- При невалидном VAT ID → срочное уведомление биллинг-менеджеру: "VAT ID провайдера [название] не найден в реестре"
- При ручном подтверждении → уведомление системному администратору: "Биллинг-менеджер [имя] подтвердил VAT ID провайдера [название] вручную"

**История проверок:**

- Система сохраняет полную историю всех проверок VAT ID для каждого провайдера:
  - Дата и время проверки
  - Результат проверки (valid/invalid/failed)
  - Источник проверки (автоматическая при регистрации / ручная в админке)
  - Пользователь, выполнивший проверку (для ручных проверок)
  - Данные из VIES API (название компании, адрес)
  - Комментарии при ручном подтверждении
- История доступна для просмотра в админке на странице провайдера

**Особенности для неплательщиков НДС:**

- Если провайдер указал статус "Неплательщик НДС" на Шаге 3:
  - Поле VAT ID становится необязательным
  - Проверка VAT ID через VIES API не выполняется
  - Провайдер может быть активирован без проверки VAT ID
  - В счетах НДС не применяется (ставка 0%)

#### Шаг 4: Получение и принятие юридических документов

1. **Автоматическое получение документов:**

   - Система получает `CountryLegalConfig` для выбранной страны
   - Извлекает `global_offer` (обязательна для всех стран)
   - Извлекает `regional_addendums` для определенного региона:
     - Фильтрация по `region_code` (например, 'EU' для DE)
     - Может быть несколько региональных дополнений
   - **Примечание:** Privacy Policy и Terms of Service не показываются в мастере (читаются при регистрации пользователя)
2. **Подстановка переменных в документы:**

   - Из `BillingConfig` глобальной оферты:
     - `{{commission_percent}}` - процент комиссии
     - `{{payment_deferral_days}}` - дни отсрочки платежа
     - `{{invoice_period_days}}` - период выставления счетов
   - Из `VATRate` для страны провайдера (только если провайдер является плательщиком НДС):
     - `{{vat_rate}}` - ставка НДС (автоматически подставляется, если провайдер - плательщик НДС)
     - Если провайдер - неплательщик НДС, переменная `{{vat_rate}}` заменяется на "0%" или "не применяется"
   - Из `LegalDocument.variables` (если есть):
     - Дополнительные переменные для региональных дополнений
   - Из `LegalDocument.change_notification_days`:
     - `{{change_notification_days}}` - дни уведомления об изменениях
3. **Отображение документов:**

   - Показывается `global_offer` с подставленными переменными
   - Показываются все `regional_addendums` для региона провайдера
   - Документы отображаются на выбранном языке (из настроек пользователя или по умолчанию 'en')
   - **Требование к прочтению:**
     - Чекбокс "Я принимаю условия оферты и региональные дополнения" **недоступен** (disabled) до тех пор, пока:
       - Провайдер не прокрутил документы до конца (scroll position >= document height)
       - Или не прошло минимальное время чтения (например, 30 секунд на 1000 слов)
     - Система отслеживает:
       - Позицию прокрутки каждого документа
       - Время, проведенное на странице с документами
       - Взаимодействие с документом (скроллинг, клики)
     - Только после полного прочтения всех документов чекбокс становится активным
   - Провайдер должен прочитать документы и осознанно принять их
4. **Принятие документов:**

   - После того, как чекбокс стал доступным, провайдер ставит галочку "Я принимаю условия оферты и региональные дополнения"
   - При принятии автоматически создается `DocumentAcceptance`:
     - `document` - ссылка на `global_offer`
     - `accepted_regional_addendums` - M2M связь с принятыми региональными дополнениями
     - `accepted_at` - время принятия
     - `ip_address` - IP адрес провайдера
     - `user_agent` - User Agent браузера
     - `is_active` - True

#### Шаг 5: Автоматическое создание заявки и назначение биллинг-менеджера

1. **Автоматическое создание заявки:**

   - После завершения всех шагов мастером (Шаги 1-4), система **автоматически** создает заявку `ProviderForm` со статусом `pending`
   - Заявка содержит все данные, заполненные в мастере:
     - Базовые данные организации (название, адрес, контакты, категории)
     - **Три обязательных email:** владельца (owner_email), менеджера провайдера (provider_manager_email), администратора с доступом в админку провайдера (admin_email). Одно лицо может совмещать роли — один email в нескольких полях; при рассылке отправляется ровно одно письмо на каждый уникальный email.
     - Юридические реквизиты (Tax ID, Registration Number, валюта, статус НДС)
     - Страна регистрации
     - Принятые документы (DocumentAcceptance)
   - Система автоматически определяет биллинг-менеджера:
     - Если есть доступные биллинг-менеджеры, назначается первый доступный (по принципу round-robin или наименьшей загрузке)
     - Если биллинг-менеджеров нет, заявка остается со статусом `pending` и ожидает ручного назначения системным администратором
2. **Автоматическое одобрение заявки:**

   - После назначения биллинг-менеджера (автоматически или вручную), система **автоматически** одобряет заявку:
     - Статус заявки меняется на 'approved'
     - Создается организация (Provider) с данными из заявки:
       - Название, юридический адрес, контакты
       - Реквизиты (Tax ID, Registration Number, валюта, статус НДС)
       - Страна регистрации
       - Категории услуг
     - Создаются связи **EmployeeProvider** и назначаются **user_types** для лиц, указанных в заявке:
       - По полям `ProviderForm.owner_email`, `ProviderForm.provider_manager_email`, `ProviderForm.admin_email` (все три обязательны) определяются пользователи (должны быть зарегистрированы в системе; валидация на Шаге 2).
       - Для каждого уникального email: получить User, при необходимости добавить user_types (owner, provider_manager, provider_admin), получить или создать Employee, создать записи EmployeeProvider с ролями owner / provider_manager / provider_admin в соответствии с тем, в каких полях указан этот email. Один пользователь может получить несколько ролей (несколько записей EmployeeProvider).
       - Уведомления при активации отправляются **по одному письму на каждый уникальный email** (не дублировать одному адресату).
     - Создается связь `BillingManagerProvider` (связь назначенного биллинг-менеджера с организацией)
     - Создается событие `BillingManagerEvent` типа 'assigned' для истории назначений
     - Статус провайдера устанавливается в `activation_required` (готов к активации)
3. **Ручное назначение биллинг-менеджера (если требуется):**

   - Если автоматическое назначение не удалось (нет доступных биллинг-менеджеров), системный администратор может:
     - Просмотреть заявку в админке Django
     - Использовать действие "Одобрить и назначить биллинг-менеджера"
     - Выбрать биллинг-менеджера из списка доступных менеджеров по биллингу
     - После выбора система автоматически выполняет шаг 2 (одобрение заявки)

#### Шаг 6: Автоматическая активация провайдера

1. **Автоматическая активация:**

   - После одобрения заявки (Шаг 5), система **автоматически** активирует провайдера:
     - Статус провайдера переводится из `activation_required` в `active`
     - Поле `is_active` устанавливается в `True`
     - Провайдер получает полный доступ к системе
2. **Автоматическая отправка уведомлений:**

   - При активации автоматически отправляются письма: **по одному на каждый уникальный email** из owner_email, provider_manager_email, admin_email (если один и тот же email указан в нескольких полях — одно письмо). В каждом письме: информация об активации провайдера, ссылка на приложение «Админка провайдеров», инструкция по входу (email/пароль или Google), контакты назначенного биллинг-менеджера, ссылка на инструкцию (`/provider-setup-guide`).
3. **Ручная деактивация (если требуется):**

   - Системный администратор может вручную деактивировать провайдера, если обнаружены проблемы:
     - Изменение статуса с `active` на `inactive` или `rejected`
     - Провайдер теряет доступ к системе до повторной активации

**Дополнительные возможности:**

- Системный администратор может добавлять/удалять категории для провайдера
- Провайдер может запросить добавление новой категории через поддержку
- Логирование всех изменений категорий
- Уведомления провайдеру об изменениях в доступных категориях

**Управление статусами провайдера:**

- Системный администратор может изменить статус провайдера:
  - `activation_required` → `active` (активировать провайдера)
  - `activation_required` → `rejected` (отклонить)
  - `active` → `inactive` (деактивировать активного провайдера)
  - `active` → `rejected` (отклонить активного провайдера)
  - `rejected` → `activation_required` (вернуть на активацию)
  - `inactive` → `active` (реактивировать провайдера)
- Поле `is_active` автоматически устанавливается в `True` при статусе `active` и в `False` для всех остальных статусов
- Биллинг-менеджер видит только свои назначенные организации (через BillingManagerProvider)
- Биллинг-менеджер НЕ видит заявки (ProviderForm) в админке - только одобренные организации
- Биллинг-менеджер НЕ может изменять поле `is_active` напрямую - оно управляется через `activation_status`
- Биллинг-менеджер НЕ может изменять настройки блокировки (`exclude_from_blocking_checks`, `blocking_exclusion_reason`) - это прерогатива только системного администратора

**Права доступа после активации:**

- **Определение владельца, менеджера и администратора провайдера:**
  - В заявке (ProviderForm) обязательны три поля: **owner_email** (владелец), **provider_manager_email** (менеджер провайдера), **admin_email** (администратор — лицо с доступом в приложение «Админка провайдеров»). Все три пользователя должны быть зарегистрированы в системе (валидация на Шаге 2). Одно лицо может совмещать роли (один email в нескольких полях).
  - При одобрении создаются записи EmployeeProvider (owner, provider_manager, provider_admin) и при необходимости назначаются user_types; уведомления — по одному на каждый уникальный email.
- Администратор провайдера и персонал провайдера работают в **приложении «Админка провайдеров»** (Provider Admin App), а не в админке Django. Вход — по email/пароль или через Google OAuth (тот же бэкенд и те же JWT, что и в основном приложении). После входа доступ только к своей организации: локации, услуги, сотрудники, расписания.
- Администратор провайдера создаёт и редактирует локации в приложении «Админка провайдеров» с **немедленным вводом адреса** (как в мастере добавления провайдеров в основном фронте): поле адреса с автодополнением, геокодирование, привязка координат.
- Администратор провайдера имеет доступ только к своей организации и не может создавать локации для других организаций.

### 5.2 Управление локациями учреждения

**Общее описание:**
Организация (Provider) может иметь множество локаций (ProviderLocation) - физических мест предоставления услуг. Каждая локация имеет свой адрес, контактные данные, набор услуг и расписание работы. Это позволяет сетям учреждений управлять несколькими филиалами в рамках одной организации.

**Структура локации:**

- **Связь с организацией** - каждая локация принадлежит одной организации
- **Название локации** - например, "Филиал на Тимирязевской" или адрес локации
- **Адрес и координаты** - структурированный адрес локации (может совпадать с юридическим адресом организации)
- **Контактная информация** - email и телефон локации (НЕ уникальны на уровне локаций, разные локации могут иметь одинаковые контакты)
- **Услуги** - набор услуг, доступных в этой локации (из категорий уровня 0 организации)
- **Расписание работы** - часы работы локации (LocationSchedule):
  - День недели (понедельник-воскресенье)
  - Время открытия и закрытия
  - Статус выходного дня
  - Каждая локация может иметь разное расписание для разных дней недели
  - Расписание локации используется для валидации расписания сотрудников
- **Статус активности** - активна/неактивна локация

**Алгоритм создания локации:**

1. Администратор организации входит в приложение «Админка провайдеров» после получения учетных данных (по ссылке из письма об активации).
2. Переходит в раздел управления локациями (точками).
3. Нажимает «Добавить локацию».
4. Заполняет обязательные поля с **немедленным вводом адреса** (как в мастере регистрации провайдера в основном приложении):
   - Название локации
   - Адрес локации — ввод через поле с автодополнением, геокодирование и привязка координат выполняются сразу при вводе
   - Email локации
   - Телефон локации
5. Выбирает услуги из доступных категорий уровня 0 организации
6. Настраивает расписание работы локации
7. Система создает локацию и делает её доступной для бронирования

**Бизнес-правила:**

- Услуги локации должны быть из категорий уровня 0, утвержденных для организации
- Email и телефон локации НЕ уникальны (разные локации могут иметь одинаковые контакты)
- Email и телефон организации уникальны на уровне организации
- Адрес локации может совпадать с юридическим адресом организации
- Все локации организации работают по одному договору (биллинг на уровне организации)
- Блокировка по задолженности применяется ко всей организации (всем локациям)

**Управление локациями:**

- Администратор организации может добавлять, редактировать и деактивировать локации своей организации
- Администратор организации имеет доступ только к своей организации и не может управлять локациями других организаций
- Системный администратор может управлять локациями всех организаций
- При деактивации локации:
  - Все активные и будущие бронирования в этой локации автоматически отменяются
  - Пользователям отправляются уведомления об отмене бронирований
  - История бронирований сохраняется для отчетности
- При блокировке организации все её локации автоматически деактивируются системой
- История изменений локаций логируется для аудита

**Интеграция с другими системами:**

- **Бронирования** - работают на уровне локации, пользователь бронирует услугу в конкретной локации
- **Поиск услуг** - поиск работает на уровне локаций, пользователи видят конкретные локации с адресами
- **Биллинг** - остается на уровне организации, договоры заключаются с организацией
- **Расписания** - каждое расписание привязано к конкретной локации
- **Сотрудники** - работают в организации, могут быть привязаны к конкретным локациям (опционально)

**API эндпоинты:**

- `GET /api/providers/{id}/locations/` - список локаций организации
- `POST /api/providers/{id}/locations/` - создание новой локации
- `GET /api/providers/{id}/locations/{location_id}/` - детали локации
- `PUT /api/providers/{id}/locations/{location_id}/` - редактирование локации
- `DELETE /api/providers/{id}/locations/{location_id}/` - деактивация локации
- `GET /api/locations/search/` - поиск локаций (для пользователей)

**Интерфейс управления локациями:**

- Управление локациями (точками) выполняется в **приложении «Админка провайдеров»** (Provider Admin App), а не в Django admin. Добавление локации — с немедленным вводом адреса (автодополнение и геокодирование, как в мастере регистрации провайдера).
- Администратор провайдера может добавлять/редактировать локации только своей организации
- Администратор провайдера не имеет доступа к другим организациям
- Все сущности (расписания, услуги, сотрудники) привязаны к локациям
- Биллинг-менеджеры работают с организациями в Django admin (договоры и биллинг)
- Отдельной роли «Админ локации» не требуется — достаточно роли администратора провайдера для управления всеми локациями своей организации

### 5.3 Управление услугами учреждения

**Алгоритм добавления услуги:**

1. Администратор выбирает услугу из глобального справочника
2. Указывает стоимость услуги для своего учреждения
3. Назначает специалистов, которые оказывают услугу
4. Устанавливает длительность процедуры
5. Система проверяет уникальность услуги для учреждения
6. Услуга становится доступной для бронирования

**Примечание:** Услуги теперь управляются на уровне локаций, а не организации. Каждая локация может иметь свой набор услуг и цен.

**Алгоритм добавления услуги в локацию:**

1. Администратор выбирает локацию
2. Выбирает услугу из доступных категорий уровня 0 организации
3. Указывает стоимость услуги для этой локации
4. Назначает специалистов, которые оказывают услугу в этой локации
5. Устанавливает длительность процедуры
6. Система проверяет уникальность услуги для локации
7. Услуга становится доступной для бронирования в этой локации

### 5.4 Управление персоналом учреждения

**Актуальный процесс добавления сотрудника в филиал (invite-only):**

1. Управляющий филиалом/организацией отправляет инвайт типа `specialist` на email через `/api/v1/invites/`.
2. Приглашаемый пользователь вводит 6-значный код (`/api/v1/invites/accept/`).
3. Система создаёт/обновляет связи `Employee`, `EmployeeProvider` и привязку к филиалу.
4. В филиале сотруднику назначаются услуги (`/api/v1/provider-locations/<location_pk>/staff/<employee_id>/services/`) и график (`/api/v1/provider-locations/<location_pk>/staff/<employee_id>/schedules/`).
5. Один сотрудник может работать в нескольких филиалах одного провайдера, при этом локальная активность управляется по каждому филиалу отдельно (`EmployeeLocationRole.is_active`).

**Расписание и отсутствия персонала:**

- Рабочий график хранится как недельный паттерн по филиалу (дни недели, рабочий/выходной, start/end, break).
- Валидация расписания сотрудника выполняется относительно расписания филиала:
  - начало смены не раньше открытия филиала;
  - окончание смены не позже закрытия филиала;
  - в закрытый день филиала сотрудник не может быть отмечен как рабочий.
- Отсутствия оформляются отдельными сущностями (отпуск/больничный) с поддержкой:
  - разрешения конфликтов с будущими бронированиями (cancel/reassign/granular);
  - удаления записи отсутствия;
  - досрочного выхода (изменение `end_date` в пределах периода отсутствия).

**Шахматка (график работы персонала):**

- Источник данных: `/api/v1/provider-locations/<location_id>/aggregated-schedule/` с выбранным диапазоном дат.
- По каждому сотруднику в выдаче объединяются регулярные смены и периоды отсутствий.
- Из интерфейса шахматки доступны операции: оформление отпуска/больничного, удаление записи отсутствия, досрочный выход, периодная навигация.

**Примечание:** Сотрудники привязаны и к провайдеру (уровень организации), и к филиалам. Управление доступом и активностью выполняется на двух уровнях: `EmployeeProvider` (организация) и `EmployeeLocationRole` (конкретный филиал).

#### 5.4.1 Шаблоны расписаний сотрудников

**Общее описание:**
Система шаблонов расписаний позволяет создавать типовые расписания для сотрудников локаций, которые можно быстро применять к новым или существующим сотрудникам. Это упрощает процесс настройки рабочих графиков и обеспечивает единообразие расписаний в рамках локации. Шаблоны привязаны к конкретной локации, что позволяет иметь разные шаблоны для разных филиалов одной организации.

**Функциональные требования:**

**Создание шаблонов расписаний:**

- Создание шаблона с названием и описанием для конкретной локации
- Настройка рабочих часов для каждого дня недели (понедельник-воскресенье)
- Указание времени начала и окончания работы для каждого дня
- Возможность отметить день как выходной
- Сохранение шаблона с привязкой к локации

**Структура шаблона:**

- **SchedulePattern** - основной шаблон с названием, описанием и привязкой к локации (provider_location)
- **PatternDay** - дни недели в шаблоне с рабочими часами
- Уникальная связь между шаблоном и днем недели
- Поддержка всех 7 дней недели (0=понедельник, 6=воскресенье)

**Применение шаблонов:**

- API эндпоинт для применения шаблона к сотруднику
- Автоматическое создание записей `Schedule` на основе шаблона
- Возможность применения к одному или нескольким сотрудникам
- Перезапись существующих расписаний при применении

**Управление шаблонами:**

- Просмотр списка шаблонов локации
- Редактирование существующих шаблонов
- Удаление неиспользуемых шаблонов
- Копирование шаблонов для создания новых

**Бизнес-правила:**

- Шаблоны привязаны к конкретной локации (через SchedulePattern.provider_location)
- Только администратор учреждения может создавать/редактировать шаблоны
- При применении шаблона существующие расписания сотрудника перезаписываются
- Валидация времени работы (начало < окончание)
- Поддержка выходных дней (без указания времени работы)
- При применении шаблона созданные расписания сотрудников валидируются на соответствие расписанию локации

**Интеграция:**

- Связь с моделью `Schedule` для создания индивидуальных расписаний
- API для применения шаблонов (`SchedulePatternApplyAPIView`)
- Административный интерфейс для управления шаблонами
- Автоматическое создание записей расписаний при применении

**Технические особенности:**

- Модели `SchedulePattern` и `PatternDay` в `providers/models.py`
- Уникальные ограничения для предотвращения дублирования
- Автоматическое отслеживание времени создания и обновления
- Индексы для оптимизации запросов

**API эндпоинты:**

- `GET /api/schedule-patterns/` - список шаблонов учреждения
- `POST /api/schedule-patterns/` - создание нового шаблона
- `PUT /api/schedule-patterns/{id}/` - редактирование шаблона
- `DELETE /api/schedule-patterns/{id}/` - удаление шаблона
- `POST /api/schedule-patterns/{id}/apply/` - применение шаблона к сотруднику

**Управление доступом:**

1. Назначение ролей сотрудникам
2. Ограничение доступа к функциям
3. Блокировка/разблокировка аккаунтов

## 6. ГЕОЛОКАЦИЯ И ПОИСК УСЛУГ

### 6.1 Система геолокации

#### 6.1.1 Структурированные адреса

**Общее описание:**
Система должна поддерживать структурированное хранение и валидацию адресов для всех сущностей (пользователи, учреждения, питомцы). Адреса должны быть разбиты на отдельные компоненты для обеспечения точности геолокации и упрощения поиска.

**Компоненты адреса:**

- **Страна** - обязательное поле, код страны по ISO 3166-1 alpha-2
- **Регион/Область** - административная единица первого уровня
- **Город** - обязательное поле, название населенного пункта
- **Район** - административная единица города (опционально)
- **Улица** - название улицы, проспекта, переулка
- **Номер дома** - номер здания
- **Номер квартиры/офиса** - номер помещения (опционально)
- **Почтовый индекс** - код для почтовой доставки
- **Координаты** - географические координаты (широта, долгота)

**Функциональные требования:**

- **Валидация адресов** - проверка корректности введенных данных
- **Геокодирование** - автоматическое определение координат по адресу
- **Обратное геокодирование** - получение адреса по координатам
- **Кэширование** - сохранение результатов геокодирования для оптимизации
- **История изменений** - отслеживание изменений адресов пользователей
- **Поиск по радиусу** - поиск объектов в заданном радиусе от точки

**Интеграция:**

- Интеграция с внешними сервисами геокодирования (Google Maps API, Yandex Maps API)
- Поддержка различных форматов адресов для разных стран
- Автоматическое обновление координат при изменении адреса

**Бизнес-правила:**

- **Множественные пользователи по одному адресу** - несколько владельцев питомцев могут проживать по одному адресу (с точностью до номера помещения)
- **Смешанное использование адресов** - по одному адресу могут находиться как владельцы питомцев, так и поставщики услуг
- **Уникальность на уровне помещения** - адрес с номером квартиры/офиса может быть уникальным, но без номера помещения - общим для нескольких пользователей
- При изменении адреса должны обновляться все связанные записи
- Координаты должны быть обязательными для учреждений
- Поддержка множественных адресов для одного пользователя (основной и дополнительные)

#### 6.1.2 Алгоритм определения местоположения

**Алгоритм определения местоположения:**

1. **Приоритет 1: Геолокация устройства**

   - Пользователь разрешает доступ к геолокации
   - Система получает координаты (широта, долгота) с точностью до 50 метров
   - Координаты сохраняются в профиле пользователя
   - При поиске услуг используется текущее местоположение
2. **Приоритет 2: Выбор района на карте**

   - Если геолокация недоступна или пользователь отказал в доступе
   - Пользователь может выбрать район поиска на интерактивной карте
   - Система получает координаты выбранной точки
   - Координаты сохраняются в профиле пользователя
3. **Приоритет 3: Ручной ввод адреса**

   - Если карта недоступна, пользователь может ввести адрес вручную
   - Система геокодирует адрес через Google Maps API
   - Полученные координаты сохраняются в профиле пользователя

**Кэширование геоданных:**

1. Адреса учреждений геокодируются при добавлении
2. Координаты сохраняются в базе данных
3. Создаются индексы по полям latitude/longitude для быстрого поиска (пространственные индексы PostGIS/GiST не используются, требуется доработка для масштабирования)
4. Кэш обновляется при изменении адреса

#### 6.1.3 Модели данных геолокации

**Address (Структурированные адреса):**

- `country` - страна (код ISO 3166-1 alpha-2)
- `region` - регион/область
- `city` - город (обязательное поле)
- `district` - район города
- `street` - улица
- `house_number` - номер дома
- `building` - корпус/строение
- `apartment` - номер квартиры/офиса
- `postal_code` - почтовый индекс
- `formatted_address` - форматированный адрес
- `point` - географические координаты (PostGIS PointField)
- `validation_status` - статус валидации (pending/valid/invalid/error)
- `is_valid` - флаг валидности адреса
- `is_geocoded` - флаг наличия координат
- `geocoding_accuracy` - точность геокодирования
- `created_at`, `updated_at`, `validated_at` - временные метки

**AddressValidation (Валидация адресов):**

- `address` - связь с моделью Address
- `is_valid` - результат валидации
- `confidence_score` - оценка достоверности (0.00-1.00)
- `validation_errors` - список ошибок валидации (JSON)
- `suggestions` - предложения по исправлению (JSON)
- `api_provider` - провайдер API (google_maps, yandex_maps)
- `api_response` - сырой ответ API (JSON)
- `processing_time` - время обработки запроса
- `created_at` - дата создания

**AddressCache (Кэширование геоданных):**

- `cache_key` - ключ кэша (хеш адреса)
- `address_data` - кэшированные данные адреса (JSON)
- `api_provider` - провайдер API
- `created_at` - дата создания
- `expires_at` - дата истечения кэша
- `hit_count` - количество использований кэша

**LocationHistory (История местоположений):**

- `user` - пользователь
- `point` - координаты (PostGIS PointField)
- `address` - адрес
- `city`, `country`, `postal_code` - компоненты адреса
- `created_at` - дата создания записи

**SearchRadius (Радиусы поиска):**

- `user` - пользователь
- `name` - название радиуса
- `radius` - радиус в метрах
- `is_active` - активность настройки

**UserLocation (Локации пользователей):**

- `user` - пользователь (OneToOne связь)
- `point` - координаты (PostGIS PointField)
- `accuracy` - точность в метрах
- `source` - источник данных (device/map/manual)
- `last_updated` - дата последнего обновления

**Функциональность моделей:**

- **Автоматическое геокодирование** при создании адреса
- **Валидация через внешние API** с кэшированием результатов
- **Пространственные индексы** для быстрого поиска по координатам
- **История изменений** местоположений пользователей
- **Настраиваемые радиусы** поиска для каждого пользователя
- **Множественные источники** данных о местоположении

**API эндпоинты:**

- `GET/POST/PUT/PATCH/DELETE /addresses/` - CRUD адресов
- `GET/POST/PUT/PATCH/DELETE /validations/` - CRUD валидаций
- `GET/POST/PUT/PATCH/DELETE /cache/` - CRUD кэша
- `GET /autocomplete/` - автодополнение адресов
- `GET /geocode/` - геокодирование
- `GET /reverse-geocode/` - обратное геокодирование
- `POST /validate-bulk/` - массовая валидация
- `GET /user-location/` - местоположение пользователя
- `POST /device-location/` - сохранение местоположения устройства
- `POST /map-location/` - сохранение местоположения с карты

### 6.2 Поиск услуг

**Алгоритм поиска:**

1. Пользователь вводит поисковый запрос (с поддержкой синонимов) или выбирает категорию услуги (навигация по иерархическому дереву каталога).
2. **Система применяет поиск с учетом синонимов**, маршрутизируя неформальные запросы (например, "прививка" или "укол") на канонические услуги (например, "Вакцинация"). Поиск инкапсулирован на уровне PostgreSQL через `ArrayField` и фильтрацию с аппаратным ускорением с помощью `GinIndex`.
3. **Система автоматически фильтрует услуги по типу животного** (если пользователь выбрал питомца).
4. Пользователь выбирает конкретную услугу из найденных или доступных для типа животного.
5. Дата и время оказания услуги могут быть выбраны, но не являются обязательными для заполнения.

- Если не указано время — система отображает первое доступное время записи в выбранный день.
- Если не указана дата — система отображает первое доступное окно вообще.
- Если указаны дата и/или время как ограничения снизу, для всех локаций выводятся первые свободные окна начиная с указанного времени.

2. Система определяет текущее местоположение
3. Выполняется поиск **локаций** в радиусе 10 км (по умолчанию)
4. Применяются фильтры: рейтинг, цена, доступность
5. Результаты сортируются по релевантности

**Фильтрация результатов:**

- По радиусу (географическому расстоянию)
- По специализации
- По доступности записи
- По цене (диапазон)
- По рейтингу (минимальный рейтинг)

**Сортировка результатов:**

- По расстоянию (ближайшие первыми)
- По рейтингу учреждения
- По цене услуги
- По доступности услуги

**Фильтрация по питомцам (ГЭП):**

- По типу питомца (кошки, собаки, птицы и т.д.)
- По породе питомца
- По возрасту/периоду жизни питомца
- По медицинским условиям питомца
- По особым потребностям питомца
- По весу питомца (для услуг, где это важно)

**Планируемые улучшения поиска:**

- Интеграция с профилями питомцев пользователя
- Персонализированные рекомендации на основе истории питомца
- Фильтрация по совместимости услуг с конкретным питомцем
- Учет медицинских противопоказаний
- Рекомендации по периодическим процедурам

### 6.3 Отображение на карте

**Алгоритм отображения:**

1. Загружается карта с центром в местоположении пользователя
2. Отображаются маркеры учреждений и ситтеров в радиусе поиска
3. При клике на маркер показывается информация об учреждении или ситтере
4. Отображается расстояние и время в пути
5. Показывается доступность записи/услуг (для выбранной услуги и времени, либо ближайшее доступное окно)

- Если не указано время — система отображает первое доступное время записи в выбранный день.
- Если не указана дата — система отображает первое доступное окно вообще.
- Если указаны дата и/или время как ограничения снизу, для всех учреждений выводятся первые свободные окна начиная с указанного времени.

**Что отображается на карте:**

- **Локации учреждений (ProviderLocation)** - конкретные точки предоставления услуг (ветклиники, зоомагазины, груминг-салоны)
- **Ситтеры (Sitter)** - частные передержчики питомцев
- **Информация о маркерах:**
  - Название локации/ситтера
  - Адрес и контакты локации
  - Рейтинг и отзывы
  - Доступные услуги в этой локации
  - Расстояние от пользователя
  - Статус доступности (для выбранной услуги и времени, либо ближайшее доступное окно)
  - Если не указано время — система отображает первое доступное время записи в выбранный день.
- Если не указана дата — система отображает первое доступное окно вообще.
- Если указаны дата и/или время как ограничения снизу, для всех локаций выводятся первые свободные окна начиная с указанного времени.

**Фильтрация на карте:**

- По типу (только локации учреждений, только ситтеры, все)
- По услугам (конкретные услуги)
- По дате и/или времени оказания услуги
- По рейтингу (минимальный рейтинг)
- По расстоянию (радиус поиска)
- По цене (максимальная цена для ситтеров)

## 7. СИСТЕМА БРОНИРОВАНИЯ

### 7.1 Алгоритм бронирования (целевой)

1. Пользователь выбирает услугу и (опционально) дату/время.
2. Система отображает на карте **локации** в радиусе поиска с ближайшими свободными окнами для выбранной услуги, начиная с указанной даты/времени (если не указано — с 00:00:01 сегодняшнего дня), с ценами.
3. Пользователь фильтрует локации по нужным параметрам и выбирает подходящую.
4. При клике на локацию система показывает ближайшие свободные окна на заданный горизонт (например, 7/14/30 дней) от выбранной даты/времени.
5. Пользователь выбирает окно и инициирует бронирование.
6. Система пытается провести бронирование атомарно. Если окно уже занято — сообщает об ошибке, убирает окно из доступных и предлагает выбрать другое.
7. Процесс повторяется до успешного бронирования или отказа пользователя.
8. Система реализует защиту от злоупотреблений (лимиты, антибот, rate limit и т.д.).

**Примечание:** Бронирование работает на уровне **локации** (ProviderLocation), а не организации. Пользователь бронирует услугу в конкретной локации с конкретным адресом.

**Актуальная backend-логика availability и slot generation:**

1. Проверяется, что пользователь является owner/co-owner выбранного питомца и имеет право инициировать запись.
2. Для pet/service/location ищутся только активные услуги филиала, совместимые с типом животного и размерной категорией питомца.
3. Система строит реальные окна на основе расписания **локации**, расписания конкретных сотрудников, их активности в филиале, выходных, отпусков и больничных.
4. Для каждого потенциального окна выполняется единая валидация, которая используется и в поиске, и при создании бронирования.
5. Проверяются конфликты по самому питомцу, по владельцам питомца и по `escort_owner`: один и тот же человек не может сопровождать двух питомцев в пересекающиеся интервалы.
6. Для каждого окна учитывается не только длительность услуги, но и `occupied_duration_minutes` - фактическая занятая длительность, включая travel buffer, если он требуется политикой маршрутизации.
7. Проверяется логистическая достижимость соседних визитов сотрудника через routing API. Если маршрут между бронированиями не укладывается во времени, слот не показывается как доступный.
8. В результате поиск возвращает только реально bookable slots; если на этапе создания один из параметров изменился конкурентно, бронирование отклоняется атомарно с бизнес-ошибкой.

**Правила escort owner:**

- У каждого бронирования обязательно есть `escort_owner`.
- Если у питомца только один owner, он выбирается автоматически.
- Если owners несколько и возможен только один сопровождающий без конфликтов, система может определить его автоматически.
- Если owners несколько и есть неоднозначность, требуется явный выбор `escort_owner_id` перед созданием записи.
- Предвалидация бронирования возвращает `requires_escort_assignment` и `possible_escort_owner_ids`, чтобы фронтенд мог запросить выбор у пользователя до финального `create`.

### 7.2 Управление записями

**Создание записи:**

1. Фронтенд перед финальным созданием может вызвать предвалидацию черновика бронирования.
2. Предвалидация использует ту же доменную логику, что и финальное создание, и возвращает:
   - возможность бронирования (`is_bookable`);
   - необходимость явного выбора escort (`requires_escort_assignment`);
   - список допустимых owners для escort (`possible_escort_owner_ids`);
   - итоговые `start_time`, `end_time`, `occupied_duration_minutes`, цену и найденные конфликты.
3. Финальное создание требует выбранные сущности (`pet`, `service`, `provider_location`, `employee`, `start_time`) и при необходимости `escort_owner_id`.
4. При успехе создаётся запись с фиксированными `code`, `escort_owner`, `occupied_duration_minutes`, `price`, `start_time` и `end_time`.
5. Legacy DRF create и новый flow create используют один и тот же transaction service.

**Алгоритм отмены записи (и новая структура статусов):**

**Канонические статусы бронирования:**
- `active` — ожидается или в процессе;
- `completed` — успешно завершено;
- `cancelled` — отменено.

*Статусы "ожидает подтверждения", "неявка", "просрочено" не являются каноническими статусами записи в БД.* Статус "просрочено" (`overdue`) вычисляется исключительно на стороне UI, если время окончания визита (`end_time`) меньше текущего, а статус по-прежнему `active`.

**Общие правила отмены:**
1. **Любая отмена требует обязательного указания причины.** Причина состоит из кода (`cancellation_reason_code`) из справочника логических причин (`BookingCancellationReason`) и опционального текстового пояснения. Отмена и ее причина фиксируются в отдельной связанной таблице (`BookingCancellationReasonLog` или аналог в теле бронирования, как реализовано).
2. **Кто отменил (`cancelled_by`):** В системе явно фиксируется статус стороны — `client` или `provider`. От этого зависит отображение "Отменено" на фронтенде (например, "Отменено клиентом", "Отменено организацией").
3. Клиент может отменить только своё бронирование **до его начала** (`start_time`/`end_time`). Запоздалая отмена со стороны клиента ("переписывание истории" постфактум) недопустима.
4. Провайдер (управляющий или сотрудник) может отменить бронирование локации **до и после его завершения**. Это необходимо для фиксации проблем, выявленных постфактум.
5. **Неявка клиента (No-show)** — это более не статус бронирования. Это комбинация из отмены провайдером (`cancelled_by = provider`), специфической причины отмены и флага фиксации отсутствия клиента (`client_attendance = no_show`).
6. **Наличие клиента (Attendance):** Для бронирования хранится факт `client_attendance` (`unknown`, `arrived`, `no_show`), позволяющий отличить неявку клиента от отмены услуги по вине провайдера при пришедшем клиенте.
7. При больничном сотрудника система пытается автоматически перепланировать бронирования на других сотрудников той же локации. Если это невозможно — бронирование остаётся без исполнителя и информируется администратор. При деактивации локации все активные и будущие бронирования локации автоматически отменяются.

**Завершение бронирований (Ручное и Автоматическое):**

**Ручное завершение бронирований:**
- Администратор или профильный сотрудник локации может вручную перевести активное бронирование в статус `completed`.
- Допускается ручное завершение бронирования как в момент его окончания, так и значительно позже (после `end_time`).

**Автоматическое завершение бронирований:**

1. **Периодическая проверка** - система автоматически проверяет все активные бронирования с заданной периодичностью (по умолчанию раз в час).
2. **Критерии автозавершения** - бронирования, чье окно (время) закончилось, находящиеся по-прежнему в статусе `active` более N дней после завершения (`end_time`), и не завершённые вручную (настраиваемый параметр timeout, например, N=1 день).
3. **Автоматическое действие** - такие бронирования переводятся в статус `completed` (`completed_by_system=True`, `completed_by=None` или аналог) без вмешательства пользователей и без назначения "фальшивого" пользователя-исполнителя. Факт автозавершения системой явно протоколируется.
4. **Уведомления** - клиенту и учреждению отправляются уведомления об автоматическом завершении записи.

**Бизнес-правила (обновленные):**

- Применяется только к бронированиям со статусом `active`.
- Переход возможен только `active -> completed` или `active -> cancelled`. Отмененные не могут быть завершены, так же как и завершенные не могут быть отменены в штатном процессе.
- При автозавершении система логирует действие с указанием причины (system timeout).
- Все связанные данные (платежи, комиссии, рейтинги) обрабатываются как при обычном завершении.
- `escort_owner` обязателен на уровне БД и бизнес-логики
- `occupied_duration_minutes` хранит snapshot фактически занятого окна и не должен пересчитываться ретроактивно после изменения параметров услуги
- При недоступности routing API создание нового бронирования, требующего логистической проверки, не выполняется "по приближению", а возвращает явную ошибку
- Django admin для бронирований должен позволять быстро проверить creator, escort owner, всех owners питомца, provider/location, сотрудника, длительность окна и статус записи

**Интеграция:**

- Периодические задачи Celery для автоматической проверки
- Интеграция с системой уведомлений
- Логирование всех операций автозавершения для аудита

## 8. СИСТЕМА РЕЙТИНГОВ И ОТЗЫВОВ

### 8.1 Оценка услуг

**Алгоритм оценки:**

1. После завершения записи система предлагает (но не требует) оценить услугу.
2. Пользователь может поставить оценку от 1 до 5 звёзд и/или оставить текстовый отзыв (оба действия опциональны, но если создаётся отзыв — рейтинг обязателен).
3. Система рассчитывает средний рейтинг **локации** по всем оставленным оценкам.
4. Отзыв публикуется после модерации.

**Примечание:** Рейтинги и отзывы работают на уровне **локации** (ProviderLocation), а не организации. Пользователь оценивает конкретную локацию, где получил услугу.

**Расчет рейтинга:**

- Итоговый рейтинг рассчитывается по формуле:
  rating = reviews_score * reviews_weight + complaints_score * complaints_weight + cancellations_score * cancellations_weight + no_show_score * no_show_weight
- Веса по умолчанию: reviews_weight = 0.6, complaints_weight = 0.25, cancellations_weight = 0.1, no_show_weight = 0.05. Веса можно изменить через админку для каждой локации/специалиста.
- Новые отзывы должны иметь больший вес за счёт экспоненциального затухания (чем старше отзыв, тем меньше его влияние на рейтинг). В текущей реализации это не реализовано, все отзывы учитываются одинаково.
- Рейтинг организации может быть агрегирован из рейтингов всех её локаций (для отчетов, не хранится в модели).

### 8.2 Модерация отзывов

**Алгоритм модерации:**

1. Пользователь отправляет отзыв.
2. Система отправляет текст отзыва в Google Perspective API.
3. API возвращает оценки по категориям (токсичность, оскорбления, угроза, спам и др.).
4. Если все оценки ниже установленных порогов — отзыв публикуется автоматически.
5. Если хотя бы по одной категории превышен порог — отзыв отправляется на ручную модерацию или отклоняется автоматически (в зависимости от политики).
6. Пользователь получает уведомление о результате модерации.

**Пороговые значения для категорий** (например, TOXICITY > 0.8, INSULT > 0.7, THREAT > 0.7, SPAM > 0.7) настраиваются в административной панели.

**Ручные фильтры, списки запрещённых слов и локальные проверки не используются.**

**Преимущества:**

- Высокая точность и актуальность фильтрации
- Нет необходимости поддерживать собственные списки слов
- Простота масштабирования

**Опционально:**

- Для спорных случаев (оценки близки к порогам) — ручная модерация администратором.

## 9. СИСТЕМА УВЕДОМЛЕНИЙ

### 9.1 Email-уведомления

**Типы уведомлений:**

- Подтверждение регистрации
- Напоминания о записях
- Уведомления об отменах
- Новые отзывы
- Изменения в расписании

**Алгоритм отправки:**

1. Система определяет событие для уведомления
2. Выбирает шаблон письма
3. Заполняет персональные данные
4. Отправляет email через Gmail API с OAuth2 аутентификацией
5. При недоступности Gmail API автоматически переключается на SMTP
6. Логирует результат отправки

**Технические особенности:**

- **Основной канал:** Gmail API с OAuth2 токенами
- **Резервный канал:** SMTP fallback при ошибках Gmail API
- **Автоматическое управление токенами:** обновление истекших токенов
- **Поддержка форматов:** HTML и текстовые письма
- **Вложения:** поддержка файловых вложений
- **Надежность:** гарантированная доставка через fallback механизм

**Настройки системы:**

- Включение/выключение Gmail API
- Настройки SMTP fallback
- Пути к файлам токенов и учетных данных
- Области доступа OAuth2

### 9.2 Push-уведомления

**Алгоритм push-уведомлений:**

1. Пользователь разрешает push-уведомления в браузере или мобильном приложении (Android/iOS)
2. Система сохраняет токен устройства (Web Push, FCM для Android, APNS для iOS)
3. При наступлении события формируется уведомление
4. Система определяет тип устройства пользователя:
   - для браузера — отправляет через Web Push API
   - для Android — отправляет через Firebase Cloud Messaging (FCM)
   - для iOS — отправляет через Apple Push Notification Service (APNS)
5. Пользователь получает уведомление на соответствующем устройстве (браузер или мобильное приложение)

### 9.3 Напоминания о процедурах

**Алгоритм напоминаний:**

1. Система анализирует медицинские карты питомцев
2. Определяет периодические процедуры (прививки, осмотры)
3. Рассчитывает дату следующей процедуры
4. Отправляет напоминание за 7 дней
5. Повторяет напоминание за 1 день

**Алгоритм напоминаний о процедурах:**

1. При создании напоминания (например, при бронировании или для периодической процедуры) в базе данных создаётся запись с параметрами: питомец, услуга/процедура, описание, частота (ежедневно, еженедельно, ежемесячно, ежегодно, по индивидуальному интервалу), дата начала, дата окончания (опционально), и временем следующего напоминания (`next_notification`).
2. Сервис-планировщик (например, Celery Beat) периодически (раз в N минут/часов) сканирует таблицу напоминаний.
3. Для всех активных напоминаний, у которых наступило время (`next_notification <= now`), система автоматически отправляет уведомление пользователю (email, push, in-app).
4. После отправки система обновляет время последнего уведомления (`last_notified`) и пересчитывает время следующего напоминания (`next_notification`) согласно заданной частоте.
5. Пользователь может настраивать параметры напоминания (частота, канал доставки, отключение), но отправка происходит автоматически.

### 9.4 Гибкая система правил уведомлений

**Модель NotificationRule:**

- `event_type` - тип события (booking_created, booking_cancelled, price_changed, etc.)
- `condition` - условие срабатывания (Python-выражение для оценки)
- `template` - шаблон уведомления
- `priority` - приоритет правила (low, medium, high, critical)
- `channels` - каналы доставки (email, push, in_app)
- `is_active` - активность правила
- `inheritance` - наследование (global, user_specific)
- `created_by` - создатель правила
- `created_at` - дата создания
- `updated_at` - дата обновления

**Админский интерфейс управления правилами:**

1. **Создание правила:**

   - Выбор типа события из списка доступных
   - Настройка условия срабатывания через визуальный редактор
   - Выбор шаблона уведомления
   - Установка приоритета и каналов доставки
   - Настройка наследования (глобальное/пользовательское)
2. **Редактирование правила:**

   - Изменение всех параметров активного правила
   - Временное отключение/включение правила
   - Просмотр статистики срабатываний
3. **Управление шаблонами:**

   - Создание новых шаблонов уведомлений
   - Редактирование существующих шаблонов
   - Предварительный просмотр шаблонов
   - Валидация синтаксиса шаблонов

**Примеры условий срабатывания:**

- `user.is_vip == True` - только для VIP-клиентов
- `booking.total_amount > 1000` - для дорогих бронирований
- `hours_before_start < 24` - за 24 часа до начала
- `price_increase_percent > 10` - повышение цены более чем на 10%
- `user.booking_count > 5` - для постоянных клиентов

**Система приоритетов:**

- **Critical** - критические уведомления (отмены, блокировки)
- **High** - важные уведомления (VIP-клиенты, дорогие услуги)
- **Medium** - стандартные уведомления (подтверждения, напоминания)
- **Low** - информационные уведомления (новости, обновления)

**Наследование правил:**

1. **Глобальные правила** - применяются ко всем пользователям
2. **Пользовательские правила** - переопределяют глобальные для конкретного пользователя
3. **Приоритет пользовательских** - пользовательские правила имеют приоритет над глобальными

**Флоу обработки событий:**

1. **Событие происходит** (создание бронирования, отмена, изменение цены)
2. **Система определяет тип события** и получает все активные правила для этого типа
3. **Правила сортируются по приоритету** (critical → high → medium → low)
4. **Для каждого правила:**
   - Проверяется условие срабатывания
   - Если условие выполняется, отправляется уведомление по шаблону
   - Логируется срабатывание правила
5. **Пользователь получает уведомления** по выбранным каналам

**Валидация и аудит:**

- **Валидация условий** - проверка корректности Python-выражений
- **Тестирование правил** - возможность протестировать правило на тестовых данных
- **Логирование срабатываний** - запись всех срабатываний правил
- **Статистика использования** - количество срабатываний, успешных доставок
- **Мониторинг производительности** - время обработки правил

**API для управления правилами:**

- `GET /api/admin/notification-rules/` - список всех правил
- `POST /api/admin/notification-rules/` - создание нового правила
- `PUT /api/admin/notification-rules/<id>/` - редактирование правила
- `DELETE /api/admin/notification-rules/<id>/` - удаление правила
- `POST /api/admin/notification-rules/<id>/test/` - тестирование правила
- `GET /api/admin/notification-rules/<id>/stats/` - статистика правила

## 10. СИСТЕМА ПРАВИЛ ЗЛОУПОТРЕБЛЕНИЙ

### 10.1 Правила злоупотреблений по отмене бронирований

**Модель AbuseRule:**

- `name` - название правила
- `description` - описание правила
- `period` - период для подсчета отмен (день, неделя, месяц, год)
- `max_cancellations` - максимальное количество отмен за период
- `is_active` - активность правила

**Модель BookingCancellation:**

- `booking` - связь с бронированием
- `cancelled_by` - пользователь, зафиксировавший отмену
- `cancelled_by_side` - сторона отмены (`client` или `provider`)
- `reason_code` - структурированная причина отмены
- `reason` - произвольный текст/пояснение к отмене
- `client_attendance` - факт явки клиента (`unknown`, `arrived`, `no_show`)
- `is_abuse` - флаг злоупотребления
- `abuse_rule` - связь с нарушенным правилом

**Нормализация бизнес-семантики отмен:**

- `no_show` не считается отдельным lifecycle status; это provider-side cancellation reason при `client_attendance=no_show`
- кейс «клиент пришёл, но услуга не оказана» не должен оформляться как `client cancel`
- если итоговая проверка подтверждает, что услуга не была оказана по вине/невозможности провайдера, финальная запись должна становиться `cancelled` с `cancelled_by_side=provider` и `client_attendance=arrived`
- если клиент пришёл и отказался от услуги на месте, это отдельный кейс `cancelled_by_side=client` + `client_attendance=arrived`

**Алгоритм проверки злоупотреблений:**

1. При создании отмены бронирования система автоматически проверяет все активные правила
2. Для каждого правила подсчитывается количество отмен пользователя за указанный период
3. Если количество превышает лимит, отмена помечается как злоупотребление
4. Связывается с конкретным правилом, которое было нарушено

### 10.2 Защита от накрутки рейтингов

**Система обнаружения подозрительной активности:**

- **Массовые отзывы** - более 5 отзывов за 24 часа
- **Фальшивые жалобы** - более 80% отклоненных жалоб
- **Манипуляции с рейтингами** - все одинаковые оценки или крайние оценки (1/5)

**Модерация отзывов:**

- Интеграция с Google Perspective API для анализа токсичности
- Автоматическая проверка подозрительных паттернов
- Fallback на локальные правила при недоступности API

**Проверка доступа к отзывам:**

- Только пользователи, которые действительно пользовались услугами, могут оставлять отзывы
- Для сотрудников - проверка через бронирования
- Для учреждений - доступ у всех пользователей

### 10.3 Админский интерфейс

**Управление правилами злоупотреблений:**

- Создание и редактирование правил
- Настройка периодов и лимитов
- Активация/деактивация правил

**Мониторинг нарушений:**

- Просмотр отмен с флагом злоупотреблений
- Анализ подозрительной активности
- Статистика нарушений

### 10.4 Интеграция с системой рейтингов

**Влияние на рейтинг:**

- Отмены влияют на рейтинг через `cancellations_weight`
- Жалобы влияют на рейтинг через `complaints_weight`
- No-show влияет на рейтинг через `no_show_weight`
- Подтверждённый кейс «клиент пришёл, но услуга не оказана» влияет на метрики провайдера как provider-side operational cancellation, а не как клиентская отмена

**История изменений:**

- Отслеживание всех изменений рейтингов
- Причины изменений
- Автор изменений

## 11. БИЛЛИНГ И ПЛАТЕЖИ

### 11.1 Система контрактов

**Примечание:** Контракты заключаются с **организацией** (Provider), а не с локацией. Все локации организации работают по одному договору. Биллинг и проверка задолженности работают на уровне организации.

**Алгоритм создания и активации контракта:**

1. **Стандартные условия сотрудничества** (включая комиссии, сроки оплаты и базовые юридические положения) задаются администратором системы через справочник типов контрактов (`ContractType`). Для каждого типа контракта могут быть определены:
   - Стандартные комиссии (процент/фиксированная сумма, периодичность)
   - Стандартные сроки оплаты (отсрочка, период выставления счетов)
   - Общие условия сотрудничества (права, обязанности, ответственность, порядок расторжения и др.)
   - Пороги блокировки (наследуются из глобальных, но могут быть изменены менеджером)
2. **Менеджер по биллингу** может создать новый контракт для **организации**, выбрав тип контракта и, при необходимости, изменив отдельные параметры (комиссии, сроки оплаты, условия).
3. **Если все параметры контракта соответствуют стандартным для выбранного типа** (комиссии, сроки оплаты, условия сотрудничества не изменялись), менеджер может сразу активировать контракт самостоятельно — статус меняется на `'active'`.
4. **Если хотя бы один из параметров отличается от стандартных** (например, индивидуальная комиссия, особый срок оплаты, изменённые условия сотрудничества, блокировок), контракт создаётся в статусе `'draft'` и требует обязательного утверждения администратором системы. Только после утверждения админом статус может быть изменён на `'active'`.
5. **Подписание контракта**:
   Для контрактов допускается электронная оферта (акцепт условий через интерфейс, "галочка").
6. **Все действия по созданию, изменению и активации контракта логируются с указанием пользователя, времени и причины.**
7. **Все локации организации работают по одному договору** - комиссии начисляются на организацию, блокировка применяется ко всем локациям.

### 11.2 Управление менеджерами по биллингу

**Примечание:** Менеджеры по биллингу работают с **организациями** (Provider), а не с локациями. Один менеджер может управлять несколькими организациями.

**Назначение:** Обеспечивает назначение и управление менеджерами по биллингу для организаций, включая отслеживание их деятельности и автоматическое уведомление о важных событиях.

**Алгоритм работы:**

1. Системный администратор назначает менеджера по биллингу для организации при одобрении заявки (через действие "Одобрить и назначить биллинг-менеджера")
2. Менеджер может управлять несколькими организациями одновременно
3. Система отслеживает статус управления (активно, в отпуске, временно, завершено)
4. При блокировке организации система автоматически уведомляет назначенного менеджера
5. Поддерживается назначение временных заместителей на период отпуска
6. **Примечание:** Реквизиты организации (ИНН, регистрационный номер) заполняются провайдером в мастере регистрации (Шаг 3) до назначения биллинг-менеджера
7. Биллинг-менеджер имеет доступ только к своим назначенным организациям через BillingManagerProvider

**Права доступа биллинг-менеджера в админке Django:**

- **Доступные разделы:**
  - Provider (Провайдеры) - только назначенные организации (через BillingManagerProvider)
  - Contract (Договоры) - полный доступ к договорам назначенных организаций
  - Invoice (Счета) - полный доступ
  - Payment (Платежи) - полный доступ
  - Refund (Возвраты) - полный доступ
  - BillingManagerEvent (События биллинг-менеджера) - только свои события
  - BillingManagerProvider (Провайдеры биллинг-менеджера) - только просмотр (назначение утверждается системным админом)
  - BlockingNotification (Уведомления о блокировках) - полный доступ
  - BlockingRule (Правила блокировки) - только просмотр
  - ProviderBlocking (Блокировки провайдеров) - только просмотр (блокировка/разблокировка утверждается системным админом)
  - ContractType (Типы договоров) - только просмотр
  - ContractCommission (Комиссии договоров) - только просмотр (утверждается системным админом)
  - ContractDiscount (Скидки договоров) - только просмотр (утверждается системным админом)
  - Currency (Валюты) - только просмотр
- **Недоступные разделы:**
  - ProviderForm (Заявки провайдеров) - биллинг-менеджер НЕ видит заявки, только одобренные организации
  - User, UserType - управление пользователями
  - Pet, PetType, Breed - управление питомцами
  - Booking, BookingStatus - управление бронированиями
  - Employee, Schedule - управление сотрудниками
  - Service, ServiceCategory - управление услугами
  - Rating, Review - управление отзывами
  - Notification, Reminder - управление уведомлениями

**Модели данных:**

**Связь менеджера с организацией (BillingManagerProvider):**

- Связь с менеджером по биллингу (billing_manager)
- Связь с организацией (provider)
- Дата начала управления (start_date)
- Статус управления ('active', 'vacation', 'temporary', 'terminated')
- Временный менеджер (temporary_manager) - для периодов отпуска
- Примечания (notes)
- Автоматическое создание событий при изменениях

**События менеджера (BillingManagerEvent):**

- Связь с BillingManagerProvider
- Тип события ('assigned', 'vacation_start', 'vacation_end', 'temporary_assigned', 'terminated')
- Дата вступления в силу (effective_date)
- Создатель события (created_by)
- Примечания (notes)
- Полная история для аудита

**API эндпоинты:**

- `GET /api/billing-managers/` - список менеджеров по биллингу
- `GET /api/billing-managers/{id}/` - детали менеджера
- `POST /api/billing-managers/` - создание связи менеджера с организацией
- `PUT /api/billing-managers/{id}/` - обновление связи
- `DELETE /api/billing-managers/{id}/` - удаление связи
- `GET /api/billing-managers/{id}/events/` - история событий менеджера
- `POST /api/billing-managers/{id}/start_vacation/` - начало отпуска
- `POST /api/billing-managers/{id}/end_vacation/` - завершение отпуска
- `POST /api/billing-managers/{id}/terminate/` - завершение управления

**Интеграция с системой блокировок:**

- Автоматические уведомления менеджеров о блокировках организаций
- Настройка уведомлений в BlockingSystemSettings
- Получение списка активных менеджеров для организации при блокировке
- Поддержка временных менеджеров в период отпуска

**Управление статусами:**

- **active** - менеджер активно управляет организацией
- **vacation** - менеджер в отпуске, назначен временный заместитель
- **temporary** - временное управление другим менеджером
- **terminated** - управление завершено

**Безопасность и аудит:**

- Полная история всех изменений через события
- Отслеживание кто, когда и почему принял решение
- Поддержка compliance и аудита
- Автоматическое создание событий при изменениях статуса

### 11.3 Обработка платежей между поставщиками услуг и сервисом

**Примечание:** Платежи и счета работают на уровне **организации** (Provider), а не локации. Все бронирования во всех локациях организации учитываются в одном счете.

**Алгоритм обработки платежей:**

1. После завершения бронирования система фиксирует оказанную услугу в конкретной локации и начисляет комиссию согласно условиям контракта **организации**.
2. На основании данных о бронированиях во всех локациях организации и других операциях формируется счёт (Invoice) для **организации** — с детализацией начисленных комиссий, абонентских плат, штрафов и т.д.
3. **Организации** выставляется счёт на оплату через личный кабинет И по email.
4. После поступления оплаты от организации оператор или бухгалтерия вручную подтверждает платёж в системе (изменяет статус платежа/счёта на "оплачен").
5. Система обновляет статус счёта и уведомляет организацию о зачислении средств.

**Расчёт комиссии:**

- Применяется процент или фиксированная сумма согласно условиям контракта организации (Contract/ContractType).
- Учитываются все бронирования во всех локациях организации.
- Учитываются скидки, бонусы, штрафы, абонентские платы (если предусмотрено контрактом).
- Формируется отчётность для организации и платформы.

#### 11.3.1 Расчет НДС (VAT)

**Общее описание:**
Система поддерживает автоматический расчет НДС (VAT) при создании счетов (Invoice) для провайдеров услуг. Расчет НДС зависит от статуса провайдера как плательщика НДС и его географического расположения.

**Логика расчета НДС:**

**Если провайдер НЕ плательщик НДС (`is_vat_payer = False`):**

- Базовая комиссия = `booking_amount * commission_percent / 100`
- НДС начисляется сверх комиссии
- Итого в счете = `commission * (1 + vat_rate/100)`
- Пример: 100€ услуга, 5% комиссия, 20% НДС
  - Комиссия = 5€
  - НДС = 5€ * 0.20 = 1€
  - С НДС = 5€ + 1€ = 6€

**Если провайдер плательщик НДС (`is_vat_payer = True`):**

- Для ЕС: Reverse Charge (0% НДС, провайдер сам начисляет)
- Для других стран: зависит от законодательства

**Модели данных:**

**VATRate (Ставки НДС по странам):**

- `country` - код страны (ISO 3166-1 alpha-2)
- `rate` - ставка НДС (например, 20.00 для 20%)
- `effective_date` - дата вступления в силу
- `is_active` - активна ли ставка
- Метод `get_rate_for_country(country_code, date=None)` для получения актуальной ставки

**InvoiceLine (Строки счета):**

- `vat_rate` - ставка НДС (Decimal, nullable)
- `vat_amount` - сумма НДС (Decimal, default=0.00)
- `total_with_vat` - итоговая сумма с НДС (Decimal, default=0.00)

**Метод расчета комиссии с НДС:**

- `Provider.calculate_commission_with_vat()` - возвращает словарь с:
  - `commission` - базовая комиссия без НДС
  - `vat_rate` - ставка НДС (или None)
  - `vat_amount` - сумма НДС
  - `total_with_vat` - итоговая сумма с НДС

**Администрирование:**

- Ставки НДС управляются через Django Admin (`/admin/billing/vatrate/`)
- Доступ: только системные администраторы могут редактировать
- Биллинг-менеджеры могут только просматривать

**Важные замечания:**

1. **Ставки НДС по странам**: Необходимо заполнить ставки НДС для всех стран, где работают провайдеры
2. **Reverse Charge для ЕС**: Для провайдеров из ЕС с валидным VAT ID применяется Reverse Charge (0% НДС)
3. **Обновление ставок**: При изменении ставок НДС в стране создается новая запись с новой `effective_date`
4. **Исторические данные**: Старые InvoiceLine сохраняют ставку НДС, которая была актуальна на момент создания

### 11.4 Система блокировки

**Примечание:** Блокировка работает на уровне **организации** (Provider), а не локации. При блокировке организации блокируются все её локации.

**Алгоритм блокировки:**

1. Система проверяет задолженность **организации**
2. Сравнивает с пороговыми значениями
3. При превышении применяет блокировку ко всей организации (всем локациям)
4. Отправляет уведомления ответственным лицам (биллинг-менеджерам)
5. Блокирует доступ к функциям согласно уровню (все локации организации)

**Периодика проверки задолженности:**

- Автоматические проверки выполняются по расписанию (по умолчанию ежедневно в 02:00)
- Частота проверок настраивается
- Поддерживаются различные расписания: почасово, ежедневно, еженедельно, ежемесячно
- Проверки выполняются через Celery Beat задачи

**Уровни блокировки:**

- **Уровень 0:** Нет блокировки
- **Уровень 1:** Информационное уведомление
- **Уровень 2:** Исключение всех локаций организации из поиска
- **Уровень 3:** Полная блокировка всех локаций организации

**Флоу разблокировки:**

1. **Автоматическая разблокировка при оплате:** При поступлении платежа, покрывающего задолженность организации, система автоматически снимает блокировку со всех локаций (если включено в настройках)
2. **Ручная разблокировка:** Администратор или менеджер по биллингу может вручную снять блокировку через API или админ-панель
   - При ручном снятии блокировки администратор **обязательно** должен выполнить одно из действий:
     - Изменить индивидуальные пороги блокировки в контракте организации
     - Или исключить контракт из автоматических проверок блокировки
   - Это необходимо для предотвращения повторной автоматической блокировки при следующем анализе задолженности
3. **Уведомления о разблокировке:** При снятии блокировки отправляются уведомления организации и ответственным лицам (биллинг-менеджерам)

**Пороговые значения:**

- Настраиваются глобально в системе блокировки и наследуются в контракты
- Поддерживаются пороги по сумме задолженности
- Поддерживаются пороги по дням просрочки (3 уровня: информационное уведомление, исключение из поиска, полная блокировка)
- Учитывается отсрочка платежей
- Поддерживается исключение праздников и рабочих дней из расчёта просрочки

**Модели данных:**

**BlockingRule (Правила блокировки):**

- Настройка порогов по сумме и дням просрочки
- Массовая настройка по географии и типу услуг
- Приоритет правил (меньше = выше приоритет)
- Отслеживание активности правил

**ProviderBlocking (Блокировки организаций):**

- Отслеживание статуса блокировки организации (active, resolved, manual_override)
- Блокировка применяется ко всем локациям организации
- Данные о задолженности на момент блокировки
- История блокировок с датами и причинами
- Привязка к правилу блокировки

**BlockingTemplate (Шаблоны блокировки):**

- Географическая иерархия (страна → область → город)
- Автоматическое применение параметров при создании договоров
- Централизованное управление порогами блокировки
- Интеграция с системой геолокации

**BlockingSystemSettings (Настройки системы):**

- Централизованное управление настройками
- Настройка периодичности проверок
- Управление уведомлениями
- Автоматическое снятие блокировок

**BlockingNotification (Уведомления о блокировках):**

- Отслеживание статуса отправки уведомлений
- Различные типы уведомлений (предупреждение, активация, снятие)
- История уведомлений с деталями

**API эндпоинты:**

- `GET /api/blocking-rules/` - список правил блокировки
- `POST /api/blocking-rules/` - создание правила блокировки
- `PUT /api/blocking-rules/{id}/` - редактирование правила
- `DELETE /api/blocking-rules/{id}/` - удаление правила
- `GET /api/provider-blockings/` - список блокировок организаций
- `GET /api/provider-blockings/{id}/` - детали блокировки
- `POST /api/provider-blockings/{blocking_id}/resolve/` - снятие блокировки организации
- `GET /api/providers/{provider_id}/blocking-status/` - статус блокировки организации
- `GET /api/providers/{provider_id}/blocking-history/` - история блокировок организации
- `GET /api/blocking-notifications/` - уведомления о блокировках
- `POST /api/blocking-notifications/{notification_id}/retry/` - повтор уведомления

**Интеграция с системой уведомлений:**

- Автоматические уведомления биллинг-менеджеров о блокировках организаций
- Настройка уведомлений в BlockingSystemSettings
- Получение списка активных менеджеров для организации при блокировке
- Поддержка временных менеджеров в период отпуска
- Уведомления применяются ко всем локациям организации

## 12. АНАЛИТИКА И ОТЧЕТЫ

### 12.1 Статистика учреждений

**Типы отчетов:**

- Количество записей по периодам (по локациям)
- Выручка и комиссия (по организациям и локациям)
- Популярные услуги (по локациям)
- Рейтинг и отзывы (по локациям, агрегация по организациям)
- География клиентов (по локациям)
- Дебиторская задолженность (по организациям)

**Общие положения:**

- Все отчеты формируются вручную через административный интерфейс (или по API).
- Экспорт отчетов реализован только в формате Excel (XLSX).
- Автоматическая рассылка и PDF-экспорт не входят в MVP.
- Формирование отчетов доступно администраторам и менеджерам по необходимости.

**Логика формирования отчетов:**

1. **Количество записей по периодам**

   - Считается общее количество бронирований за выбранный период (день/неделя/месяц).
   - Группировка по организациям, локациям, услугам, сотрудникам.
   - Формирует администратор или менеджер по необходимости.
2. **Выручка и комиссия**

   - Суммируется стоимость завершённых бронирований за период.
   - Отдельно считается комиссия платформы (по данным биллинга).
   - Группировка по организациям, локациям, услугам, сотрудникам.
   - Биллинг работает на уровне организации, но отчеты могут быть по локациям.
   - Формирует администратор или менеджер по необходимости.
3. **Популярные услуги**

   - Считается количество бронирований по каждой услуге за период.
   - Сортировка по убыванию популярности.
   - Формирует администратор или менеджер по необходимости.
4. **Рейтинг и отзывы**

   - Средний рейтинг и количество отзывов по локациям и услугам.
   - Агрегация рейтингов локаций по организациям (для отчетов).
   - Группировка по организациям, локациям, услугам, сотрудникам.
   - Формирует администратор или менеджер по необходимости.
5. **География получателей услуг**

   - Строится распределение получателей услуг по городам/регионам (если есть данные о местоположении).
   - Формирует администратор или менеджер по необходимости.
6. **Дебиторская задолженность**

   - Считается сумма задолженности по каждой организации на выбранную дату/период.
   - Учитываются просроченные платежи по контрактам (договоры на уровне организации).
   - Все локации организации работают по одному договору.
   - Формирует администратор или менеджер по необходимости.

### 12.2 Аналитика пользователей

**Цель:** Понимание роста платформы и эффективности привлечения пользователей.

**Метрики, рассчитываемые в системе:**

1. **Рост пользователей**

   - Количество новых получателей услуг за период
   - Количество новых учреждений за период
   - Динамика регистраций по дням/неделям/месяцам
   - Активация новых пользователей (кто зашёл после регистрации)
2. **Активность пользователей**

   - Количество активных получателей услуг за период
   - Количество активных учреждений за период
   - Частота использования (сколько раз пользователь заходил)
3. **Конверсия в бронирования**

   - Процент зарегистрированных, которые совершили хотя бы одно бронирование
   - Время от регистрации до первого бронирования
   - Процент завершённых бронирований (не отменённых)

**Метрики для маркетинга (опционально, через Google Analytics):**

- Источники трафика (поиск, реклама, соцсети)
- География пользователей
- Устройства (мобильные/десктоп)
- Пути пользователя по сайту

**Формирование отчётов:**

- Все метрики рассчитываются на основе данных из моделей Django
- Отчёты формируются вручную через административный интерфейс
- Экспорт в Excel
- Автоматическая рассылка не входит в MVP

## 15. ПЕРИОДИЧЕСКИЕ ЗАДАЧИ

### 15.1 Система валют

**Обновление курсов валют:**

- Автоматическое обновление курсов валют через внешний API (exchangeratesapi.io)
- Периодичность: ежедневно в 00:00
- Поддерживаемые валюты: USD (базовая), EUR, RUB
- Обработка ошибок и логирование неудачных попыток обновления
- Fallback на последние известные курсы при недоступности API

### 15.2 Уведомления

**Запланированные уведомления:**

- Обработка запланированных уведомлений: каждую минуту
- Обработка напоминаний: каждый час
- Очистка старых уведомлений: ежедневно в 02:00
- Отправка напоминаний о задолженности: ежедневно в 09:00
- Проверка истечения инвайтов ролей: ежедневно в 08:00
- Статистика уведомлений администраторам: ежедневно в 18:00
- Очистка неактивных push-токенов: еженедельно по воскресеньям в 03:00
- Напоминания о предстоящих бронированиях: каждые 15 минут (индивидуально для каждого пользователя)

**Система напоминаний о бронированиях:**

- **Глобальная настройка:** Время напоминания по умолчанию (например, за 2 часа до бронирования)
- **Индивидуальная настройка:** Пользователь может изменить время напоминания в своих настройках
- **Логика работы:** Система проверяет каждые 15 минут, какие бронирования требуют напоминания
- **Индивидуальное время:** Каждый пользователь получает напоминание за свое настроенное время до бронирования
- **Множественные напоминания:** Поддержка нескольких напоминаний (например, за 1 день и за 2 часа)

### 15.3 Система блокировок

**Проверка блокировок:**

- Автоматическая проверка задолженности учреждений: ежедневно в 02:00 (настраивается)
- Применение правил блокировки согласно настройкам
- Автоматическое снятие блокировок при погашении задолженности
- Отправка уведомлений о блокировке/разблокировке

### 15.4 Задачи питомцев

**Обработка недееспособности владельцев:**

- `check_inactive_owners_task` - еженедельная проверка неактивных владельцев питомцев

  - Находит питомцев с неактивными основными владельцами
  - Создает записи о недееспособности
  - Отправляет уведомления всем владельцам
  - Логирует результаты выполнения
- `process_deadline_actions_task` - ежедневная обработка действий по истечении дедлайнов

  - Находит записи о недееспособности с истекшим дедлайном
  - Выполняет автоматические действия (удаление питомца или назначение совладельца)
  - Отправляет уведомления о выполненных действиях
  - Возвращает статистику выполненных операций
- `send_incapacity_notifications_task` - периодическая отправка уведомлений

  - Находит неотправленные уведомления о недееспособности
  - Отправляет их через email
  - Обновляет статус уведомлений
  - Логирует количество отправленных и неудачных уведомлений
- `cleanup_old_incapacity_records_task` - ежемесячная очистка старых записей

  - Удаляет записи о недееспособности старше 1 года
  - Очищает только завершенные случаи (resolved, auto_deleted, coowner_assigned)
  - Экономит место в базе данных
  - Логирует количество удаленных записей

**Особенности реализации:**

- Все задачи используют транзакционную обработку для обеспечения целостности данных
- Подробное логирование всех операций и ошибок
- Возврат детальной статистики выполнения
- Обработка исключений с сохранением контекста ошибки

- Добавляет поле `allowed_pet_types` в модель Service
- Создает промежуточную таблицу для связи Many-to-Many
- Обеспечивает обратную совместимость (пустое поле = доступно для всех типов)

**Обратная совместимость:**

- Существующие услуги без ограничений остаются доступными для всех типов животных
- Новые услуги могут создаваться с ограничениями по типам животных
- API продолжает работать с существующими клиентами

### 12.4 Преимущества реализации

**Безопасность:**

- Предотвращение назначения неподходящих процедур
- Автоматическая валидация совместимости
- Защита от ошибок пользователей

**Удобство использования:**

- Автоматическая фильтрация услуг по типу животного
- Понятные сообщения об ошибках
- API для проверки совместимости

**Гибкость:**

- Услуги могут быть доступны для всех типов (пустое поле)
- Легкое добавление новых типов животных
- Простое управление через админку

**Производительность:**

- Эффективные запросы с использованием индексов
- Кэширование результатов проверки совместимости
- Минимальное влияние на существующую функциональность
- Интеграция с системой уведомлений для отправки email

## 13. БРОНИРОВАНИЕ, ЛОГИКА СЛОТОВ И ВАЛИДАЦИЯ

### 13.1 Валидация бронирования на Backend (bookingFlowAPI)

При создании бронирования (`validate_booking_request` и `create_booking`) выполняются строгие проверки:

1. **Проверка доступности услуги для питомца:** Услуга должна быть доступна в выбранной локации и подходить животному (совпадение типа `allowed_pet_types`). Возвращает `serviceUnavailable`.
2. **Учет времени и статуса слота:** Бронирование в прошлом недоступно (`pastStartTime`), и слот не должен быть занят (`slotUnavailable`).
3. **Проверка сотрудника:**
   - Сотрудник должен работать в указанное время (`employeeNotWorking`).
   - Сотрудник должен оказывать выбранную услугу (`employeeMismatch` или `noEligibleEmployee`).
   - У сотрудника не должно быть конфликтов с другими записями (`employeeConflict`).
4. **Проверка питомца и дубликатов:**
   - Питомец не может телепортироваться между локациями или посещать две процедуры одновременно (`petConflict`). 
   - Проверка дубликатов (`has_duplicate_warning`): Если у питомца уже есть бронирование на эту же услугу в эту же дату, система возвращает предупреждение `duplicate_booking_warning`. При этом бронирование не блокируется, но frontend обязан показать дополнительный шаг подтверждения (Warning modal).
5. **Опциональный сопровождающий (Escort):**
   - Если владелец питомца не может присутствовать, необходимо выбрать сопровождающего.
   - При выборе сопровождающего система проверяет, чтобы у выбранного пользователя (`escort`) не было конфликтующих записей на это время (`escortConflict`) и он был доступен (`escortUnavailable`).

### 13.2 Обработка ошибок на Frontend (BookingModal)

Frontend использует единый маппинг (`ERROR_CODE_TO_I18N`) текстовых кодов, возвращаемых с Backend, на ключи переводов `i18n` в `BookingModal.tsx`:

- `past_time` -> `booking.errors.pastStartTime`
- `service_unavailable` -> `booking.errors.serviceUnavailable`
- `escort_unavailable` -> `booking.errors.escortUnavailable`
- `escort_conflict` -> `booking.errors.escortConflict`
- `employee_mismatch` -> `booking.errors.employeeMismatch`
- `no_eligible_employee` -> `booking.errors.noEligibleEmployee`
- `slot_unavailable` -> `booking.errors.slotUnavailable`
- `employee_not_working` -> `booking.errors.employeeNotWorking`
- `employee_conflict` -> `booking.errors.employeeConflict`
- `pet_conflict` -> `booking.errors.petConflict`

При получении с бэкенда флага `has_duplicate_warning: true`, модальное окно переводится в особое состояние `duplicate_confirm`. Пользователю показывается предупреждение («Этот питомец уже записан на эту услугу...») и дается выбор либо отменить, либо создать запись принудительно (`skipDuplicateCheck=true`).

### 13.3 Постфактум инциденты оказания услуги и dispute flow

Для сценария:

- клиент пришёл вовремя
- питомец присутствовал
- услуга не была оказана
- провайдер не хочет или не может сам отменить бронирование

стандартная клиентская отмена использоваться не должна.

Во v1 это именно клиентский report flow: кейс создаётся со стороны клиента, а провайдер/support только разбирают уже созданный report и принимают resolution.

#### Базовые принципы

- Канонический lifecycle бронирования остаётся минимальным: `active`, `completed`, `cancelled`.
- Жалоба клиента на неоказанную услугу не должна немедленно превращать бронирование в `cancelled_by=client`.
- Платформа должна разделять:
  - что репорт был инициирован клиентом
  - кто несёт ответственность за срыв оказания
  - какой финальный resolution применён к booking

#### Целевая модель

Для таких кейсов вводится отдельная сущность инцидента/спора по бронированию (например, `BookingServiceIssue`), а не новый booking status.

Минимальная семантика issue:

- `issue_type` - тип инцидента (`service_not_provided` как первый обязательный кейс)
- `reported_by_user`
- `reported_by_side=client`
- `client_attendance_snapshot`
- `description`
- `status` (`open`, `acknowledged`, `resolved`, `rejected`)
- `resolution_outcome`
- `resolved_by_user` / `resolved_by_actor`
- `resolved_at`
- `resolution_note`

Во v1 инициатор issue:

- владелец бронирования
- либо сопровождающий/совладелец только если текущие права уже позволяют ему действовать от имени booking owner без размывания ответственности

Провайдерская сторона не должна использовать этот report flow вместо своих обычных provider-side cancellation/completion действий.

#### Правила финализации booking

Если claim подтверждён в пользу клиента:

- booking переводится в `status=cancelled`
- `cancelled_by=provider`
- `client_attendance=arrived`
- reason берётся из provider-scoped причин (`service_not_possible`, `provider_unavailable` или эквивалент)

Если подтверждено, что услуга всё же была оказана:

- booking переводится в `completed`

Если установлено, что клиент пришёл, но отказался от услуги на месте:

- booking переводится в `cancelled`
- `cancelled_by=client`
- `client_attendance=arrived`
- используется client-scoped reason наподобие `client_refused_on_site`

Если выясняется, что клиент не пришёл:

- это не dispute `service_not_provided`, а обычный provider-side no-show case
- итоговая запись: `cancelled_by=provider`, `client_attendance=no_show`, reason=`client_no_show`

#### SLA и автоматическое разрешение

- Если issue не обработан провайдером в разумный SLA, он должен быть эскалирован в support/system flow.
- Авторазрешение, если оно вводится, должно явно писать actor=`system`.
- Даже при позднем репорте клиента после `end_time` это должен быть отдельный issue/report flow, а не late client cancellation.

#### Влияние на рейтинг, отчёты и биллинг

- Подтверждённый кейс «arrived but not served» должен считаться provider-side operational failure.
- Он не должен попадать в клиентские abuse/cancellation метрики.
- В отчётах и рейтингах он должен быть отличим от:
  - client no-show
  - обычной client-side cancellation
  - confirmed completion

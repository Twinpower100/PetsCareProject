# Промпт для сеньор Fullstack-разработчика (Вкладка Безопасность Аккаунта)

**Роль:** Senior Fullstack Developer (Django/DRF + React/TypeScript)
**Задача:** Реализовать вкладку «Безопасность» в личном кабинете пользователя по пути `http://localhost:3000/account` (компонент-таб Security). Задача охватывает проектирование и реализацию как API на бэкенде (`PetsCare-backend`), так и пользовательского интерфейса на фронтенде (`Petscare-web`).

Вкладка должна покрывать следующий функционал:
1. **Изменение пароля**
2. **Изменение email-адреса** (с двухэтапным подтверждением)
3. **Управление сессиями** (просмотр устройств и удаленный выход)

---

## 1. Изменение пароля (Change Password)

### Backend (Django / DRF)
- **Endpoint:** `POST /api/v1/users/security/change-password/`
- **Тело запроса:** `old_password`, `new_password`, `new_password_confirm`.
- **Бизнес-логика:**
  - Проверить корректность `old_password`.
  - Проверить совпадение `new_password` и `new_password_confirm`.
  - Удостовериться, что новый пароль удовлетворяет политикам безопасности (валидаторы Django: минимальная длина, не слишком распространенный и т.д.).
  - Сохранить новый пароль (`set_password`).
  - **Безопасность:** После успешной смены пароля необходимо инвалидировать остальные refresh токены (разлогинить пользователя с других устройств, кроме текущего) ИЛИ перевыпустить текущие токены, в зависимости от того, как настроен JWT (SimpleJWT).

### Frontend (React / TypeScript)
Mobile first
- **UI:** Форма с тремя инпутами для пароля. Для каждого инпута должна быть кнопка (иконка глаза) для переключения видимости (show/hide password).
- **Валидация:** Строгая валидация на клиенте через **Zod + React Hook Form**. Проверять совпадение паролей до отправки запроса. Показывать требования к паролю (например: минимум 8 символов, 1 цифра).
- **UX:** Состояние `isSubmitting` (disable кнопки, лоадер) во время отправки.
- **Уведомления:** При успехе — Success Toast, при ошибке (неверный старый пароль) выводить ошибку непосредственно под полем `old_password`.

## 2. Изменение E-mail (Change Email)

### Backend (Django / DRF)
- **Endpoints:** 
  1. `POST /api/v1/users/security/change-email/` — запрос на смену.
  2. `POST /api/v1/users/security/confirm-email/` — подтверждение кода.
- **Бизнес-логика (Двухшаговый флоу):**
  - Шаг 1: Пользователь передает `new_email` и свой **текущий пароль** (Password requirement for sensitive actions). Бэкенд проверяет уникальность email и пароль. Генерирует одноразовый OTP-код (сохраняет в Redis/БД с TTL = 10-15 минут) и отправляет его на *новый* email (дополнительно можно послать уведомление безопасности на старый email).
  - Шаг 2: Клиент отправляет `otp_code`. (Это происходит в том же UI: после успешного прохождения Шага 1 форма ввода email сменяется на форму ввода OTP-кода, либо открывается модальное окно). Если код валиден, бэкенд обновляет `email` в БД пользователя и помечает его как подтвержденный (`is_email_verified = True`).
  - **Важно (Откат/Отмена):** Если пользователь не вводит OTP-код в течение времени действия (TTL), закрывает страницу или нажимает "Отмена", текущий (старый) email остается **без изменений**. Имейл меняется только после успешного прохождения 2-го шага. При повторной попытке смены email старый сгенерированный код становится недействительным, генерируется новый.

### Frontend (React / TypeScript)
Mobile first
- **UI:**
  - **Шаг 1:** Форма с инпутами `Текущий пароль` и `Новый E-mail`.
  - **Шаг 2:** Компонент ввода OTP-кода (например, `Input-OTP` от Shadcn UI — 6 цифр).
- **UX:** Таймер (например, 60 сек) для повторной отправки кода. Кнопка возврата к шагу 1 на случай опечатки в email.
- **После успешной смены:** Success Toast, закрытие модалки/переключение шага. Обновление пользовательского стейта в приложении (чтобы в header и профиле отобразился новый email без перезагрузки вкладки, например, через обновление Zustand/Context или инвалидацию React Query).

## 3. Управление сессиями (Session Management)

### Backend (Django / DRF)
- Должен быть реализован механизм отслеживания активных Refresh токенов / сессий (через `OutstandingToken` в SimpleJWT или отдельную модель `UserSession`, где кэшируется User-Agent и IP).
- **Endpoints:**
  - `GET /api/v1/users/security/sessions/` — получить список активных сессий пользователя.
  - `DELETE /api/v1/users/security/sessions/<session_id>/` — завершить конкретную сессию (занести её token в blacklist).
  - `POST /api/v1/users/security/sessions/terminate-all/` — завершить все сессии, кроме текущей.
- **Отдаваемые данные:** Идентификатор сессии, IP-адрес, распарсенный User-Agent (ОС, Браузер), дата последнего входа, флаг `is_current` (текущая ли это сессия).

### Frontend (React / TypeScript)
Mobile first
- **UI:** Список устройств (Sessions list) под формой изменения пароля.
  - Каждая карточка сессии должна визуально показывать: Иконку (Desktop или Mobile, зависит от ОС), Название девайса (Windows/Mac/iOS/Android), Приложение/Браузер (Chrome, Safari), Локацию (по IP, если есть), время последней активности.
  - Для текущей сессии показывать бейдж `Текущий сеанс / Current Device`.
  - Для остальных — кнопку-крестик или "Завершить сеанс" (Log out).
- **Кнопка "Завершить все остальные сеансы":** Помещается над списком. Требует модального окна подтверждения (AlertDialog).
- **UX:** Оптимистичное обновление — при клике на "Удалить" скрывать сессию из списка, отправляя запрос в фоне. Показать Toast об успешном отключении устройства.

## Технические требования (Общие)

1. **Безопасность и Валидация:**
   - Все запросы требуют наличие валидного JWT (`IsAuthenticated`).
   - На эндпоинты сброса пароля и отправки email должен быть наложен Rate Limiting (Throttle) в Django для защиты от брутфорса.
   - Ошибки валидации DRF должны парситься на клиенте (Axios interceptor) и прокидываться в поля формы (React Hook Form `setError`). Никаких общих ошибок "Что-то пошло не так", если сервер ответил 400 Bad Request с конкретным полем.
2. **Типизация:** Создайте полные и точные интерфейсы (TypeScript Types / Zod Schemas) для ответов бэкенда и запросов (SessionResponse, ChangePasswordRequest и т.д.).
3. **Организация кода Frontend:** Разбейте сложные формы на отдельные компоненты: `ChangePasswordForm.tsx`, `ChangeEmailFlow.tsx`, `SessionManager.tsx`. Не пишите простыню кода в одном файле `SecurityTab.tsx`. Если уже есть существующие компоненты, то используйте их. Переиспользование кода и компонентов обязательно!
4. **Стилизация:** Придерживайтесь концепта "чистого и современного" UI. Используйте TailwindCSS + компоненты Shadcn/UI (Form, Input, Button, Card, Dialog, Toast, Badge). Раздел "Управление сессиями" и "Смена пароля" должны быть разделены визуальными блоками/карточками (`Card`, `CardHeader`, `CardContent`).

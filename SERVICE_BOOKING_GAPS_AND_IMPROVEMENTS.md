# Service Booking Flow — гэпы и улучшения

После сверки с `SERVICE_BOOKING_IMPLEMENTATION_PROMPT.md` и проверки текущей реализации.

---

## Исправленные гэпы

### Backend (flow_views.py)

1. **`pet.type` → `pet.pet_type`**  
   В `LocationSlotsAPIView` и `CreateAppointmentAPIView` использовалось несуществующее поле `pet.type`. В модели `Pet` тип задаётся полем `pet_type` (FK). Исправлено на `pet.pet_type`.

2. **Учёт размера питомца (ProviderLocationService)**  
   `ProviderLocationService` имеет уникальность по `(location, service, pet_type, size_code)`. Для слотов и создания брони теперь используется размер питомца: `pet.get_current_size_category()` (S/M/L/XL). Если размер не задан, берётся первая подходящая запись по типу.

3. **Фильтр «грубого времени» в поиске**  
   Реализована фильтрация по параметру `date`: при передаче даты в поиск остаются только локации, у которых есть сотрудники с рабочим днём в этот день недели. Полная проверка наличия свободных слотов в выбранную дату не делается (см. улучшения).

4. **Сортировка слотов**  
   Слоты в ответе `slots_by_date` сортируются по `start_time` внутри каждого дня.

5. **Дубликаты услуг в ответе поиска**  
   В ответе поиска для каждой локации услуги дедуплицируются по `service_id`; при наличии размера питомца показывается цена/длительность по его размеру.

### Frontend

1. **Параметр даты при поиске**  
   Чипы «Сегодня», «Завтра», «На выходных» теперь переводятся в конкретную дату и передаются в API как `date` (YYYY-MM-DD). Добавлена функция `bookingFlowAPI.roughTimeToDate()`.

2. **Выбор услуги в модалке бронирования**  
   В модалке записи используется `selectedServiceId` из контекста: если у выбранной локации есть эта услуга, бронируется она, иначе — первая услуга локации.

3. **Синхронизация карта/список**  
   Подсветка по hover уже реализована: `MapPlaceholder` использует `hoveredLocationId` из контекста.

## Рекомендуемые улучшения

### Backend

1. **Жёсткая проверка «грубого времени»**  
   Сейчас при фильтре по дате отсекаются только локации без расписания в этот день недели. Можно усилить: для выбранной даты (или диапазона «выходные») вызывать логику слотов и оставлять в выдаче только локации с хотя бы одним свободным слотом. Учесть нагрузку на БД и кэширование.

2. **Рейтинг провайдера/локации**  
   В ответе поиска сейчас заглушка `rating: 5.0`. Подключить реальный рейтинг (например, из отзывов/агрегатов).

3. **EmployeeLocationService при расчёте слотов**  
   В `LocationSlotsAPIView` слоты строятся по всем сотрудникам с расписанием в локации. По ТЗ слоты должны быть только у сотрудников, оказывающих данную услугу в этой локации. Имеет смысл фильтровать по `EmployeeLocationService` (location + service).

4. **Перерывы в расписании (break_start/break_end)**  
   В генерации слотов не учитываются поля `Schedule.break_start` и `break_end`. Их стоит вычитать из доступного окна.

5. **HolidayShift и производственный календарь**  
   Для дат, объявленных праздниками, использовать `HolidayShift` вместо обычного `Schedule`, чтобы не показывать слоты в нерабочие дни.

6. **select_related/prefetch в поиске**  
   В цикле по локациям для каждой вызывается запрос `ProviderLocationService.objects.filter(...)`. Можно сократить число запросов через `prefetch_related('location_services')` и фильтрацию в Python или через Prefetch с фильтром по `pet_type`/`size_code`.

### Frontend

1. **Автопереход на первый свободный день**  
   В модалке при открытии уже выбирается первый день со слотами. Можно явно показывать подсказку: «Сегодня занято, доступно во вторник» и подсвечивать первый доступный день.

2. **Skeleton loaders**  
   В модалке слоты уже прикрыты плейсхолдером. Имеет смысл добавить скелетоны для списка результатов поиска и карточек локаций при загрузке.

3. **Каталог услуг (Cascader/Tree)**  
   В промпте указана кнопка «Каталог» и навигация по дереву `Service`. Сейчас в Omnibox есть только категории верхнего уровня и дочерние. Добавить полноценный каскадный выбор по иерархии каталога и breadcrumbs при глубокой вложенности.

4. **Карта**  
   Сейчас используется плейсхолдер с условными маркерами. Подключить Leaflet/Google Maps, реальные координаты из ответа поиска (при наличии полей lat/lon в ответе), центрирование по `userLocation` и маркеры по локациям.

5. **Ошибки API**  
   Показывать пользователю текст ошибки от бэкенда (например, «Service not available for this pet at this location») вместо общего alert.

6. **Выбор услуги в карточке локации**  
   Если у локации несколько услуг, можно добавить выбор конкретной услуги в карточке перед открытием модалки (или явно передавать выбранную услугу в контекст при клике «Записаться»).

### Общее

1. **Тесты**  
   Добавить тесты на `ProviderSearchAPIView`, `LocationSlotsAPIView`, `CreateAppointmentAPIView` (параметры, права, размер питомца, конфликты слотов).

2. **Локализация**  
   Проверить ключи `booking.*` во всех локалях (en, ru, me, de) и при необходимости добавить недостающие.

3. **Документация API**  
   Описать в OpenAPI/Swagger эндпоинты `GET /api/v1/booking/search/`, `GET /api/v1/booking/locations/<id>/slots/`, `POST /api/v1/booking/appointments/` (параметры, форматы дат, коды ошибок).

## Краткая проверка флоу

- Выбор питомца → ограничивает услуги по типу животного (бэкенд + при желании фронт по каталогу).  
- Поиск: `pet_id`, опционально `q`, `lat`/`lon`, `date`, `category_id` — выдаёт локации с дистанцией и услугами (с учётом размера питомца).  
- Грубое время: чипы переводятся в дату и передаются как `date`; бэкенд отфильтровывает локации без расписания в этот день недели.  
- Слоты: `location_id`, `service_id`, `pet_id`, `date_start`, `date_end` — слоты по сотрудникам и расписанию с учётом длительности и tech_break.  
- Создание брони: `provider_location_id`, `service_id`, `pet_id`, `employee_id`, `start_time` — проверка доступности и создание записи.  
- Модалка использует выбранную в контексте услугу или первую по локации; карта и список синхронизированы по hover.

После внесённых правок основной флоу соответствует промпту; оставшиеся пункты выше — рекомендации для следующей итерации.

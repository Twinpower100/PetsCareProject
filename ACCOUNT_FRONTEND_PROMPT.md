# Промпт для Frontend-разработчика (Реализация страницы Аккаунта)

**Роль:** Senior Frontend Developer
**Задача:** Реализовать страницу управления личным профилем в закладке Профиль по пути `http://localhost:3000/account` в приложении `Petscare-web`.
Вкладка Профильдолжна содержать два основных функциональных блока:
1. Редактирование личных данных профиля.
2. Деактивация (удаление) учетной записи пользователя ("Danger Zone").

---

## 1. Редактирование профиля (Profile Edit Flow)

### Источник данных (API)
- **GET** `/api/v1/users/profile/` — получение текущих данных пользователя (имя, фамилия, телефон, фото профиля и т.д.).
- **PATCH / PUT** `/api/v1/users/profile/` — обновление данных профиля.

### UI/UX и Логика
- **Форма:** Используйте `React Hook Form` в связке с `Zod` для строгой валидации на клиенте (например, правильный формат номера телефона, обязательные поля для имени).
- **Состояния загрузки:**
  - При монтировании компонента показывать скелетон (Skeleton) или спиннер, пока данные пользователя загружаются.
  - Состояние загрузки (`isSubmitting`) на кнопке «Сохранить» при отправке данных (disable кнопки + иконка загрузки).
- **Уведомления:** При успешном обновлении данных показывать Toast-уведомление (Success). При ошибке со стороны сервера (например, телефон уже занят) — выводить `Error Toast` или подсвечивать конкретные поля формы.

---

## 2. Деактивация учетной записи (Deactivation Flow)

В системе реализован механизм "Soft Delete" (деактивация). Пользователь не удаляется из БД физически, но его профиль деактивируется (`is_active=False`), токены инвалидируются.

### Источник данных (API)
- **POST** `/api/v1/users/deactivate-account/` (или соответствующий роут, привязанный к `AccountDeactivationView`).
- Эндпоинт не требует тела запроса (или требует подтверждения паролем/строкой, сверяйтесь с документацией или текущим Swagger).

### UI/UX и Логика
- Этот блок должен быть визуально отделен от редактирования данных профиля. Оформите его как **"Danger Zone"** (зона опасных действий) — с красной кнопкой «Деактивировать аккаунт».
- **Модальное окно подтверждения (Confirmation Dialog):** 
  При клике на кнопку деактивации пользователь **не должен** удаляться сразу. Необходимо показать модальное окно (AlertDialog) с текстом-предупреждением: 
  *«Вы уверены, что хотите деактивировать аккаунт? Отменить это действие будет невозможно.»*
- В модалке попросите пользователя ввести слово `УДАЛИТЬ` (или `DEACTIVATE`), чтобы избежать случайной деактивации, либо оставить просто две кнопки: Отмена и Подтвердить.

### Вариативность ответа Backend'а (ОЧЕНЬ ВАЖНО)
Бэкенд содержит строгие проверки бизнес-логики (прописано в `PetsCare_Описание_продукта.md`). Деактивация завершается **Ошибкой 400 Bad Request** со специфичным текстом в случаях, если:
1. Пользователь является **основным владельцем (Main Owner)** активного питомца (сначала нужно передать права или деактивировать питомца).
2. У пользователя есть **активные смены/задачи** в роли сотрудника клиники.
3. У пользователя есть **незавершенные активные заявки на передержку**.

**Требование к Frontend:**
- Вы должны корректно перехватывать эти 400 HTTP ошибки (AxiosError).
- Вместо стандартного "Что-то пошло не так", парсить сообщение об ошибке от API и выводить его пользователю в модалке или `Error Toast`, чтобы он понимал, *что именно* ему нужно завершить (передать питомца, завершить передержку), прежде чем он сможет удалиться.

### При успешном удалении (Status 200 OK):
1. Очистить локальное хранилище состояния (`localStorage`, cookies) от JWT токенов (access/refresh).
2. Сбросить глобальный стейт пользователя (Zustand, Redux или Context).
3. Принудительно запустить редирект (через `next/navigation` или ваш роутер) на публичную главную страницу (`/`) или страницу входа (`/login`).
4. Показать прощальный Toast: *«Ваш аккаунт был успешно деактивирован»*.

## Технические и стилистические требования
1. **Дизайн:** Используйте современный дизайн. Компоненты UI должны быть красивыми, использовать скругления, тени, корректные hover/focus стейты. Используйте вашу текущую библиотеку компонентов (TailwindCSS, Shadcn UI и т.д.).
2. **Типизация:** Задекларируйте точные Typescript-интерфейсы (или типы) для User Profile API Response и Error Response.
3. **Оптимизация:** Желательно реализовать данный флоу без создания "висячих" промисов. Используйте `React Query` (или SWR), если они заложены в архитектуру frontend'а, или кастомные хуки.`

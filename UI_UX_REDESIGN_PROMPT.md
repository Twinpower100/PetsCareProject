# PetsCare — UI/UX Redesign Prompt

> **Роль исполнителя:** Senior UI/UX Designer + Frontend Developer (React, Tailwind CSS, i18n).
> **Стек:** React 18+, Tailwind CSS, react-i18next, react-router-dom.
> **Проект:** PetsCare — агрегатор услуг для домашних животных (груминг, ветеринария, передержка, выгул, дрессировка и т.д.).
> **Целевая аудитория:** владельцы домашних животных, 25–55 лет, смартфон — основное устройство.

---

## 1. Общая дизайн-философия

### 1.1 Тон и настроение
- **«Тёплый профессионализм»** — строгий, но уютный. Не «казённый портал», а место, где заботятся о твоём любимце.
- Дизайн должен вызывать **доверие** (как у врача) и **теплоту** (как дома). Аналогия: хороший ветеринарный кабинет с мягким освещением.
- Микроанимации должны создавать ощущение **живости** интерфейса, но не отвлекать.

### 1.2 Референсы для вдохновения
| Сервис | Что взять |
|---|---|
| **Rover.com** | Карточки с крупным фото питомца, тёплые тона, CTA-кнопки с иконками лапок |
| **PetDesk** | Чистый медицинский стиль с мягкими закруглениями, палитра «медицинский уют» |
| **Wag!** | Яркие, дружелюбные иллюстрации, игривые микроанимации |
| **Petio (pet farewell UX)** | Эмпатичный подход к деликатным действиям (смерть питомца) |
| **Booking.com** | Карточки услуг с чёткой иерархией, отзывы, доверительные бейджи |

## 3. Типографика

### 3.1 Текущая проблема
`Inter` — нейтральный, но **холодный**. Для pet-care сервиса нужна шрифтовая пара, передающая дружелюбие.

### 3.2 Рекомендация

```css
@import url('https://fonts.googleapis.com/css2?family=Nunito:wght@400;500;600;700;800&family=Inter:wght@400;500;600&display=swap');
```

| Элемент | Шрифт | Почему |
|---|---|---|
| Заголовки (h1–h3) | **Nunito** (700/800) | Округлый, дружелюбный, «мягкий» — идеален для pet-тематики |
| Основной текст, кнопки | **Inter** (400/500/600) | Остаётся для читабельности, не меняем |
| Иконки-эмодзи | Нативные или Lucide React | Заменить текстовые эмодзи (🐾📅🏥) на SVG-иконки из Lucide |

## 4. Карточка питомца — полный редизайн

### 4.1 Текущие проблемы (критические)
1. **Карточка сливается с фоном** — `bg-white` на `bg-gray-50` почти не отличается.
2. **Кнопки «Архивировать»** — чудовищный нейминг. Пользователь часто удаляет питомца, потому что тот **умер**. Это травма. Слово «Архивировать» — из лексикона разработчика.
3. **«Передать права основного владельца»** — виден всегда, даже если совладельцев нет (некому передавать).
4. **Цвета кнопок** — розовый «Архивировать» выглядит как опасное действие. Нет визуальной иерархии.

### 4.2 Новая структура карточки

```
┌─────────────────────────────────────────────────┐
│  ┌──────┐                                       │
│  │ ФОТО │  Имя питомца              [Владелец]  │
│  │  64px │  Порода · Возраст                     │
│  │ round │  Вес: 5 кг                            │
│  └──────┘                                       │
│─────────────────────────────────────────────────│
│                                                 │
│  [ 🔍 Найти услуги ]          ← accent, full-w  │
│                                                 │
│  [ ✏️ ]  [ 👥 Совладельцы ]  [ ··· ]            │
│   edit     co-owners        more menu           │
│                                                 │
└─────────────────────────────────────────────────┘
```

### 4.3 Ключевые изменения

#### A. Визуальное отделение от фона
```
Фон страницы: bg-warm-50 (#faf9f7)
Карточка: bg-white rounded-2xl shadow-md border border-warm-200/60
Hover: shadow-lg translate-y-[-2px] transition-all duration-300
Selected: ring-2 ring-primary-400 shadow-primary-100
```

#### B. Фото питомца
- Если фото есть — круглое, 64px на мобиле, 80px на десктопе. Border `ring-2 ring-primary-200`.
- Если фото нет — gradient-аватар с иконкой типа питомца (кот/собака/птица) из Lucide, `bg-gradient-to-br from-primary-100 to-primary-200`.
- **Добавить возможность кликнуть на фото** для его увеличения (lightbox).

#### C. Кнопки — новая иерархия и нейминг

##### Основные действия (всегда видны на карточке):

| Действие | Старый текст | Новый текст (RU) | Новый текст (EN) | Стиль |
|---|---|---|---|---|
| Найти услуги | «Найти услуги» | **«Найти услуги для {name}»** | «Find services for {name}» | `bg-accent-500 text-white` — **главный CTA** |
| Редактировать | «Редактировать» | Иконка ✏️ (без текста, tooltip) | Pencil icon + tooltip | `border-warm-300 text-warm-600` — icon-only, компактно |

##### Меню «···» (kebab / three-dot menu):
Все остальные действия перенести в **выпадающее меню** (popover/dropdown). Это уберёт визуальный мусор с карточки:

| Пункт меню | Иконка | Условие видимости |
|---|---|---|
| Совладельцы | 👥 | Всегда (для main owner) |
| Передать ответственность | 🔄 | **Только если есть хотя бы 1 совладелец** |
| — разделитель — | | |
| {name} ушёл | 🕊️ | Только для main owner |

##### «{name} ушёл» вместо «Архивировать» / «Удалить»

**Это самое важное UX-изменение в этом документе.**

Формулировка лаконичная и мягкая. Покрывает все сценарии: смерть, потеря, передача другому хозяину.

Когда пользователь нажимает «{name} ушёл», открывается **не window.confirm**, а **кастомный модальный диалог** с эмпатичным контентом:

```
┌──────────────────────────────────────────────────┐
│                                                  │
│                 🕊️                                │
│                                                  │
│   Мы понимаем, что это непросто.                 │
│   {name} навсегда останется в вашем сердце       │
│   и в нашей системе.                             │
│                                                  │
│   История визитов {name} будет сохранена          │
│   для специалистов, которые о нём заботились.    │
│                                                  │
│   [ Вернуться ]              [ Подтвердить ]     │
│    ghost button          muted, не красный       │
│                                                  │
└──────────────────────────────────────────────────┘
```

**Важно:**
- Кнопка подтверждения **не красная** (это не ошибка — это горе или утрата). Использовать `bg-warm-600 text-white` или `bg-stone-600`.
- Текст кнопки в меню — `"{name} ушёл"` / `"{name} has gone"`, **не** `"Удалить"`, **не** `"Архивировать"`.
- **Никогда** не использовать `window.confirm()` для этого действия.

##### ХОТЕЛКИ (не MVP) — Адаптивный диалог прощания
В будущем диалог может спрашивать **причину** и менять тон:
- 🌈 *«{name} ушёл на радугу»* (умер) → эмпатичный текст, поле для тёплых слов (`farewell_note`)
- 🔍 *«{name} потерялся»* → надежда на возврат, профиль можно восстановить
- 🏠 *«{name} теперь у другого хозяина»* → деловой тон
- 📝 *«Другое»* → нейтральный

Это НЕ MVP — реализуется после базовой версии.

##### Совладелец — действия для не-main-owner:
| Действие | Старый текст | Новый текст (RU) | Новый текст (EN) |
|---|---|---|---|
| Покинуть | «Покинуть питомца (удалить себя из совладельцев)» | **«Отвязать себя от {name}»** | «Remove yourself from {name}'s profile» |

Длинный пояснительный текст — в tooltip или в тексте модального подтверждения, **не на кнопке**.

## 5. Панель совладельцев — редизайн

### 5.1 Текущие проблемы
- «Передать права основного владельца» видна **всегда**, даже без совладельцев — некому передавать.
- UX-поток: нажать на питомца → появляется панель совладельцев внизу → можно пригласить → можно передать. Это разнесено и неинтуитивно.

### 5.2 Рекомендация
- Переместить панель совладельцев в **модальное окно** или **slide-over panel** (справа), открывающееся по кнопке «Совладельцы» из kebab-меню.
- **«Передать ответственность»** — показывать **только** рядом с конкретным совладельцем, а не как глобальное действие.
- Предпочтительный формат:

## 6. Анимации и микровзаимодействия

### 6.1 Принцип
Анимации должны быть **функциональными** (помогают понять, что произошло), а не декоративными. Общая длительность: 200–350ms, easing: `ease-out` или `cubic-bezier(0.16, 1, 0.3, 1)`.

### 6.2 Конкретные анимации

| Элемент | Анимация | CSS / Tailwind |
|---|---|---|
| Карточка питомца hover | Плавное «всплытие» (shadow + translate) | `transition-all duration-300 hover:shadow-lg hover:-translate-y-1` |
| Карточка появление | Fade-in + slide-up при первой загрузке | staggered: каждая карточка с задержкой +50ms |
| Кнопки | Scale 0.95 → 1.0 при нажатии | `active:scale-95 transition-transform duration-150` |
| Модальные окна | Fade + scale от 0.95 к 1.0 | `animate-modal-appear` (уже есть в конфиге) |
| Сворачиваемые блоки | Плавное раскрытие (height auto) | `transition-[max-height,opacity] duration-300` или библиотека `react-collapse` |
| Фото питомца | Мягкий zoom при hover | `transition-transform duration-500 hover:scale-105` |
| Загрузка | Скелетон (shimmer) вместо спиннера | Tailwind `animate-pulse` на placeholder-блоках |
| Добавление карточки | Появление с анимацией scale + fade | `animate-in zoom-in-90 fade-in duration-300` |
| Удаление карточки | Fade-out + collapse | `animate-out fade-out zoom-out-95 duration-200` |
| Tab switching | Slide-контент слева/справа | `transition-transform` + `translate-x` |

### 6.3 Staggered Animation для списка карточек
```tsx
// Каждая карточка появляется с задержкой
<article
  style={{ animationDelay: `${index * 60}ms` }}
  className="opacity-0 animate-[fadeSlideUp_0.4s_ease-out_forwards]"
>
```
```css
@keyframes fadeSlideUp {
  from { opacity: 0; transform: translateY(12px); }
  to   { opacity: 1; transform: translateY(0); }
}
```

## 7. Mobile-First подход

### 7.1 Карточки питомцев
- **Мобиль (< 640px):** Одна колонка, карточки в полную ширину. Фото 56px. Кнопки стековые.
- **Планшет (640–1023px):** Две колонки.
- **Десктоп (1024px+):** Три колонки.

### 7.2 Touch-friendly
- Минимальный размер tap-target: **44px × 44px** (Apple HIG).
- Свайп-жесты: рассмотреть swipe-left на карточке для быстрого доступа к меню (опционально, v2).
- Кнопки на мобиле: `min-h-[48px]` для удобного нажатия пальцем.
- Увеличенные отступы между интерактивными элементами: `gap-3` минимум.

### 7.3 Bottom Sheet вместо Dropdown
На мобиле kebab-меню (···) должно открывать **bottom sheet** (снизу экрана), а не маленький dropdown. Это нативный паттерн для iOS и Android.

## 9. Предложения для агрегатора услуг — дополнительные UX-идеи

### 9.1 Раздел «Мои питомцы» — отдельная страница `/my-pets`
Вынести раздел «Мои питомцы» из вкладки Account Management в **полноценную страницу** `/my-pets`:
- Список всех питомцев (карточки-превью) — это основной экран
- Клик на карточку → переход на **страницу профиля питомца** `/pets/:id` с:
  - Крупным фото (hero-стиль, размытый фон + фото по центру)
  - Таймлайном визитов и бронирований
  - Секцией здоровья (вакцинации, хронические заболевания)
  - QR-кодом профиля (для передачи специалисту)
- Кнопка «Добавить питомца» — на странице списка
- У пользователя может быть **много** питомцев — страница должна хорошо масштабироваться

> ⚡ Это реализуется **сразу**, не откладывается.

### 9.2 «Дашборд заботы» (Care Dashboard) на главной
После логина показывать:
- **Ближайшие визиты** — карточки с обратным отсчётом
- **Напоминания** — вакцинации, которые скоро истекают (жёлтый бейдж)
- **Рекомендуемые услуги** — на основе профиля питомца (порода → типичные потребности)

### 9.3 Доверительные элементы
- **Бейджи специалистов:** «Проверен», «Сертифицирован», «Топ рейтинга» — как на Booking.com.
- **Отзывы с фото** — «после посещения» с фото довольного питомца.
- **Счётчик:** «Более 500 счастливых питомцев в вашем городе».

### 9.4 Элементы «мимими» (emotional design)
- **Фоновые иллюстрации:** Тонкие, полупрозрачные силуэты лапок/хвостиков в фоне секций. Не на каждой странице — точечно.
- **Easter-egg при добавлении питомца:** короткая анимация конфетти из лапок (200ms, subtle).
- **Сезонные акценты:** зимой — снежинки на лого, весной — цветочки. Ненавязчиво, через CSS-переменные.
- **Именные обращения:** «Привет! У Барсика через 3 дня визит к грумеру» — не обезличенно.
